Data Quality

Accuracy Basic

Accuracy measures how closely data values match the true or accepted values. Inaccurate data leads to wrong conclusions.

Common Issues

  • Transcription errors (typos)
  • Unit conversion mistakes
  • Stale/outdated values
  • Measurement errors

Detection Methods

-- Range validation SELECT * FROM products WHERE price < 0 OR price > 1000000; -- Cross-reference check SELECT o.* FROM orders o LEFT JOIN customers c ON o.customer_id = c.id WHERE c.id IS NULL;

Completeness Basic

Completeness measures the presence of required values. Missing data can bias analysis and break pipelines.

-- Check completeness per column SELECT COUNT(*) AS total_rows, COUNT(name) AS name_present, COUNT(email) AS email_present, ROUND(COUNT(name)::DECIMAL / COUNT(*) * 100, 1) AS name_pct, ROUND(COUNT(email)::DECIMAL / COUNT(*) * 100, 1) AS email_pct FROM users; -- Find incomplete records SELECT * FROM users WHERE name IS NULL OR email IS NULL OR email = '';

Completeness Check Flow

Define Requirements → Scan Columns → Calculate % → Alert if < Threshold

Consistency Basic

Consistency ensures data follows the same rules across systems and over time.

Types of Consistency

  • Format: Same date format everywhere
  • Units: Same currency, measurement units
  • Reference: Same lookup values
  • Cross-system: Matching data across sources

Example Issues

-- Inconsistent formats "2024-01-15" -- ISO format "01/15/2024" -- US format "Jan 15, 2024" -- Text format -- Inconsistent categories "US", "USA", "United States", "U.S." -- Fix with CASE UPDATE users SET country = 'US' WHERE country IN ('USA', 'United States', 'U.S.');

Timeliness Basic

Timeliness measures how current the data is. Data that arrives too late loses value.

-- Check data freshness SELECT MAX(updated_at) AS last_update, NOW() - MAX(updated_at) AS staleness, CASE WHEN NOW() - MAX(updated_at) > INTERVAL '24 hours' THEN 'STALE' ELSE 'FRESH' END AS status FROM orders; -- SLA monitoring SELECT pipeline_name, MAX(completed_at) AS last_run, AVG(duration_seconds) AS avg_duration FROM pipeline_runs GROUP BY pipeline_name;

Validity Basic

Validity ensures data conforms to defined formats, types, and business rules.

-- Email format validation SELECT * FROM users WHERE email !~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z]{2,}$'; -- Phone number format SELECT * FROM contacts WHERE phone !~ '^\+?[0-9]{10,15}$'; -- Enum validation SELECT * FROM orders WHERE status NOT IN ('pending', 'confirmed', 'shipped', 'delivered'); -- Date range validation SELECT * FROM events WHERE event_date NOT BETWEEN '2020-01-01' AND CURRENT_DATE;

Uniqueness Basic

Uniqueness ensures each record can be uniquely identified. Duplicate records cause inflated metrics and errors.

-- Find duplicate emails SELECT email, COUNT(*) AS cnt FROM users GROUP BY email HAVING COUNT(*) > 1; -- Find all duplicate rows SELECT * FROM users WHERE email IN ( SELECT email FROM users GROUP BY email HAVING COUNT(*) > 1 ); -- Prevent with unique constraint ALTER TABLE users ADD CONSTRAINT uq_users_email UNIQUE (email);

Handling NULL Values Intermediate

NULL values require special handling. Strategy depends on the cause and impact of missing data.

Strategies

  • Remove: Delete rows with NULLs
  • Default: Replace with 0, "", or NA
  • Impute: Mean, median, mode, or model
  • Flag: Create is_missing column

PostgreSQL Handling

-- Replace NULLs SELECT name, COALESCE(email, 'N/A') AS email, COALESCE(age, 0) AS age FROM users; -- NULL-safe comparison SELECT * FROM users WHERE email IS NOT DISTINCT FROM 'alice@co.com';

Deduplication Intermediate

Remove duplicate records while preserving the most complete or recent version.

-- Method 1: ROW_NUMBER to find duplicates WITH numbered AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY email ORDER BY updated_at DESC ) AS rn FROM users ) DELETE FROM numbered WHERE rn > 1; -- Method 2: Keep only distinct records CREATE TABLE users_deduped AS SELECT DISTINCT ON (email) * FROM users ORDER BY email, updated_at DESC; -- Method 3: Merge (upsert) INSERT INTO users_final SELECT * FROM users_staging ON CONFLICT (email) DO UPDATE SET name = EXCLUDED.name, updated_at = EXCLUDED.updated_at;

Outlier Detection Intermediate

Outliers are extreme values that deviate significantly from other observations. They may indicate errors or interesting phenomena.

-- Method 1: IQR (Interquartile Range) WITH stats AS ( SELECT PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY amount) AS q1, PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY amount) AS q3 FROM orders ) SELECT * FROM orders, stats WHERE amount < ROUND(q1 - 1.5 * (q3 - q1), 2) OR amount > ROUND(q3 + 1.5 * (q3 - q1), 2); -- Method 2: Z-score WITH stats AS ( SELECT AVG(amount) AS mean, STDDEV(amount) AS stddev FROM orders ) SELECT * FROM orders, stats WHERE ABS(amount - mean) / stddev > 3; -- Method 3: Percentile bounds SELECT * FROM orders WHERE amount NOT BETWEEN PERCENTILE_CONT(0.01) WITHIN GROUP (ORDER BY amount) AND PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY amount);

Data Standardization Intermediate

Standardization converts data to a consistent format for reliable comparison and analysis.

-- Standardize names UPDATE users SET name = INITCAP(TRIM(name)); -- Standardize emails (lowercase) UPDATE users SET email = LOWER(TRIM(email)); -- Standardize phone numbers UPDATE contacts SET phone = REGEXP_REPLACE(phone, '[^0-9]', '', 'g'); -- Standardize dates UPDATE events SET event_date = TO_DATE(event_date_text, 'MM/DD/YYYY'); -- Standardize categories UPDATE orders SET status = LOWER(TRIM(status)); -- Normalize text encoding UPDATE articles SET content = CONVERT(content, 'UTF8', 'LATIN1');

Database Constraints Basic

Constraints enforce data quality at the database level, preventing bad data from entering the system.

CREATE TABLE orders ( id SERIAL PRIMARY KEY, -- NOT NULL: Required fields customer_id INTEGER NOT NULL, total DECIMAL(10,2) NOT NULL, -- CHECK: Business rules status VARCHAR(20) NOT NULL CHECK (status IN ('pending', 'confirmed', 'shipped')), quantity INTEGER CHECK (quantity > 0), total DECIMAL(10,2) CHECK (total >= 0), -- UNIQUE: No duplicates order_number VARCHAR(20) UNIQUE NOT NULL, -- FOREIGN KEY: Referential integrity customer_id INTEGER REFERENCES customers(id), -- DEFAULT: Sensible defaults created_at TIMESTAMPTZ DEFAULT NOW() ); -- Deferrable constraint (for bulk loads) ALTER TABLE order_items ADD CONSTRAINT fk_order FOREIGN KEY (order_id) REFERENCES orders(id) DEFERRABLE INITIALLY DEFERRED;

Data Profiling Intermediate

Data profiling analyzes data to understand its structure, content, and quality.

-- Basic profiling query SELECT column_name, data_type, COUNT(*) AS total_rows, COUNT(*) AS non_null_count, COUNT(*) - COUNT(column_name) AS null_count, ROUND(COUNT(column_name)::DECIMAL / COUNT(*) * 100, 2) AS completeness_pct, COUNT(DISTINCT column_name) AS cardinality FROM my_table, UNNEST(ARRAY['col1', 'col2', 'col3']) AS column_name GROUP BY column_name; -- Numeric column stats SELECT MIN(amount) AS min_val, MAX(amount) AS max_val, AVG(amount) AS mean_val, STDDEV(amount) AS stddev, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY amount) AS median FROM orders; -- String column stats SELECT AVG(LENGTH(name)) AS avg_length, MIN(LENGTH(name)) AS min_length, MAX(LENGTH(name)) AS max_length, MODE() WITHIN GROUP (ORDER BY name) AS most_common FROM users;
