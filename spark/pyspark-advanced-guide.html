1) File Handling Guide

Choosing the right format and read strategy defines base performance. Prefer columnar formats (Parquet/ORC) for analytics and enforce schema on raw text formats.

Input (CSV rows)

idnamesalarydept
1Ana9000Eng
2SambadHR

Output (clean parquet)

idnamesalarydept
1Ana9000Eng
2SamnullHR
from pyspark.sql.types import StructType, StructField, IntegerType, StringType

schema = StructType([
    StructField("id", IntegerType(), True),
    StructField("name", StringType(), True),
    StructField("salary", IntegerType(), True),
    StructField("dept", StringType(), True)
])

raw = spark.read.option("header", True).schema(schema).csv("/raw/emp.csv")
raw.write.mode("overwrite").parquet("/curated/emp")

2) Window Functions

Window functions compute row-aware analytics without collapsing rows like groupBy. Use partitioning and ordered frames carefully to avoid large shuffle costs.

Input (dept salaries)

empdeptsalary
AEng9000
BEng7000
CHR6000

Output (ranked)

empdeptsalaryrank_in_dept
AEng90001
BEng70002
CHR60001
from pyspark.sql import Window
from pyspark.sql.functions import row_number, sum as Fsum

w = Window.partitionBy("dept").orderBy("salary")
ranked = df.withColumn("rank_in_dept", row_number().over(w.orderBy(df.salary.desc())))

running = df.withColumn(
    "running_salary",
    Fsum("salary").over(Window.partitionBy("dept").orderBy("salary").rowsBetween(Window.unboundedPreceding, Window.currentRow))
)

3) Memory-Related Issues

Typical symptoms are GC overhead, spill to disk, and executor OOM. Fix by reducing shuffle data, tuning partition size, and selecting sane persistence levels.

SymptomRoot causeFix
Executor OOMWide rows + high concurrencyIncrease memory, reduce columns
Frequent spillsOversized partitionsRepartition and tune shuffle partitions
Cache evictionOver-cachingPersist selective hot datasets only

spark.conf.set("spark.sql.shuffle.partitions", "240")

4) Shuffle-Related Issues

Shuffle is often the largest cost center. Avoid unnecessary repartitions, use broadcast joins for small dimensions, and enable AQE for skew handling.

from pyspark.sql.functions import broadcast

spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

joined = fact.join(broadcast(dim), "id", "left")

5) CPU-Related Issues

Low CPU usage indicates poor parallelism; high CPU with low throughput often means heavy UDF or expression complexity.

# Better: built-in functions over Python UDF where possible
from pyspark.sql.functions import upper, regexp_replace

optimized = df.withColumn("clean_name", upper(regexp_replace("name", "[^A-Za-z ]", "")))

6) I/O-Related Issues

Small files and poor write partitioning kill read performance. Write larger files and partition by high-value query columns.

# Coalesce output files and partition intelligently
curated = df.select("event_date", "region", "amount")
curated.coalesce(64).write.mode("overwrite").partitionBy("event_date").parquet("/warehouse/sales")

7) Query Optimization Issues

Common anti-patterns: early explode, late filters, and nested UDF chains. Push filters and projections as early as possible.

optimized = (df
    .select("id", "region", "amount")
    .filter("amount > 100")
    .groupBy("region")
    .sum("amount"))

optimized.explain("formatted")

8) Network-Related Issues

Network hotspots appear in large shuffles and oversized broadcasts. Reduce shuffled data volume and limit cross-node transfer.

PatternImpactMitigation
Large fact-fact joinHuge shufflePre-aggregate and prune columns
