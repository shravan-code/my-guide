Introduction to Apache Spark

Spark is a unified distributed engine for ETL, SQL, machine learning, and streaming with strong performance and scalability.

Spark evolved from a research project into a production platform because it solved slow disk-heavy processing and fragmented toolchains. It made iterative analytics faster and development simpler.

Evolution Flow

2009 Research->RDD Core->SQL / DataFrames->Streaming + AQE
from pyspark.sql import SparkSession spark = SparkSession.builder.appName("intro").getOrCreate() print(spark.version)

Spark Architecture

Spark architecture separates planning (driver) from execution (executors) through a cluster manager.

ComponentRoleWhy it matters
DriverBuilds DAG, schedules tasksControls job lifecycle and optimization
Cluster ManagerAllocates resourcesEnsures executors are available
ExecutorsRun tasks, store cachePerform real distributed compute
Worker NodesHost executorsScale throughput with horizontal compute

Resilient Distributed Dataset (RDD)

RDD is Spark's low-level immutable distributed abstraction with lineage-based fault tolerance. It is useful when you need full control over partitioning, custom distributed logic, or pair-RDD style operations.

RDD operations are grouped as transformations (lazy) and actions (execution trigger). Wide transformations (like reduceByKey or join) create shuffle boundaries and new stages, while narrow transformations (like map) stay within partitions.

RDD Execution Flow

Create RDD->Narrow Transform->Wide Transform (Shuffle)->Action

DataFrames and Datasets

DataFrames provide schema-aware, SQL-like transformations and are optimized by Catalyst + Tungsten. Datasets add strong typing in Scala/Java; in PySpark, DataFrame is the primary API.

Compared to RDD, DataFrames carry column metadata and type information, which allows Spark to prune columns, push filters to data sources, reorder operations, and generate efficient physical plans. This makes DataFrames faster and easier to maintain for most production workloads.

AbstractionStrengthTrade-offBest use
RDDMaximum low-level controlManual optimization effortCustom partition/algorithm logic
DataFrameSchema + optimizer supportLess low-level controlETL, BI, analytical pipelines
Dataset (JVM)Compile-time type safetyMostly JVM languagesStrongly typed Spark applications

DataFrame Optimization Flow

Read + Schema->Logical plan->Catalyst optimize->Physical plan->Execution
from pyspark.sql.types import StructType, StructField, IntegerType, StringType schema = StructType([ StructField("id", IntegerType(), False), StructField("name", StringType(), True), StructField("region", StringType(), True), ]) df = spark.read.schema(schema).json("/raw/users.json") result = df.filter("region = 'APAC'").select("id", "name") result.explain("formatted")

Lazy Evaluation

Spark postpones execution so it can optimize the complete transformation graph before running any tasks.

When you chain transformations, Spark builds lineage metadata and a DAG. No cluster compute starts until an action appears. This allows Spark to collapse compatible operations, push filters downward, and reduce shuffle/network cost before execution.

Planning Phase

filter()->select()->groupBy()->DAG only

Execution Phase

count()->Job created->Stages/Tasks run
StepOperationExecuted now?Reason
1filter()NoTransformation stored in logical plan
2select()NoStill plan-building, no action yet
3groupBy().agg()NoDAG extends with aggregation node
4count()YesAction triggers full optimized execution
lazy_df = (df .filter("amount > 100") .select("region", "amount") .groupBy("region").sum("amount")) lazy_df.explain("formatted") # inspect planned execution print(lazy_df.count()) # action triggers execution

Catalyst Optimizer

Catalyst is Spark SQL's optimization framework that transforms parsed queries into highly efficient execution plans.

Catalyst applies rule-based and cost-aware optimizations such as constant folding, predicate pushdown, projection pruning, and join strategy selection. Its goal is to lower I/O, reduce shuffle volume, and improve CPU efficiency.

Catalyst Full Flowchart

SQL / DataFrame query->Unresolved logical plan->Analyzed logical plan->Optimized logical plan->Physical plan->Codegen + execution
OptimizationWhat it changesBenefit
Predicate pushdownMoves filters near data sourceLess data read from storage
Projection pruningDrops unused columns earlyLower memory and I/O cost
Constant foldingPrecomputes static expressionsLess runtime computation
Reorder/simplify expressionsRefines execution expression treeBetter CPU efficiency
Join selectionChooses broadcast/sort-merge/etc.Lower shuffle and faster joins
query = (df .filter("region = 'US'") .select("product", "amount") .groupBy("product").sum("amount")) query.explain("extended") # logical + physical plans query.explain("formatted") # readable physical plan view

Jobs, Stages, and Tasks

An action creates a Job. A Job splits into Stages at shuffle boundaries. Each Stage runs many Tasks, typically one task per partition.

This hierarchy explains why some workloads are slow: if a single stage is expensive (for example, heavy shuffle), the full job waits. If a few tasks are much slower than others, you likely have skewed partitions or uneven data distribution.

Hierarchy Diagram

Action->Job->Stages->Tasks

Stage Split Rule

Narrow transforms->Same stage->Shuffle->New stage
LevelTriggered byTypical countDebug focus
JobAction call (count, show, write)1 per actionEnd-to-end runtime and retries
StageShuffle boundariesMultiple per jobShuffle read/write volume
TaskPartition count in a stageMany per stageSkew, stragglers, task duration spread
spark.conf.set("spark.sql.shuffle.partitions", "200") result = (df .filter("amount > 0") # narrow .groupBy("region") # wide (shuffle boundary) .sum("amount")) result.explain("formatted") result.count() # triggers job with stage/task execution

Join Strategies

Spark chooses join strategies from table size, key distribution, and runtime statistics. Correct strategy selection is one of the biggest performance levers.

Join Strategy Decision Flow

Small right table?->Broadcast HashelseLarge-Large?->Sort Merge / Shuffle Hash
StrategyHow it worksBest whenTrade-offs
Broadcast Hash JoinBroadcast small side to every executor; hash-lookup per partitionOne side is small (dimension table)Fails if broadcast side is too large for executor memory
Sort Merge JoinShuffle both sides by key, sort partitions, then mergeBoth sides are large and join keys are sortableHigh shuffle + sort cost, sensitive to skew
Shuffle Hash JoinShuffle by key, build hash map on smaller post-shuffle sideModerate-size joins where hash build is feasibleMemory pressure on hash-building side
Broadcast Nested Loop JoinBroadcast one side and compare rows (no hash key required)Non-equi joins with small broadcast sideCan be expensive for large row sets
Shuffle-and-Replicate NLReplicate one side across shuffled partitionsRare edge cases when no better strategy appliesVery costly network/compute pattern
Cartesian / Cross JoinProduces every pair of rowsIntentional combinatorial use cases onlyExplosive output size; avoid by default

Practical rule: use broadcast for small dimensions, prefer sort-merge for very large tables, enable AQE for runtime switching and skew handling.

from pyspark.sql.functions import broadcast # Force broadcast strategy when dimension is small joined = fact.join(broadcast(dim_small.select("id", "segment")), "id", "left") # Let Spark/AQE decide for large joins spark.conf.set("spark.sql.adaptive.enabled", "true") big_join = large_a.join(large_b, "id", "inner")

Adaptive Query Execution (AQE)

AQE adapts query plans at runtime based on real stage statistics, reducing sensitivity to wrong estimates and evolving data shape.

Without AQE, Spark executes a static physical plan. With AQE enabled, Spark can coalesce tiny shuffle partitions, split skewed partitions, and switch join strategy (for example, from sort-merge to broadcast) once runtime sizes are known.

AQE Runtime Loop

Initial physical plan->Execute stage->Collect runtime stats->Re-optimize next stages
AQE featureWhat it changesTypical impact
Partition coalescingMerges many small shuffle partitionsLower scheduler overhead + better task sizing
Skew join handlingSplits oversized skewed partitionsReduces straggler tasks
Dynamic join switchChanges join strategy based on runtime table sizeLower shuffle volume
Local shuffle readerImproves data locality for shuffle readsLower network latency
spark.conf.set("spark.sql.adaptive.enabled", "true") spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true") spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true") spark.conf.set("spark.sql.adaptive.localShuffleReader.enabled", "true") # Optional tuning spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "64MB") spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")

Common File Formats

Spark supports row formats, columnar formats, and modern table formats. The right choice depends on ingest flexibility, query performance, update patterns, and governance needs.

In practice: use CSV/JSON/XML in raw zones, convert to Parquet/ORC in curated zones, and use Delta/Iceberg/Hudi for governed, update-friendly lakehouse tables.

FormatTypeStrengthsLimitationsBest usage
CSV / TSVRow textPortable, easy to inspectWeak typing, larger files, slow scansRaw exchange and basic ingest
JSON / NDJSONSemi-structured textSupports nested structuresHigher parse cost, larger storageEvent logs and semi-structured raw zones
TextUnstructuredUniversal ingestNo schema; manual parsing neededLog lines and free-form payloads
XMLSemi-structured textRich hierarchical dataVerbose and expensive parsingLegacy enterprise integrations
ParquetColumnarCompression, pruning, pushdownNot ideal for row-level updatesDefault analytics storage
ORCColumnarStrong compression/read performanceTooling preference may varyWarehouse-centric analytics
AvroRow binarySchema evolution for messagesLess optimal for wide scansStreaming/message interchange
Delta LakeTable formatACID, schema enforcement, time travelEngine/runtime dependenciesReliable lakehouse tables
Apache HudiTable formatUpserts + CDC friendlyOperational complexityNear-real-time lake ingestion
Apache IcebergTable formatStrong metadata and hidden partitioningRequires compatible compute enginesLarge multi-engine lakehouse platforms

Recommended Storage Pattern

Raw (CSV/JSON/XML)->Schema standardization->Parquet/ORC->Delta/Iceberg/Hudi

Partitioning and Bucketing

Partitioning stores data in directory paths by key (for pruning reads). Bucketing hashes rows into fixed buckets (for stable file grouping and join locality).

Partitioning is ideal for high-selectivity filters like date, region, or tenant. Bucketing is useful when frequent joins/grouping happen on high-cardinality keys where directory partitioning would create too many tiny folders.

Layout Decision Flow

Filter by date/region often?->Partition by that keyandFrequent joins on id?->Bucket by join key
TechniqueBest forProsCons
PartitioningLow-to-medium cardinality filter columnsFast partition pruning, less scan I/OToo many partitions can cause small-file problems
BucketingFrequent joins/aggregations on large tablesBetter join locality, consistent file groupingOperational complexity and table-management constraints
# 1) Partition by date for pruning (df .write .mode("overwrite") .partitionBy("event_date") .parquet("/warehouse/events")) # 2) Bucket by customer_id for join-heavy workloads # bucketBy is typically used with saveAsTable in Spark SQL catalogs (df .write .mode("overwrite") .bucketBy(32, "customer_id") .sortBy("customer_id") .saveAsTable("sales_bucketed")) # 3) Read with partition filter (pruning) daily = spark.read.parquet("/warehouse/events").filter("event_date = '2026-03-01'")

Repartition and Coalesce

repartition() reshuffles data to rebalance or increase partition count. coalesce() reduces partitions with minimal shuffle and is typically used near final writes.

Choosing between them depends on intent: if you need even data redistribution or more parallelism, use repartition. If you only need fewer output files and already-balanced data, use coalesce.

Method Choice Flow

Need more partitions / rebalance?->repartition()elseNeed fewer output files?->coalesce()
MethodShuffleProsConsIdeal use case
repartition(n)Yes (full shuffle)Even distribution, good parallelism controlExpensive network + shuffle costBefore wide joins/aggregations on skewed data
repartition(n, key)YesKey-aware redistributionShuffle-heavy if overusedAlign partitions with join/group key
coalesce(n)Mostly noCheap reduction in partition countCan produce uneven partition sizesReduce files before write
# Start with raw data base = spark.read.parquet("/silver/sales") # Rebalance by join/group key for heavy computation balanced = base.repartition(240, "region") # Compute heavy logic agg = balanced.groupBy("region").sum("amount") # Reduce output files before sink write out = agg.coalesce(24) out.write.mode("overwrite").parquet("/gold/region_sales")

Executor Memory

Executor memory is shared between execution memory (shuffle/sort/aggregation) and storage memory (cache/persist). Bad sizing causes spills, GC overhead, and out-of-memory failures.

Memory tuning is not only increasing heap size. You should also control partition size, reduce unnecessary columns, cache selectively, and avoid huge broadcast objects. These workload-level optimizations often give bigger gains than config-only changes.

Memory Pressure Pattern

Large rows->High shuffle memory->Spill / GC->Slow tasks / OOM
ConfigPurposeTypical valueWhen to tune
spark.executor.memoryHeap size per executor4g - 16g+Task OOM or frequent spills
spark.memory.fractionExecution + storage memory pool0.6 - 0.8Balance shuffle vs cache pressure
spark.memory.storageFractionStorage share within unified memory0.5 defaultHeavy caching workloads
spark.executor.memoryOverheadOff-heap / JVM overhead10% or moreContainer OOM outside heap
spark.sql.shuffle.partitionsShuffle partition countWorkload-dependentToo-large task memory footprint
spark.conf.set("spark.executor.memory", "8g") spark.conf.set("spark.executor.memoryOverhead", "2g") spark.conf.set("spark.memory.fraction", "0.7") spark.conf.set("spark.memory.storageFraction", "0.4") # Reduce memory footprint by pruning columns early lean = df.select("id", "region", "amount") # Reduce task memory pressure with sane shuffle partitioning spark.conf.set("spark.sql.shuffle.partitions", "240")

Dynamic Partition Pruning

DPP prunes fact-table partitions dynamically using runtime filter values coming from the dimension side of a join.

DPP is highly effective in star-schema patterns where a large partitioned fact table joins with smaller dimensions. Instead of scanning all fact partitions first, Spark uses filtered dimension keys to skip irrelevant partitions before expensive reads.

DPP Flow

Filter dimension (small)->Extract join keys->Prune fact partitions->Join remaining data
ConditionDPP impactGuidance
Fact table partitioned on join-related columnHigh benefitAlign partition key with query filters and joins
Dimension heavily filteredHigh benefitPush dimension filters early
No relevant partitioning on factLow/noneRedesign partition strategy
Small fact tableLimited gainDPP not critical; focus elsewhere
spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.enabled", "true") result = spark.sql(""" SELECT f.order_id, f.amount, d.region_name FROM fact_sales f JOIN dim_region d ON f.region_id = d.region_id WHERE d.region_name IN ('APAC', 'EMEA') """) result.explain("formatted") # verify pruning in plan

Dynamic Resource Allocation

Dynamic allocation automatically scales executors up/down using backlog and idle signals, improving cluster utilization and reducing costs.

When task backlog grows, Spark requests more executors. When executors stay idle for configured time, Spark releases them. Correct min/max bounds prevent under-provisioning and avoid resource thrashing.

Elasticity Loop

Pending tasks increase->Request executors->Throughput stabilizes->Idle timeout->Release executors
ConfigRoleExampleTuning guidance
spark.dynamicAllocation.enabledEnable featuretrueTurn on for variable workloads
spark.dynamicAllocation.minExecutorsLower bound2Set enough for baseline SLA
spark.dynamicAllocation.maxExecutorsUpper bound20Cap to control cost and quota
spark.dynamicAllocation.initialExecutorsStartup baseline4Avoid cold-start under-provisioning
spark.dynamicAllocation.executorIdleTimeoutIdle release threshold60sToo low causes thrashing, too high wastes cost
spark.dynamicAllocation.schedulerBacklogTimeoutScale-up trigger delay1sLower for bursty queues
spark.conf.set("spark.dynamicAllocation.enabled", "true") spark.conf.set("spark.dynamicAllocation.minExecutors", "2") spark.conf.set("spark.dynamicAllocation.initialExecutors", "4") spark.conf.set("spark.dynamicAllocation.maxExecutors", "24") spark.conf.set("spark.dynamicAllocation.executorIdleTimeout", "60s") spark.conf.set("spark.dynamicAllocation.schedulerBacklogTimeout", "1s") # monitor in Spark UI to verify healthy scaling behavior
