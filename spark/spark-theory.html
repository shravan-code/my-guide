Spark Theory

1

Introduction to Apache Spark

Apache Spark is an open-source distributed computing engine for large-scale data processing. It lets teams run batch ETL, SQL analytics, machine learning, and streaming pipelines using one unified engine instead of separate tools for each workload.

Spark evolved from a Berkeley research project (2009) into an Apache top-level project (2014), then expanded with DataFrames, Spark SQL, Structured Streaming, and AQE. This evolution made Spark both developer-friendly and production-ready for modern data platforms.

What Spark solved

It addressed slow disk-heavy MapReduce patterns, difficult low-level APIs, and fragmented data stacks by enabling in-memory execution, DAG optimization, and reusable high-level APIs.

Why Spark became popular

One engine for multiple workloads, strong cloud support, fault tolerance via lineage, and excellent ecosystem integration with data lakes and warehouses.

Spark Evolution Flow

2009 Research-> RDD Era-> DataFrames / SQL-> Streaming + AQE

From speed-focused research engine to full data platform runtime.

2

Spark Architecture

Spark architecture has clear runtime components. The Driver builds and schedules the execution plan. The Cluster Manager allocates resources. Executors run tasks and store cached partitions. Worker nodes host executors. Jobs, stages, and tasks define execution granularity.

Driver

Parses user code, builds DAG, coordinates stage execution, and collects action results.

Cluster Manager

Manages resources and launches executors (Standalone, YARN, Kubernetes, etc.).

Executors

Run partition-level tasks, perform shuffle operations, and cache/persist intermediate data.

Tasks and Stages

A job splits into stages at shuffle boundaries; each stage runs many parallel tasks.

Component Flowchart

User Code-> Driver-> Cluster Manager-> Workers-> Executors-> Tasks

Control plane on top, distributed compute plane underneath.

3

Transformations and Actions

Block A: Transformations (Lazy)

Transformations create a new DataFrame/RDD from existing data without executing immediately. Spark builds a logical plan and defers execution until an action appears, enabling whole-query optimization.

select()filter()withColumn() join()groupBy()repartition()

Used for shaping data pipelines and composing business logic.

Block B: Actions (Execution Trigger)

Actions trigger actual computation and either return results to the driver or write output to storage. They materialize the optimized physical plan as Spark jobs.

count()show()collect() take()first()write()

Used when you need final output, metrics, or persisted data products.

Execution Relationship

Transform-> Transform-> Transform-> Action-> Job Runs

No action means no execution; action triggers full plan evaluation.

4

Resilient Distributed Dataset (RDD)

RDD is Spark's foundational immutable distributed data abstraction. It tracks lineage rather than storing every intermediate state, which enables selective recomputation on failure.

RDD provides fine-grained control but requires manual optimization. DataFrames are usually preferred in production due to optimizer support, but RDD remains valuable for low-level custom logic.

Fault Recovery

RDD A->RDD B->RDD C

Lineage allows targeted recomputation.

5

DataFrames and Datasets

DataFrames add schema and relational semantics, enabling Spark to apply advanced optimizations such as predicate pushdown, projection pruning, and expression simplification.

Datasets add typed semantics in JVM languages. For Python data engineering, DataFrames are the practical default for maintainable, high-performance workflows.

Abstraction Ladder

RDD->DataFrame->Optimized SQL plan

Higher abstraction yields stronger automatic tuning.

6

Lazy Evaluation

Lazy evaluation delays physical execution until a result is required. This gives Spark visibility over the full plan and allows operation reordering for better cost profiles.

In real workloads, lazy evaluation reduces unnecessary compute by applying selectivity early and minimizing data movement before wide operations.

Optimization Window

Build plan->Optimize->Execute

Deferred execution improves global query efficiency.

7

Catalyst Optimizer

Catalyst transforms logical intent into efficient physical plans via rule-based and cost-aware passes. It performs simplification, pushdown, and operator-level rewrites.

Optimizer-friendly code is declarative, deterministic, and schema-explicit. Avoid opaque logic when built-ins can express intent.

Catalyst Pipeline

Logical plan->Optimization rules->Physical plan

Query clarity directly helps optimizer quality.

8

Jobs, Stages, and Tasks

An action creates a job. Jobs split into stages at shuffle boundaries. Stages split into tasks per partition. This hierarchy explains runtime behavior in Spark UI and cluster observability tools.

Slow stages usually indicate heavy shuffle or skew; long-tail tasks usually indicate uneven partition sizes or hot keys.

Execution Hierarchy

Job->Stages->Tasks

Partitioning strategy controls practical parallelism.

9

Join Strategies

Spark selects join strategies based on data shape and statistics. Broadcast hash joins suit small dimensions; sort-merge joins suit large-large joins; skew-aware handling is critical where key distributions are uneven.

Join strategy is often the highest-leverage performance decision in analytical pipelines.

Join Decision

Small + Large->Broadcast
Large + Large->Sort-Merge

Correct strategy minimizes network and shuffle cost.

10

Adaptive Query Execution (AQE)

AQE revises plans at runtime using real statistics from completed stages. It can coalesce partitions, alter join strategies, and split skewed partitions dynamically.

It improves latency consistency when incoming data distributions vary over time.

AQE Adaptation

Initial plan->Runtime stats->Adaptive plan

AQE aligns execution with actual data behavior.

11

Common File Formats

CSV/JSON are flexible at ingest but expensive for analytics. Parquet/ORC are columnar and support selective reads and compression. Mature lakehouse architectures use text formats at boundaries and columnar formats in curated layers.

Format selection should follow query behavior and workload profile, not habit.

Format Positioning

Ingest->CSV / JSON
Curated->Parquet / ORC

Store by access pattern to avoid wasted I/O.

12

Partitioning and Bucketing

Partitioning physically groups data by directory-level keys to prune scans early. Bucketing uses hash grouping to improve join/aggregation locality. Done well, both reduce expensive data movement.

Bad partition strategy causes tiny-file explosions or broad scans. Balance key cardinality with query selectivity.

Storage Layout

Table->Partitions->Buckets

Physical layout choices drive downstream performance.

13

Repartition and Coalesce

Repartition reshuffles to rebalance or increase partition count. Coalesce reduces partition count with minimal shuffle, usually before write operations.

Use repartition strategically after key-changing operations, and coalesce for output file quality and sink efficiency.

Partition Control

Imbalance->repartition()
Too many files->coalesce()

Partition shape controls speed and file layout.

14

Executor Memory

Executor memory is shared between execution and storage. Memory pressure appears as spills, long GC, and out-of-memory failures. Good tuning blends memory sizing with row-width control and partition sizing.

Increasing memory alone is not enough; workload structure and caching decisions are equally important.

Memory Split

Executor heap->Execution+Storage

Balanced usage prevents spill-heavy execution.

15

Dynamic Partition Pruning

Dynamic partition pruning applies join-derived runtime filters to skip irrelevant partitions before scanning large fact datasets. It is especially effective for star schema queries.

This reduces read amplification and improves query latency without major query rewrites.

Pruning Path

Filtered dimension->Prune fact partitions->Join

Prune first, then process less data.

16

Dynamic Resource Allocation

Dynamic allocation scales executors with workload pressure. It helps shared clusters stay efficient by adding capacity for bursts and reclaiming resources during idle periods.

Stable results require sensible min/max boundaries and healthy shuffle/service settings to avoid thrashing.

Elasticity Loop

Backlog up->Add executors
Backlog down->Remove executors

Elastic compute with guardrails improves cost/perf balance.
