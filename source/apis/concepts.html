API Concepts

REST, GraphQL, gRPC - core API concepts and patterns.

What is an API? Getting Started

An API (Application Programming Interface) is a set of rules and protocols that allows one software application to communicate with another. It defines the methods and data formats that applications can use to request and exchange information.

Think of an API like a waiter in a restaurant:

You (Client) → Waiter (API) → Kitchen (Server) Place order Carries request Prepares food ← ← ← Receive food Delivers response Returns result
  • You don't need to know how the kitchen works
  • The waiter knows exactly how to communicate your order
  • You get the result without seeing the internal process

Types of APIs Core

REST API

REST is the most widely used API architecture. It uses standard HTTP methods and is stateless — each request contains all the information needed to process it.

Core Principles:

  • Stateless — No session stored on the server; every request is independent
  • Client-Server — Clear separation between UI and data storage
  • Cacheable — Responses can be cached to improve performance
  • Uniform Interface — Consistent, predictable URL structure
  • Layered System — Can include load balancers, proxies, gateways

Example REST Request:

GET https://api.example.com/users/42 Authorization: Bearer eyJhbGc... Content-Type: application/json

Example REST Response:

{ "id": 42, "name": "Alice Johnson", "email": "alice@example.com", "role": "admin" }

GraphQL API

GraphQL is a query language for APIs developed by Facebook (Meta) in 2012. Instead of multiple endpoints, it uses a single endpoint where the client specifies exactly what data it needs.

Example GraphQL Query:

query { user(id: 42) { name email posts { title createdAt } } }

Key Insight: With REST, fetching a user's posts might require 2–3 separate requests. With GraphQL, it's a single query.

SOAP API

SOAP is a protocol (not just an architectural style) that uses XML for message formatting and typically runs over HTTP or SMTP. It is strict, formal, and has built-in error handling and security.

Where it's still used:

  • Banking and financial systems
  • Healthcare (HL7 integrations)
  • Government and enterprise legacy systems
  • Payment gateways

gRPC API

gRPC is a high-performance RPC framework developed by Google. It uses Protocol Buffers (protobuf) instead of JSON/XML for serialization, making it significantly faster.

Best for:

  • Microservices communicating internally
  • Low-latency, high-throughput systems
  • Streaming large amounts of data
  • Polyglot environments (supports 10+ languages)

WebSocket API

WebSocket provides a persistent, full-duplex communication channel between a client and server over a single TCP connection. Unlike HTTP, WebSocket allows the server to push data to the client at any time.

Use Cases:

  • Live chat applications (Slack, Discord)
  • Real-time dashboards and analytics
  • Multiplayer online games
  • Live sports scores and stock tickers

Webhooks

Webhooks are reverse APIs — instead of the client polling the server for updates, the server sends data to the client automatically when an event occurs.

Common Webhook Use Cases:

  • Payment confirmation (Stripe)
  • CI/CD pipelines (GitHub)
  • Form submissions
  • E-commerce order updates

Pros & Cons of Each API Type Intermediate

REST API

✅ Pros

  • Simple and easy to understand
  • Widely adopted
  • Works with HTTP caching
  • Language and platform agnostic
  • Great tooling (Postman, Swagger)

❌ Cons

  • Over-fetching (too much data)
  • Under-fetching (multiple requests)
  • No real-time support
  • Versioning can become complex

GraphQL API

✅ Pros

  • Fetch exactly the data you need
  • Single endpoint for all operations
  • Strongly typed schema
  • Reduces network requests

❌ Cons

  • Steeper learning curve
  • Complex queries can strain server
  • Harder to cache than REST
  • N+1 query problem

gRPC API

✅ Pros

  • Extremely fast (protobuf binary)
  • Supports streaming
  • Strongly typed contracts
  • Auto-generates client code

❌ Cons

  • Not human-readable
  • Limited browser support
  • Requires protobuf tooling
  • Not suitable for public APIs

What Problems Do APIs Solve? Getting Started

1. Interoperability

Problem: Different software systems, built in different languages and on different platforms, cannot naturally communicate.

How APIs solve it: APIs create a universal language between systems.

2. Modularity & Separation of Concerns

Problem: Monolithic applications become impossible to maintain and scale.

How APIs solve it: APIs allow teams to build independent services (microservices) that communicate through well-defined interfaces.

3. Reusability & Faster Development

Problem: Rebuilding functionality that already exists wastes time.

How APIs solve it: APIs allow developers to reuse existing functionality — integrate SendGrid for email, Stripe for payments, etc.

4. Security & Abstraction

Problem: Exposing internal systems directly is a security risk.

How APIs solve it: APIs act as a controlled gateway — authentication, rate-limiting, hiding internal implementation.
