API Types

Different API architectures and when to use each.

What is an API? Basic

An API (Application Programming Interface) is a contract between two software systems defining how they communicate. It abstracts internal implementation details and exposes only what is needed.

REST API Basic

REST (Representational State Transfer) is an architectural style defined by Roy Fielding in 2000. It uses standard HTTP methods and treats every resource as a uniquely addressable URL.

Core Constraints of REST

ConstraintDescription
StatelessEach request must contain all info needed; server stores no session
Client-ServerUI and data storage are decoupled
CacheableResponses define themselves as cacheable or not
Uniform InterfaceConsistent resource-based URLs and HTTP verbs
Layered SystemClient cannot tell if connected directly or through proxy

REST Resource URL Design

Good REST URL structure: GET /users → List all users POST /users → Create a new user GET /users/{id} → Get user by ID PUT /users/{id} → Replace user PATCH /users/{id} → Update user fields DELETE /users/{id} → Delete user GET /users/{id}/posts → Get all posts by user

check_circle Pros

Simplicity

Widely understood and easy to implement across all platforms.

Caching

Native HTTP caching support for high performance.

Scalability

Stateless architecture allows for easy horizontal scaling.

cancel Cons

Data Fetching

Common issues with over-fetching or under-fetching data.

Real-time

Lacks native support for push notifications or real-time data.

Versioning

API versioning can lead to maintenance complexity over time.

GraphQL API Intermediate

GraphQL is a query language for APIs developed by Facebook (Meta) in 2012. Unlike REST's fixed endpoints, GraphQL exposes a single endpoint where clients specify exactly the data they need.

GraphQL Core Concepts

ConceptDescription
SchemaStrongly-typed definition of all data and operations
QueryRead operation — fetch data
MutationWrite operation — create, update, delete
SubscriptionReal-time operation — subscribe to events
ResolverFunction that fetches data for each field

GraphQL Query Example

query GetUserWithPosts { user(id: "42") { name email posts { title tags } } }

GraphQL — Pros & Cons

check_circle Pros

Precision

Clients fetch exactly the data they need, reducing payload size.

Single Endpoint

Access all data and operations through a single gateway.

Typed Schema

Strongly-typed contracts between client and server.

cancel Cons

Complexity

Steeper learning curve and more difficult server-side setup.

Caching

Harder to implement standard HTTP caching for flexible queries.

Server Load

Complex queries can significantly strain backend resources.

SOAP API Intermediate

SOAP (Simple Object Access Protocol) is a messaging protocol using XML for message format and HTTP, SMTP, or TCP for transport. It's a formal protocol with a strict specification.

SOAP Message Structure

<?xml version="1.0" encoding="UTF-8"?> <soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/"> <soap:Header> <AuthToken>Bearer abc123token</AuthToken> </soap:Header> <soap:Body> <GetAccountBalance xmlns="http://bank.example.com/services"> <AccountNumber>1234567890</AccountNumber> <Currency>USD</Currency> </GetAccountBalance> </soap:Body> </soap:Envelope>

SOAP Security Standards

StandardPurpose
WS-SecurityMessage-level encryption and signing
WS-TrustToken-based authentication (SAML, X.509)
WS-ReliableMessagingGuaranteed message delivery
WS-AtomicTransactionDistributed ACID transactions

check_circle Pros

Formal Contract

Strict interface definition through WSDL and XML schemas.

Security

Built-in WS-Security standards for message-level encryption.

Transactions

Native support for distributed ACID transactions (WS-Atomic).

cancel Cons

Overhead

Extremely verbose XML leading to increased payload size.

Complexity

Steep learning curve to implement and troubleshoot protocols.

Performance

Slow parsing and processing speeds compared to REST/gRPC.

cancel Cons

Debugging

WebSocket API Intermediate

WebSocket provides full-duplex (bidirectional), persistent connections over a single TCP connection. After an initial HTTP handshake, the connection is upgraded.

WebSocket Handshake

-- Client HTTP Upgrade Request -- GET /chat HTTP/1.1 Host: ws.example.com Upgrade: websocket Connection: Upgrade Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ== -- Server Upgrade Response -- HTTP/1.1 101 Switching Protocols Upgrade: websocket Connection: Upgrade

When to Use WebSocket

Live Chat

Instant messaging platforms like Slack, Discord, or WhatsApp Web requiring zero latency.

Online Gaming

Multiplayer games where rapid state synchronization is critical for gameplay.

Collaborative Tools

Real-time editing environments like Google Docs, Figma, or interactive whiteboards.

Financial Feeds

High-frequency data streams for stock tickers, crypto prices, and trading terminals.

Webhooks Intermediate

A Webhook is a user-defined HTTP callback — a reverse API. Instead of polling, the server sends an HTTP POST to the client's URL when an event occurs.

Webhook vs Polling

❌ Polling (Inefficient): Client → "Any new payments?" → Server: "No" Client → "Any new payments?" → Server: "No" Client → "Any new payments?" → Server: "Yes! Here's the data" ✅ Webhook (Efficient): Server → POST /your-webhook-url → Client (Instantly, only when event fires)

Webhook Payload Example (Stripe)

{ "id": "evt_1234567890", "type": "payment_intent.succeeded", "created": 1609459200, "data": { "object": { "id": "pi_abc123", "amount": 4999, "currency": "usd", "status": "succeeded" } } }

When to Use Webhooks

Payment Events

Capturing subscription renewals, successful charges, or chargebacks from Stripe/PayPal.

Automation Triggers

Triggering CI/CD pipelines (GitHub) or Zapier workflows when a specific event occurs.

Form Routing

Instantly sending new lead data from a landing page to a CRM or email service.

SSE Intermediate

Server-Sent Events (SSE) is a server push technology over a standard HTTP connection. The server streams events to the client. Unlike WebSocket, SSE is one-directional (server → client only).

When to Use SSE

Activity Streams

Social media feeds, news updates, or real-time notification centers.

Live Analytics

Streaming log data or server performance metrics to a browser dashboard.

Scoreboards

Displaying live sports scores or election results without requiring user refresh.

JSON-RPC Intermediate

JSON-RPC is a lightweight, stateless remote procedure call protocol encoded in JSON.

JSON-RPC Request/Response

-- Request -- { "jsonrpc": "2.0", "method": "getUserById", "params": { "id": 42 }, "id": 1 } -- Response -- { "jsonrpc": "2.0", "result": { "id": 42, "name": "Alice" }, "id": 1 }

When to Use JSON-RPC

Blockchain Nodes

Querying Ethereum, Bitcoin, or Solana nodes for transaction data and smart contract state.

Internal Services

Extremely lightweight internal micro-services where REST overhead is unnecessary.

API Gateway Intermediate

An API Gateway is a server that acts as the single entry point for all client requests. It handles cross-cutting concerns like authentication, rate limiting, logging, and routing.

API Gateway Responsibilities

FeatureDescription
RoutingDirect requests to the right service
AuthenticationVerify JWT, API keys, OAuth tokens
Rate LimitingPrevent abuse (100 req/min per user)
SSL TerminationHandle HTTPS, pass plain HTTP internally
CachingCache responses to reduce backend load
LoggingCentralized observability

OpenAPI / Swagger Intermediate

OpenAPI (formerly Swagger) is a specification standard for describing REST APIs in machine-readable YAML or JSON. It enables automatic generation of documentation, client SDKs, and server stubs.
