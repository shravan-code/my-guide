Apache Kafka

Distributed event streaming platform for real-time data.

Introduction

Apache Kafka is an open-source distributed event streaming platform used by thousands of companies for high-performance data pipelines, streaming analytics, data integration, and mission-critical applications. Originally developed at LinkedIn and donated to the Apache Software Foundation in 2011, Kafka has become the de facto standard for real-time data streaming at massive scale.

What is Kafka?

Kafka is a distributed publish-subscribe messaging system that can handle trillions of events per day. It is designed around three key capabilities:

send

Publish and Subscribe

Publish and subscribe to streams of records, similar to a message queue or enterprise messaging system.

storage

Store

Store streams of records durably and reliably in a fault-tolerant way.

bolt

Process

Process streams of records as they occur in real-time.

Why Kafka?

Traditional databases and message queues struggle with high-throughput, low-latency requirements at scale. Kafka was built specifically to solve these challenges.

Core Concepts & Architecture

Kafka Cluster Overview

A Kafka deployment is called a cluster. A cluster consists of multiple broker servers that coordinate with each other using Apache ZooKeeper (or KRaft in newer versions) for distributed coordination.

dns

Broker

A Kafka broker is a server that stores data and serves client requests. Each broker is identified by a unique integer ID. A typical production cluster has 3-10+ brokers for fault tolerance and throughput.

  • Handles read and write requests from clients
  • Manages partition assignment and replication
  • Each broker can handle hundreds of thousands of reads/writes per second
  • Stateless — data is stored in logs on disk
storage

ZooKeeper / KRaft

ZooKeeper was traditionally used for cluster metadata management, leader election, and configuration storage. Kafka 3.x+ introduces KRaft (Kafka Raft) mode which eliminates the ZooKeeper dependency, simplifying operations significantly.

stars Kafka 3.3+ supports KRaft mode as production-ready. New deployments should prefer KRaft over ZooKeeper.

Topics

A topic is a named stream of records. Topics in Kafka are multi-subscriber — meaning a single topic can be consumed by any number of consumer groups simultaneously.

Partitions

Partitions are the fundamental unit of parallelism in Kafka. Each topic is split into one or more partitions. Partitions allow Kafka to scale horizontally by distributing data across multiple brokers.

check_circle

Ordering

Messages within a partition are strictly ordered by offset

warning

Note

Ordering is NOT guaranteed across different partitions

tune

Parallelism

Partition count determines maximum consumer parallelism

Partition Assignment

When a producer sends a message, it chooses which partition to write to based on:

  • Explicit partition key — Kafka hashes the key to determine partition (same key always → same partition)
  • Round-robin — if no key is specified, messages are distributed evenly across partitions
  • Custom partitioner — developer-defined partitioning logic

Offsets

An offset is a unique, sequential integer assigned to every message within a partition. Offsets are immutable and always increasing. Consumers track their position in each partition using offsets.

info Offsets are partition-specific, not topic-wide. Partition 0 offset 5 is a completely different message than Partition 1 offset 5.

Producers

Producers are client applications that publish (write) messages to Kafka topics. Producers are responsible for deciding which partition to write to.

Consumers & Consumer Groups

Consumers are applications that read (consume) messages from Kafka topics. Consumer Groups enable parallel consumption — multiple consumers working together to process a topic.

  • Each partition is consumed by exactly ONE consumer within a group
  • A consumer can read from multiple partitions
  • Multiple groups can independently consume the same topic
  • Adding consumers beyond partition count provides no parallelism benefit

Consumer Group Rebalancing

When consumers join or leave a group, Kafka triggers a rebalance to redistribute partitions:

Protocol Description
Eager Stop-the-world — all consumers stop, give up partitions, get reassigned
Cooperative Incremental — only partitions that need to move are reassigned (preferred for production)
Static Membership Consumers rejoin with the same partition without triggering a full rebalance

Replication & Fault Tolerance

Kafka replicates each partition across multiple brokers for fault tolerance. The replication factor determines how many copies exist.

person

Leader

Each partition has one leader — all reads and writes go to the leader

group

Followers

Followers replicate data from the leader

swap_horiz

Failover

If a leader fails, a follower is automatically elected as the new leader

Basic Producer — confluent-kafka

from confluent_kafka import Producer import json import logging logging.basicConfig(level=logging.INFO) logger = logging.getLogger(__name__) conf = { 'bootstrap.servers': 'localhost:9092', 'client.id': 'my-python-producer', 'acks': 'all', 'enable.idempotence': True, 'retries': 2147483647, 'max.in.flight.requests.per.connection': 5, 'compression.type': 'lz4', 'batch.size': 65536, 'linger.ms': 10, } producer = Producer(conf) def delivery_report(err, msg): if err is not None: logger.error(f'Message delivery failed: {err}') else: logger.info(f'Message delivered to {msg.topic()} [{msg.partition()}] at {msg.offset()}') events = [ {'user_id': 'u001', 'action': 'page_view', 'page': '/home'}, {'user_id': 'u002', 'action': 'purchase', 'amount': 49.99}, {'user_id': 'u001', 'action': 'click', 'element': 'buy-btn'}, ] for event in events: producer.produce( topic='user-events', key=event['user_id'], value=json.dumps(event), callback=delivery_report ) producer.poll(0) producer.flush(timeout=10)

Basic Consumer — confluent-kafka

from confluent_kafka import Consumer, KafkaError, KafkaException import json import signal import logging logger = logging.getLogger(__name__) conf = { 'bootstrap.servers': 'localhost:9092', 'group.id': 'analytics-consumer-group', 'client.id': 'analytics-consumer-1', 'auto.offset.reset': 'earliest', 'enable.auto.commit': False, 'max.poll.interval.ms': 300000, 'session.timeout.ms': 45000, 'heartbeat.interval.ms': 3000, } consumer = Consumer(conf) running = True def shutdown(signum, frame): global running running = False signal.signal(signal.SIGINT, shutdown) signal.signal(signal.SIGTERM, shutdown) try: consumer.subscribe(['user-events']) logger.info('Consumer started. Listening on user-events...') while running: msg = consumer.poll(timeout=1.0) if msg is None: continue if msg.error(): if msg.error().code() == KafkaError._PARTITION_EOF: logger.debug(f'End of partition: {msg.topic()} [{msg.partition()}] at {msg.offset()}') else: raise KafkaException(msg.error()) else: event = json.loads(msg.value().decode('utf-8')) key = msg.key().decode('utf-8') if msg.key() else None logger.info(f'[Partition {msg.partition()} | Offset {msg.offset()}] Key: {key} Event: {event}') consumer.commit(asynchronous=False) finally: consumer.close()

Avro Serialization with Schema Registry

Apache Avro is the recommended serialization format for Kafka. The Confluent Schema Registry stores and manages Avro schemas.

from confluent_kafka import SerializingProducer from confluent_kafka.schema_registry import SchemaRegistryClient from confluent_kafka.schema_registry.avro import AvroSerializer from confluent_kafka.serialization import StringSerializer schema_str = """ { "type": "record", "name": "UserEvent", "namespace": "com.example.events", "fields": [ {"name": "user_id", "type": "string"}, {"name": "action", "type": "string"}, {"name": "timestamp", "type": "long"}, {"name": "amount", "type": ["null", "double"], "default": null} ] } """ schema_registry_client = SchemaRegistryClient({'url': 'http://localhost:8081'}) avro_serializer = AvroSerializer(schema_registry_client, schema_str) producer_conf = { 'bootstrap.servers': 'localhost:9092', 'key.serializer': StringSerializer('utf_8'), 'value.serializer': avro_serializer, } producer = SerializingProducer(producer_conf) producer.produce(topic='user-events-avro', key='u001', value={ 'user_id': 'u001', 'action': 'purchase', 'timestamp': 1735689600, 'amount': 99.99 }) producer.flush()

import faust app = faust.App( 'order-processor', broker='kafka://localhost:9092', value_serializer='json', ) class Order(faust.Record, serializer='json'): order_id: str user_id: str amount: float status: str orders_topic = app.topic('orders', value_type=Order) enriched_topic = app.topic('orders-enriched', value_type=Order) user_spend = app.Table('user-total-spend', default=float) @app.agent(orders_topic) async def process_orders(orders): async for order in orders: print(f'Processing order {order.order_id} for user {order.user_id}') user_spend[order.user_id] += order.amount await enriched_topic.send(key=order.order_id, value=order) @app.timer(interval=60.0) async def log_top_spenders(): sorted_users = sorted(user_spend.items(), key=lambda x: x[1], reverse=True) for user_id, total in sorted_users[:10]: print(f'Top spender: {user_id} - ${total:.2f}') if __name__ == '__main__': app.main()

Exactly-Once Semantics

Kafka supports three message delivery guarantees. Understanding them is critical for building reliable applications.

Guarantee Description Risk
At Most Once Messages may be lost, never duplicated Data loss
At Least Once Messages never lost, may be duplicated Duplicates
Exactly Once Never lost, never duplicated Higher latency

Transactional Producer

from confluent_kafka import Producer producer = Producer({ 'bootstrap.servers': 'localhost:9092', 'transactional.id': 'payment-producer-v1', 'enable.idempotence': True, 'acks': 'all', }) producer.init_transactions() try: producer.begin_transaction() producer.produce('payments', key='pay001', value='{"amount": 100}') producer.produce('audit-log', key='pay001', value='{"event": "payment_created"}') producer.commit_transaction() print('Transaction committed successfully') except Exception as e: print(f'Transaction failed: {e}') producer.abort_transaction()

Topic Compaction

Log compaction is an alternative to time-based retention. In a compacted topic, Kafka keeps only the latest message for each unique key.

# Create a compacted topic kafka-topics.sh --bootstrap-server localhost:9092 \ --create --topic user-profiles \ --partitions 6 \ --replication-factor 3 \ --config cleanup.policy=compact \ --config min.cleanable.dirty.ratio=0.1 \ --config segment.ms=3600000 # Update existing topic to add compaction kafka-configs.sh --bootstrap-server localhost:9092 \ --entity-type topics --entity-name user-profiles \ --alter --add-config cleanup.policy=compact

from confluent_kafka import Consumer, TopicPartition consumer = Consumer({ 'bootstrap.servers': 'localhost:9092', 'group.id': 'offset-demo-group', 'auto.offset.reset': 'earliest', 'enable.auto.commit': False, }) consumer.subscribe(['user-events']) # Manually seek to specific offset def on_assign(consumer, partitions): for p in partitions: p.offset = 100 consumer.assign(partitions) consumer.subscribe(['user-events'], on_assign=on_assign) # Or seek to beginning / end partitions = consumer.assignment() consumer.seek_to_beginning(*partitions) consumer.seek_to_end(*partitions) # Commit specific offsets msg = consumer.poll(1.0) if msg: tp = TopicPartition(msg.topic(), msg.partition(), msg.offset() + 1) consumer.commit(offsets=[tp], asynchronous=False)

Best Practices

Topic Design Guidelines

  • Partitions: Start with 3x the number of consumers you expect. Prefer multiples of 3 for even distribution.
  • Replication Factor: Always use 3 in production. Never use 1.
  • Naming Convention: Use kebab-case with clear domain prefixes (e.g., orders-created, payments-processed).
  • Message Size: Keep messages under 1MB. Use external storage (S3) for large payloads.
  • Keys: Always use meaningful keys for ordering guarantees.
