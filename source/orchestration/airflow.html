Apache Airflow

Workflow orchestration platform for programmatically authoring, scheduling, and monitoring workflows.

Introduction

Apache Airflow is a workflow orchestration platform that helps you programmatically author, schedule, and monitor workflows. Think of Airflow as a tool that helps you create, schedule, and track tasks. These workflows are defined as Directed Acyclic Graphs (DAGs), where each node represents a task and edges define dependencies between tasks.

At its core, Airflow is designed to help you automate and manage ETL (Extract, Transform, Load) processes, data pipelines, and any kind of batch processing.

Understanding Cron

Before diving deep into Airflow, you must understand cron expressions. Cron is a time-based job scheduler in Unix-like operating systems that automates the execution of scripts, commands, or programs at scheduled times or intervals. Airflow uses cron expressions extensively to define when your DAGs execute.

Cron Expression Format

* Minute 0-59
* Hour 0-23
* Day 1-31
* Month 1-12
* Weekday 0-7

Core Components

DAG (Directed Acyclic Graph)

At the heart of Airflow lies the DAG. A DAG represents the workflow you want to schedule and run. Each DAG is composed of tasks, which define the work to be done.

The "directed" and "acyclic" properties mean:

  • Directed — tasks flow in a specific direction (from one task to another)
  • Acyclic — there are no cycles: a task cannot depend on itself or create a circular dependency

Tasks and Task Instances

Each DAG consists of tasks — the individual units of work. A task in Airflow is defined by an Operator, which determines the nature of the work performed.

Each time a DAG runs, each task creates a Task Instance. A task instance represents the execution of a task at a specific point in time and includes its state.

Processes that execute the tasks assigned by the Executor. Each worker pulls tasks from the queue and executes them.

Introduced in Airflow 2.2. Handles tasks asynchronously using Python's asyncio for long-running sensor tasks.

Task States

State Description
none Task has not yet been queued
scheduled Task is ready and waiting to be queued
queued Task is in the queue, waiting for a worker
running Task is currently being executed
success Task completed successfully
failed Task encountered an error and failed
skipped Task was skipped (e.g., due to branching)
up_for_retry Task failed but will be retried

Operators

In Apache Airflow, an Operator is a predefined template that encapsulates the logic of a specific task in a workflow.

Operator Categories

Category Purpose Examples
Action Operators Perform an action: run a function, command, or API request PythonOperator, BashOperator, EmailOperator
Transfer Operators Move data from one system to another S3ToRedshiftOperator, MySqlToS3Operator
Sensor Operators Wait for a condition to be met before proceeding FileSensor, HttpSensor, S3KeySensor

PythonOperator

Execute any Python callable as a task.

from airflow.operators.python import PythonOperator def process_data(**kwargs): execution_date = kwargs["execution_date"] print(f"Processing data for: {execution_date}") return "Data processed successfully" with DAG("python_operator_example", ...) as dag: process_task = PythonOperator( task_id="process_data_task", python_callable=process_data, provide_context=True, )

BashOperator

Run bash shell commands or scripts.

from airflow.operators.bash import BashOperator with DAG("bash_operator_example", ...) as dag: list_files = BashOperator( task_id="list_files", bash_command="ls -la /data/input/", ) run_script = BashOperator( task_id="run_etl_script", bash_command="/scripts/etl_pipeline.sh {{ ds }}", env={"DB_HOST": "localhost"}, ) list_files >> run_script

HttpOperator

Send HTTP requests to interact with REST APIs.

Sensor Operators

Wait for a certain condition to be met before allowing downstream tasks to proceed.

from airflow.sensors.filesystem import FileSensor from airflow.providers.http.sensors.http import HttpSensor from airflow.providers.amazon.aws.sensors.s3 import S3KeySensor # Wait for a file wait_for_file = FileSensor( task_id="wait_for_input_file", filepath="/data/input/daily_report.csv", poke_interval=30, timeout=3600, ) # Wait for an API check_api = HttpSensor( task_id="check_api_health", http_conn_id="weather_api", endpoint="/health", poke_interval=10, timeout=300, ) # Wait for S3 object wait_for_s3 = S3KeySensor( task_id="wait_for_s3_file", bucket_name="my-data-bucket", bucket_key="data/{{ ds }}/input.csv", aws_conn_id="aws_default", poke_interval=120, timeout=3600, )

Building Your First DAG

from airflow import DAG from airflow.operators.python import PythonOperator from airflow.operators.bash import BashOperator from datetime import datetime, timedelta # Define default arguments default_args = { "owner": "airflow", "depends_on_past": False, "start_date": datetime(2024, 1, 1), "email_on_failure": False, "email_on_retry": False, "retries": 1, "retry_delay": timedelta(minutes=5), } # Define Python functions def hello_world(**kwargs): print("Hello from Apache Airflow!") print(f"Execution date: {kwargs['execution_date']}") return "Hello World!" def process_data(**kwargs): ti = kwargs["ti"] previous_result = ti.xcom_pull(task_ids="print_hello") print(f"Previous task returned: {previous_result}") print("Processing data...") # Create the DAG with DAG( dag_id="first_dag", default_args=default_args, description="My very first Airflow DAG", schedule_interval=timedelta(days=1), catchup=False, tags=["example", "beginner"], ) as dag: start = BashOperator( task_id="start_pipeline", bash_command="echo 'Pipeline started at $(date)'", ) hello_task = PythonOperator( task_id="print_hello", python_callable=hello_world, provide_context=True, ) process_task = PythonOperator( task_id="process_data", python_callable=process_data, provide_context=True, ) end = BashOperator( task_id="end_pipeline", bash_command="echo 'Pipeline completed successfully'", ) start >> hello_task >> process_task >> end

DAG Dependency Syntax

# Method 1: Bitshift operators (recommended) task_a >> task_b >> task_c task_a >> [task_b, task_c] [task_b, task_c] >> task_d # Method 2: Explicit set_upstream / set_downstream task_b.set_upstream(task_a) task_b.set_downstream(task_c)

Advanced DAG Concepts

Branching with BranchPythonOperator

from airflow.operators.python import BranchPythonOperator from airflow.operators.empty import EmptyOperator import random def decide_branch(**kwargs): execution_hour = kwargs["execution_date"].hour if execution_hour < 12: return "morning_pipeline" else: return "evening_pipeline" with DAG("branching_dag", ...) as dag: start = EmptyOperator(task_id="start") branch = BranchPythonOperator( task_id="decide_branch", python_callable=decide_branch, provide_context=True, ) morning = BashOperator(task_id="morning_pipeline", bash_command="echo 'Running morning ETL'") evening = BashOperator(task_id="evening_pipeline", bash_command="echo 'Running evening ETL'") join = EmptyOperator(task_id="join", trigger_rule="none_failed_or_skipped") start >> branch >> [morning, evening] >> join

Trigger Rules

Trigger Rule Behavior
all_success (default) Triggers only if ALL upstream tasks succeeded
all_failed Triggers only if ALL upstream tasks failed
all_done Triggers once ALL upstream tasks are done (any state)
one_failed Triggers if ANY upstream task fails
one_success Triggers if ANY upstream task succeeds
none_failed_or_skipped Triggers if no upstream tasks failed or were skipped

XCom — Cross-Task Communication

# Task 1: Push data to XCom def push_data(**kwargs): result = {"records_processed": 1500, "status": "success"} kwargs["ti"].xcom_push(key="processing_result", value=result) return result # Task 2: Pull data from Task 1 def pull_data(**kwargs): ti = kwargs["ti"] result = ti.xcom_pull(task_ids="push_task", key="processing_result") print(f"Records processed: {result['records_processed']}")

info XComs are designed for SMALL data (metadata, IDs, flags). Do NOT use XCom to pass large DataFrames or binary data.

Hooks and Sensors

What are Hooks?

Hooks are interfaces to external systems that provide methods for connecting, interacting, and executing commands.

from airflow.providers.http.hooks.http import HttpHook def fetch_data_via_hook(**kwargs): hook = HttpHook(method="GET", http_conn_id="weather_api") response = hook.run( endpoint="/data/2.5/weather", data={"q": "Seattle", "units": "metric"}, ) return response.json() # Example: Using PostgresHook from airflow.providers.postgres.hooks.postgres import PostgresHook def query_database(**kwargs): hook = PostgresHook(postgres_conn_id="my_postgres") records = hook.get_records("SELECT id, name FROM users LIMIT 100") for row in records: print(f"ID: {row[0]}, Name: {row[1]}")

Connections and Variables

Connections

Connections store credentials and configuration for external systems (databases, APIs, cloud services).

# Creating a Connection programmatically from airflow.models import Connection from airflow import settings new_conn = Connection( conn_id="my_postgres", conn_type="postgres", host="db.example.com", login="admin", password="secret", schema="my_database", port=5432, ) session = settings.Session() session.add(new_conn) session.commit() # Using environment variables # AIRFLOW_CONN_MY_POSTGRES="postgresql://admin:secret@db.example.com:5432/my_database" # Using in Hooks hook = PostgresHook(postgres_conn_id="my_postgres") records = hook.get_records("SELECT * FROM orders")
