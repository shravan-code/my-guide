SQL Window Functions

Perform calculations across sets of rows related to the current row without collapsing results.

Window Function Concepts Basic

Window functions perform calculations across a "window" of rows related to the current row. Unlike aggregates, they don't collapse rows.

Window vs Aggregate

-- AGGREGATE: Collapses rows SELECT department, AVG(salary) AS avg_salary FROM employees GROUP BY department; -- Returns 1 row per department -- WINDOW: Preserves rows SELECT name, department, salary, AVG(salary) OVER ( PARTITION BY department ) AS dept_avg FROM employees; -- Returns all rows with dept average

Key Concepts

  • Window - Set of rows related to current row
  • Partition - Group of rows (like GROUP BY)
  • Frame - Subset of partition for calculation
  • Current Row - The row being processed
  • OVER clause - Defines the window

Key difference: Window functions return a value for every row, while aggregates return one value per group. Use window functions when you need both detail rows and summary calculations.

OVER Clause Syntax Basic

The OVER clause defines the window over which the function operates.

-- Basic OVER syntax function_name() OVER window_definition -- Window definition components OVER ( [PARTITION BY column1, column2, ...] [ORDER BY column [ASC|DESC], ...] [frame_clause] ) -- Empty OVER() - entire table as window SELECT name, salary, SUM(salary) OVER () AS total_payroll FROM employees; -- ORDER BY only - defines order within window SELECT name, hire_date, ROW_NUMBER() OVER (ORDER BY hire_date) AS hire_order FROM employees; -- PARTITION BY only - groups rows SELECT name, department, salary, AVG(salary) OVER (PARTITION BY department) AS dept_avg FROM employees; -- Both PARTITION BY and ORDER BY SELECT name, department, salary, SUM(salary) OVER ( PARTITION BY department ORDER BY hire_date ) AS running_dept_total FROM employees;

PARTITION BY Basic

PARTITION BY divides rows into groups (partitions) for the window function. Each partition is processed independently.

Single Partition

-- Rank within each department SELECT name, department, salary, RANK() OVER ( PARTITION BY department ORDER BY salary DESC ) AS dept_rank FROM employees; -- Count per department SELECT name, department, COUNT(*) OVER (PARTITION BY department) AS dept_size FROM employees;

Multiple Partitions

-- Partition by multiple columns SELECT name, department, region, salary, AVG(salary) OVER ( PARTITION BY department, region ) AS dept_region_avg FROM employees; -- No partition = entire table SELECT name, salary, SUM(salary) OVER () AS company_total, salary * 100.0 / SUM(salary) OVER () AS pct_of_total FROM employees;
-- Multiple window functions with different partitions SELECT name, department, salary, RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank, RANK() OVER (ORDER BY salary DESC) AS company_rank, AVG(salary) OVER (PARTITION BY department) AS dept_avg, AVG(salary) OVER () AS company_avg FROM employees;

ORDER BY in Windows Basic

ORDER BY within OVER determines the order of rows within each partition. Required for ranking and cumulative calculations.

-- ORDER BY is required for ranking functions SELECT name, salary, ROW_NUMBER() OVER (ORDER BY salary DESC) AS overall_rank, ROW_NUMBER() OVER (ORDER BY hire_date) AS hire_order FROM employees; -- ORDER BY affects running totals SELECT order_date, amount, SUM(amount) OVER (ORDER BY order_date) AS running_total FROM orders; -- Multiple ORDER BY columns SELECT department, salary, hire_date, ROW_NUMBER() OVER ( PARTITION BY department ORDER BY salary DESC, hire_date ASC ) AS rank FROM employees; -- NULLS FIRST/LAST (PostgreSQL) SELECT name, bonus, ROW_NUMBER() OVER (ORDER BY bonus NULLS LAST) AS rank FROM employees;

ROW_NUMBER Basic

Assigns a unique sequential integer to each row within a partition. Ties get different numbers.

-- Simple row numbering SELECT name, salary, ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num FROM employees; -- Row number per department SELECT name, department, salary, ROW_NUMBER() OVER ( PARTITION BY department ORDER BY salary DESC ) AS dept_row_num FROM employees; -- Get top N per group (PostgreSQL pattern) WITH ranked AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY department ORDER BY salary DESC ) AS rn FROM employees ) SELECT * FROM ranked WHERE rn <= 3; -- Top 3 per department -- Alternative: DISTINCT ON (PostgreSQL) SELECT DISTINCT ON (department) * FROM employees ORDER BY department, salary DESC; -- Random row selection SELECT * FROM ( SELECT *, ROW_NUMBER() OVER () AS rn FROM products ) t WHERE rn = FLOOR(RANDOM() * (SELECT COUNT(*) FROM products)) + 1;

RANK and DENSE_RANK Basic

Rank rows based on values. RANK skips numbers after ties; DENSE_RANK does not.

RANK vs DENSE_RANK

SELECT name, salary, RANK() OVER (ORDER BY salary DESC) AS rank, DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank FROM employees; -- Results: -- salary rank dense_rank -- 100000 1 1 -- 100000 1 1 -- 90000 3 2 ← RANK skips 2 -- 80000 4 3

Use Cases

-- Find salary ties SELECT name, salary FROM ( SELECT name, salary, RANK() OVER (ORDER BY salary DESC) AS rnk FROM employees ) t WHERE rnk IN ( SELECT RANK() OVER (ORDER BY salary DESC) FROM employees GROUP BY salary HAVING COUNT(*) > 1 ); -- Dense rank per department SELECT name, department, salary, DENSE_RANK() OVER ( PARTITION BY department ORDER BY salary DESC ) AS dept_rank FROM employees;

NTILE Intermediate

Distributes rows into a specified number of roughly equal groups (buckets).

-- Divide into quartiles (4 groups) SELECT name, salary, NTILE(4) OVER (ORDER BY salary DESC) AS quartile FROM employees; -- Quintiles (5 groups) SELECT name, salary, NTILE(5) OVER (ORDER BY salary DESC) AS quintile FROM employees; -- Deciles (10 groups) SELECT name, salary, NTILE(10) OVER (ORDER BY salary) AS decile FROM employees; -- Salary percentiles using NTILE SELECT NTILE(100) OVER (ORDER BY salary) AS percentile, MIN(salary) AS min_salary, MAX(salary) AS max_salary, AVG(salary) AS avg_salary FROM employees GROUP BY NTILE(100) OVER (ORDER BY salary) ORDER BY percentile; -- NTILE per department SELECT name, department, salary, NTILE(3) OVER ( PARTITION BY department ORDER BY salary DESC ) AS salary_tercile FROM employees;

LAG and LEAD Intermediate

Access values from previous (LAG) or following (LEAD) rows without self-joins.

LAG - Previous Rows

-- Previous row's salary SELECT name, salary, LAG(salary) OVER (ORDER BY hire_date) AS prev_salary FROM employees; -- With offset (2 rows back) SELECT name, salary, LAG(salary, 2) OVER (ORDER BY hire_date) AS salary_2_back FROM employees; -- With default value SELECT name, salary, LAG(salary, 1, 0) OVER (ORDER BY hire_date) AS prev_salary FROM employees;

LEAD - Following Rows

-- Next row's salary SELECT name, salary, LEAD(salary) OVER (ORDER BY hire_date) AS next_salary FROM employees; -- Compare with next row SELECT order_date, amount, LEAD(amount) OVER (ORDER BY order_date) AS next_amount, LEAD(amount) OVER (ORDER BY order_date) - amount AS change FROM orders; -- Within partitions SELECT name, department, hire_date, LAG(name) OVER ( PARTITION BY department ORDER BY hire_date ) AS previous_hire FROM employees;
-- Calculate days between events SELECT customer_id, order_date, LAG(order_date) OVER ( PARTITION BY customer_id ORDER BY order_date ) AS prev_order, order_date - LAG(order_date) OVER ( PARTITION BY customer_id ORDER BY order_date ) AS days_between FROM orders; -- Compare with previous and next SELECT name, salary, LAG(salary, 1, 0) OVER (ORDER BY salary) AS prev, LEAD(salary, 1, 0) OVER (ORDER BY salary) AS next, salary - LAG(salary, 1, 0) OVER (ORDER BY salary) AS diff_prev, LEAD(salary, 1, 0) OVER (ORDER BY salary) - salary AS diff_next FROM employees ORDER BY salary;

FIRST_VALUE and LAST_VALUE Intermediate

Access the first or last value in the window frame.

FIRST_VALUE

-- First value in partition SELECT name, department, salary, FIRST_VALUE(name) OVER ( PARTITION BY department ORDER BY salary DESC ) AS highest_paid FROM employees; -- Compare each row to first SELECT name, salary, FIRST_VALUE(salary) OVER (ORDER BY hire_date) AS first_salary, salary - FIRST_VALUE(salary) OVER (ORDER BY hire_date) AS diff_from_first FROM employees;

LAST_VALUE (careful!)

-- LAST_VALUE needs explicit frame -- WRONG: Only sees current row SELECT name, salary, LAST_VALUE(name) OVER (ORDER BY salary) AS last_name FROM employees; -- CORRECT: Explicit frame SELECT name, salary, LAST_VALUE(name) OVER ( ORDER BY salary ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ) AS last_name FROM employees; -- Alternative: Use FIRST_VALUE with reverse ORDER SELECT name, salary, FIRST_VALUE(name) OVER ( ORDER BY salary DESC ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ) AS last_name FROM employees;

Warning: The default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. This means LAST_VALUE only sees up to the current row. Use an explicit frame clause to see the true last value.

NTH_VALUE Intermediate

Access the nth value in the window frame. Also requires proper framing to work correctly.

-- Nth value with proper frame SELECT name, salary, NTH_VALUE(name, 2) OVER ( ORDER BY salary DESC ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ) AS second_highest FROM employees; -- Top 3 values per department SELECT department, name, salary, NTH_VALUE(name, 1) OVER w AS top_1, NTH_VALUE(name, 2) OVER w AS top_2, NTH_VALUE(name, 3) OVER w AS top_3 FROM employees WINDOW w AS ( PARTITION BY department ORDER BY salary DESC ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ); -- Compare to first, second, third values SELECT name, salary, NTH_VALUE(salary, 1) OVER full_window AS first_val, NTH_VALUE(salary, 2) OVER full_window AS second_val, NTH_VALUE(salary, 3) OVER full_window AS third_val FROM employees WINDOW full_window AS ( ORDER BY salary DESC ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING );

Aggregate Window Functions Basic

Standard aggregate functions (SUM, AVG, COUNT, MIN, MAX) can be used as window functions.

-- Running total SELECT order_date, amount, SUM(amount) OVER (ORDER BY order_date) AS running_total FROM orders; -- Running count SELECT name, hire_date, COUNT(*) OVER (ORDER BY hire_date) AS employees_hired_so_far FROM employees; -- Running min/max SELECT order_date, amount, MIN(amount) OVER (ORDER BY order_date) AS running_min, MAX(amount) OVER (ORDER BY order_date) AS running_max FROM orders; -- Percentage of partition total SELECT name, department, salary, salary * 100.0 / SUM(salary) OVER (PARTITION BY department) AS pct_of_dept FROM employees; -- Running average with frame SELECT order_date, amount, AVG(amount) OVER ( ORDER BY order_date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW ) AS moving_avg_3 FROM orders; -- Cumulative distinct count SELECT order_date, customer_id, COUNT(DISTINCT customer_id) OVER ( ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS cumulative_customers FROM orders;

Running Totals Intermediate

Calculate cumulative sums and other running aggregates using window functions.

-- Simple running total SELECT order_date, amount, SUM(amount) OVER (ORDER BY order_date) AS running_total FROM orders ORDER BY order_date; -- Running total per customer SELECT customer_id, order_date, amount, SUM(amount) OVER ( PARTITION BY customer_id ORDER BY order_date ) AS customer_running_total FROM orders; -- Running percentage of total SELECT product_name, sales, SUM(sales) OVER (ORDER BY sales DESC) AS cumulative, SUM(sales) OVER (ORDER BY sales DESC) * 100.0 / SUM(sales) OVER () AS cum_pct FROM products; -- Year-to-date calculation SELECT sale_date, amount, SUM(amount) OVER ( PARTITION BY EXTRACT(YEAR FROM sale_date) ORDER BY sale_date ) AS ytd_total FROM sales; -- Running count with conditions SELECT order_date, status, SUM(CASE WHEN status = 'completed' THEN 1 ELSE 0 END) OVER (ORDER BY order_date) AS completed_running_count FROM orders;

Moving Averages Intermediate

Calculate rolling averages using frame clauses to smooth out data over a window.

ROWS Frame (Fixed count)

-- 3-day moving average SELECT sale_date, amount, AVG(amount) OVER ( ORDER BY sale_date ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING ) AS ma_3 FROM sales; -- 7-day moving average AVG(amount) OVER ( ORDER BY sale_date ROWS BETWEEN 3 PRECEDING AND 3 FOLLOWING ) AS ma_7

RANGE Frame (Time-based)

-- 30-day rolling average SELECT sale_date, amount, AVG(amount) OVER ( ORDER BY sale_date RANGE BETWEEN INTERVAL '30 days' PRECEDING AND CURRENT ROW ) AS rolling_30d FROM sales; -- 7-day lookback (PostgreSQL) AVG(amount) OVER ( ORDER BY sale_date RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW ) AS week_avg
-- Multiple moving averages SELECT date, close_price, AVG(close_price) OVER ( ORDER BY date ROWS BETWEEN 4 PRECEDING AND CURRENT ROW ) AS ma_5, AVG(close_price) OVER ( ORDER BY date ROWS BETWEEN 19 PRECEDING AND CURRENT ROW ) AS ma_20, AVG(close_price) OVER ( ORDER BY date ROWS BETWEEN 49 PRECEDING AND CURRENT ROW ) AS ma_50 FROM stock_prices; -- Weighted moving average using window SELECT date, value, SUM(value * weight) OVER w / SUM(weight) OVER w AS wma FROM weighted_data WINDOW w AS ( ORDER BY date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW );

Frame Clause Basics Intermediate

The frame clause defines which rows are included in the window calculation relative to the current row.

-- Frame syntax {ROWS | RANGE | GROUPS} frame_extent frame_extent = {UNBOUNDED PRECEDING | offset PRECEDING | CURRENT ROW} {UNBOUNDED FOLLOWING | offset FOLLOWING | CURRENT ROW} -- Common frame patterns -- Start to current row (default) ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW -- Entire partition ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING -- Sliding window ROWS BETWEEN 2 PRECEDING AND 2 FOLLOWING -- Current row and following ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING -- Examples with SUM SELECT id, value, SUM(value) OVER ( ORDER BY id ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_total, SUM(value) OVER ( ORDER BY id ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ) AS total, SUM(value) OVER ( ORDER BY id ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING ) AS centered_sum FROM data;

ROWS vs RANGE vs GROUPS Advanced

Three frame types that determine how rows are selected relative to the current row.

ROWS (Physical rows)

-- ROWS: Counts physical rows SELECT id, value, SUM(value) OVER ( ORDER BY id ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING ) AS sum_neighbors FROM data; -- Always includes exactly 3 rows -- (previous, current, next)

RANGE (Value range)

-- RANGE: Based on ORDER BY values SELECT salary, COUNT(*) OVER ( ORDER BY salary RANGE BETWEEN 1000 PRECEDING AND 1000 FOLLOWING ) AS within_1000 FROM employees; -- INCLUDES all rows within value range -- Multiple rows with same value are included -- Default frame type when ORDER BY is specified
-- GROUPS (Value groups - PostgreSQL) -- Includes all rows with the same ORDER BY value SELECT salary, COUNT(*) OVER ( ORDER BY salary GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING ) AS groups_count FROM employees; -- Visual comparison (salary values with ties): -- salary ROWS RANGE GROUPS -- 50000 1 1 1 -- 50000 2 2 2 ← same value -- 60000 3 3 3 -- 70000 2 4 4 ← RANGE counts 60k and 80k -- 80000 1 3 3 -- Comparison example SELECT salary, COUNT(*) OVER w_rows AS rows_count, COUNT(*) OVER w_range AS range_count, COUNT(*) OVER w_groups AS groups_count FROM employees WINDOW w_rows AS (ORDER BY salary ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING), w_range AS (ORDER BY salary RANGE BETWEEN 1000 PRECEDING AND 1000 FOLLOWING), w_groups AS (ORDER BY salary GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING);

GROUP and EXCLUDE Advanced

PostgreSQL extensions for advanced frame control.

-- EXCLUDE clause (PostgreSQL) -- Remove specific rows from the frame SELECT id, value, SUM(value) OVER ( ORDER BY id ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING EXCLUDE CURRENT ROW ) AS neighbors_sum, SUM(value) OVER ( ORDER BY id ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING EXCLUDE TIES ) AS no_ties_sum, SUM(value) OVER ( ORDER BY id ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING EXCLUDE GROUP ) AS no_group_sum FROM data; -- EXCLUDE options: -- EXCLUDE NO OTHERS - Default, include everything -- EXCLUDE CURRENT ROW - Exclude the current row -- EXCLUDE TIES - Exclude others with same value -- EXCLUDE GROUP - Exclude entire group of ties -- Example: Sum of neighbors (excluding self) SELECT id, value, AVG(value) OVER ( ORDER BY id ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING EXCLUDE CURRENT ROW ) AS neighbor_avg FROM temperatures;

PERCENT_RANK and CUME_DIST Intermediate

Statistical ranking functions for relative position calculations.

PERCENT_RANK

-- Relative rank (0 to 1) -- Formula: (rank - 1) / (total_rows - 1) SELECT name, salary, RANK() OVER (ORDER BY salary DESC) AS rank, PERCENT_RANK() OVER (ORDER BY salary DESC) AS pct_rank FROM employees; -- Results: -- salary rank pct_rank -- 100000 1 0.00 ← First is always 0 -- 90000 2 0.25 -- 80000 3 0.50 -- 70000 4 0.75 -- 60000 5 1.00 ← Last is always 1

CUME_DIST

-- Cumulative distribution (0 to 1) -- Formula: rows <= current / total_rows SELECT name, salary, CUME_DIST() OVER (ORDER BY salary) AS cum_dist FROM employees; -- Results: -- salary cum_dist -- 60000 0.20 ← 1 of 5 rows ≤ 60k -- 70000 0.40 ← 2 of 5 rows ≤ 70k -- 80000 0.60 -- 90000 0.80 -- 100000 1.00 ← Always ends at 1 -- Find salary percentiles SELECT DISTINCT salary, CUME_DIST() OVER (ORDER BY salary) AS percentile FROM employees ORDER BY salary;

Distribution Functions Intermediate

PostgreSQL's PERCENTILE_CONT and PERCENTILE_DISC for percentile calculations.

-- PERCENTILE_CONT - Continuous (interpolated) SELECT department, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median, PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY salary) AS q1, PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY salary) AS q3 FROM employees GROUP BY department; -- PERCENTILE_DISC - Discrete (actual value) SELECT department, PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY salary) AS median_salary FROM employees GROUP BY department; -- Multiple percentiles SELECT department, PERCENTILE_CONT(UNNEST(ARRAY[0.25, 0.5, 0.75, 0.9])) WITHIN GROUP (ORDER BY salary) AS percentile FROM employees GROUP BY department; -- Window version SELECT name, salary, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) OVER (PARTITION BY department) AS dept_median FROM employees;

Window Aliases Intermediate

Define reusable window specifications with the WINDOW clause to avoid repetition.

-- Without window alias (repetitive) SELECT name, department, salary, RANK() OVER ( PARTITION BY department ORDER BY salary DESC ) AS rank, DENSE_RANK() OVER ( PARTITION BY department ORDER BY salary DESC ) AS dense_rank, ROW_NUMBER() OVER ( PARTITION BY department ORDER BY salary DESC ) AS row_num FROM employees; -- With window alias (cleaner) SELECT name, department, salary, RANK() OVER w AS rank, DENSE_RANK() OVER w AS dense_rank, ROW_NUMBER() OVER w AS row_num FROM employees WINDOW w AS ( PARTITION BY department ORDER BY salary DESC ); -- Multiple window aliases SELECT name, department, salary, RANK() OVER dept_order AS dept_rank, RANK() OVER company_order AS company_rank, SUM(salary) OVER dept_total AS dept_salary_sum, SUM(salary) OVER company_total AS company_salary_sum FROM employees WINDOW dept_order AS (PARTITION BY department ORDER BY salary DESC), company_order AS (ORDER BY salary DESC), dept_total AS (PARTITION BY department), company_total AS (); -- Window with frame specification SELECT date, value, AVG(value) OVER ma_3 AS ma_3day, AVG(value) OVER ma_7 AS ma_7day, SUM(value) OVER running AS running_total FROM daily_data WINDOW ma_3 AS (ORDER BY date ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING), ma_7 AS (ORDER BY date ROWS BETWEEN 3 PRECEDING AND 3 FOLLOWING), running AS (ORDER BY date);

Tip: Use window aliases when you need to reference the same window specification multiple times in a query. It improves readability and maintainability.

Optimization Tips Advanced

Best practices for writing efficient window functions in PostgreSQL.

Do's

  • Index columns used in PARTITION BY and ORDER BY
  • Use CTEs to break complex window operations
  • Use WINDOW clause to reuse specifications
  • Choose appropriate frame type (ROWS vs RANGE)
  • Use covering indexes for window queries

Don'ts

  • Avoid large frames with many rows
  • Don't use RANGE without ORDER BY
  • Don't nest window functions unnecessarily
  • Avoid DISTINCT with window functions
  • Don't forget frame boundaries
-- Check performance with EXPLAIN EXPLAIN ANALYZE SELECT name, department, RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rank FROM employees; -- Useful indexes for window functions CREATE INDEX idx_emp_dept_salary ON employees(department_id, salary DESC); CREATE INDEX idx_orders_date ON orders(order_date); -- Materialize CTE for complex calculations WITH ranked_data AS ( SELECT *, ROW_NUMBER() OVER (w) AS rn FROM large_table WINDOW w AS (PARTITION BY category ORDER BY created_at) ) SELECT * FROM ranked_data WHERE rn <= 10; -- Limit partitions with WHERE before window SELECT name, salary, RANK() OVER (ORDER BY salary DESC) AS rank FROM employees WHERE department = 'Engineering'; -- Filter first! -- Use RANGE for time-series (often faster) SUM(amount) OVER ( ORDER BY sale_date RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW ) AS rolling_7_days;

Performance note: Window functions require sorting, which can be expensive on large datasets. Always ensure appropriate indexes exist on the columns used in ORDER BY and PARTITION BY clauses.
