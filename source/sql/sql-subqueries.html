SQL Subqueries

Nest queries within queries and use Common Table Expressions for complex data operations in PostgreSQL.

Subquery Concepts Basic

A subquery (inner query) is a SELECT statement nested within another SQL statement. They can return scalar values, rows, or entire tables.

Types of Subqueries

  • Scalar - Returns single value
  • Row - Returns single row
  • Table - Returns multiple rows/columns
  • Correlated - References outer query
  • Non-correlated - Independent execution

Where Subqueries Can Appear

  • SELECT - Column expressions
  • FROM - Derived tables
  • WHERE - Filter conditions
  • HAVING - Group filters
  • JOIN - Join conditions
-- Basic structure SELECT column1, column2, ... FROM table1 WHERE column2 IN ( SELECT column2 FROM table2 WHERE condition ); -- Subqueries can be nested multiple levels deep SELECT * FROM employees WHERE dept_id IN ( SELECT id FROM departments WHERE location_id IN ( SELECT id FROM locations WHERE country = 'USA' ) );

Scalar Subqueries Basic

A scalar subquery returns exactly one value (one row, one column). Use it anywhere a single value is expected.

-- In SELECT clause SELECT name, salary, ( SELECT AVG(salary) FROM employees ) AS company_avg, salary - ( SELECT AVG(salary) FROM employees ) AS diff_from_avg FROM employees; -- In WHERE clause SELECT * FROM employees WHERE salary > ( SELECT AVG(salary) FROM employees ); -- In ORDER BY SELECT * FROM employees ORDER BY salary - ( SELECT AVG(salary) FROM employees ); -- In comparison operators SELECT * FROM products WHERE price > ( SELECT AVG(price) FROM products WHERE category = 'Electronics' ); -- COALESCE with scalar subquery SELECT name, COALESCE( department_id, ( SELECT id FROM departments WHERE dept_name = 'Unassigned' ) ) AS dept FROM employees; -- Multiple scalar subqueries SELECT COUNT(*) AS total_employees, (SELECT COUNT(*) FROM departments) AS total_depts, (SELECT SUM(salary) FROM employees) AS total_payroll FROM employees;

Row Subqueries Intermediate

A row subquery returns a single row with multiple columns.

-- Row constructor comparison SELECT * FROM employees WHERE (dept_id, salary) = ( SELECT dept_id, MAX(salary) FROM employees WHERE dept_id = 1 GROUP BY dept_id ); -- Find top earner per department SELECT e.* FROM employees e WHERE (e.dept_id, e.salary) IN ( SELECT dept_id, MAX(salary) FROM employees GROUP BY dept_id ); -- Row subquery with comparison operators SELECT name, salary FROM employees WHERE (salary, hire_date) > ( SELECT salary, hire_date FROM employees WHERE name = 'Alice' ); -- Using ROW() constructor (PostgreSQL) SELECT * FROM employees WHERE ROW(dept_id, salary) IN ( SELECT ROW(dept_id, MAX(salary)) FROM employees GROUP BY dept_id );

IN Subqueries Basic

Use IN with subqueries to check if a value matches any value in a result set.

Basic IN Subquery

-- Employees in NYC departments SELECT name FROM employees WHERE dept_id IN ( SELECT id FROM departments WHERE location = 'NYC' ); -- Customers who have ordered SELECT * FROM customers WHERE id IN ( SELECT DISTINCT customer_id FROM orders WHERE order_date >= '2024-01-01' );

NOT IN (with caution)

-- Employees in departments outside NYC SELECT name FROM employees WHERE dept_id NOT IN ( SELECT id FROM departments WHERE location = 'NYC' ); -- WARNING: NOT IN fails if subquery returns NULL! -- This returns NO rows if any dept_id is NULL: WHERE id NOT IN (1, 2, NULL) -- Safe alternative: NOT EXISTS SELECT name FROM employees e WHERE NOT EXISTS ( SELECT 1 FROM departments d WHERE d.id = e.dept_id AND d.location = 'NYC' );

EXISTS Subqueries Intermediate

EXISTS checks for the presence of rows in a subquery without returning actual values. Often more efficient than IN.

-- Employees with orders (EXISTS) SELECT e.name FROM employees e WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.employee_id = e.id AND o.status = 'completed' ); -- Departments with at least one employee earning > 70k SELECT d.dept_name FROM departments d WHERE EXISTS ( SELECT 1 FROM employees e WHERE e.dept_id = d.id AND e.salary > 70000 ); -- NOT EXISTS: Products never ordered in 2024 SELECT p.product_name FROM products p WHERE NOT EXISTS ( SELECT 1 FROM order_items oi JOIN orders o ON o.id = oi.order_id WHERE oi.product_id = p.id AND o.order_date >= '2024-01-01' ); -- EXISTS vs IN: Use EXISTS when: -- 1. Checking for existence only -- 2. Subquery may contain NULLs -- 3. Large result set but early exit possible -- IN is better when: -- 1. Small, static list of values -- 2. You need the actual values

Comparison Operators Intermediate

Use comparison operators with scalar subqueries for conditions like "greater than all" or "equal to any".

ANY / SOME

-- Salary greater than ANY (at least one) SELECT name, salary FROM employees WHERE salary > ANY ( SELECT salary FROM employees WHERE dept_id = 3 ); -- Same as: salary > min(salary in dept 3) SELECT name, salary FROM employees WHERE salary > ( SELECT MIN(salary) FROM employees WHERE dept_id = 3 ); -- SOME is synonymous with ANY WHERE salary > SOME (subquery)

ALL

-- Salary greater than ALL (every single one) SELECT name, salary FROM employees WHERE salary > ALL ( SELECT salary FROM employees WHERE dept_id = 3 ); -- Same as: salary > max(salary in dept 3) SELECT name, salary FROM employees WHERE salary > ( SELECT MAX(salary) FROM employees WHERE dept_id = 3 ); -- Products priced higher than all items in a category SELECT product_name, price FROM products WHERE price > ALL ( SELECT price FROM products WHERE category = 'Books' );

FROM Subqueries (Derived Tables) Intermediate

A derived table is a subquery in the FROM clause that acts as a temporary table.

-- Basic derived table SELECT dept_stats.department, dept_stats.avg_salary FROM ( SELECT department, AVG(salary) AS avg_salary, COUNT(*) AS emp_count FROM employees GROUP BY department ) AS dept_stats WHERE dept_stats.avg_salary > 50000; -- Multiple derived tables SELECT top_depts.dept_name, top_depts.total_salary, top_employees.top_earner FROM ( SELECT d.dept_name, SUM(e.salary) AS total_salary FROM departments d JOIN employees e ON d.id = e.dept_id GROUP BY d.dept_name ) AS top_depts JOIN ( SELECT DISTINCT ON (dept_id) dept_id, name AS top_earner FROM employees ORDER BY dept_id, salary DESC ) AS top_employees ON top_depts.dept_name = top_employees.dept_id; -- Pivot using derived tables SELECT category, total_sales, total_orders FROM ( SELECT p.category, SUM(oi.quantity * oi.price) AS total_sales, COUNT(DISTINCT oi.order_id) AS total_orders FROM order_items oi JOIN products p ON p.id = oi.product_id GROUP BY p.category ) AS stats WHERE total_sales > 10000;

Tip: Derived tables must have an alias in PostgreSQL. Use meaningful aliases to improve query readability.

LATERAL Subqueries Advanced

LATERAL allows a subquery to reference columns from previous FROM items, enabling correlated derived tables.

-- Top N per group (classic LATERAL use) SELECT d.dept_name, e.name, e.salary FROM departments d LEFT JOIN LATERAL ( SELECT name, salary FROM employees WHERE dept_id = d.id -- References outer dept ORDER BY salary DESC LIMIT 3 ) e ON TRUE; -- Latest transaction per account SELECT a.account_id, a.balance, lt.last_transaction_date, lt.last_amount FROM accounts a CROSS JOIN LATERAL ( SELECT MAX(transaction_date) AS last_transaction_date, ( SELECT amount FROM transactions t2 WHERE t2.account_id = a.id ORDER BY transaction_date DESC LIMIT 1 ) AS last_amount FROM transactions t WHERE t.account_id = a.id ) lt; -- Unnest with conditions SELECT p.id, p.title, tag, tag_index FROM posts p, LATERAL UNNEST(p.tags) WITH ORDINALITY AS t(tag, tag_index) WHERE tag_index <= 3; -- First 3 tags only -- Generate series per row SELECT d.date::DATE, e.name FROM generate_series( '2024-01-01'::DATE, '2024-01-07'::DATE, '1 day' ) d CROSS JOIN employees e;

Basic CTEs Intermediate

Common Table Expressions (CTEs) provide a cleaner, more readable alternative to subqueries and derived tables.

-- Basic CTE syntax WITH dept_stats AS ( SELECT dept_id, COUNT(*) AS emp_count, AVG(salary) AS avg_salary, SUM(salary) AS total_salary FROM employees GROUP BY dept_id ) SELECT d.dept_name, ds.* FROM departments d JOIN dept_stats ds ON d.id = ds.dept_id WHERE ds.avg_salary > 50000; -- Multiple CTEs WITH high_earners AS ( SELECT * FROM employees WHERE salary > 80000 ), dept_summary AS ( SELECT dept_id, COUNT(*) AS high_earner_count, AVG(salary) AS avg_high_salary FROM high_earners GROUP BY dept_id ) SELECT d.dept_name, ds.high_earner_count, ds.avg_high_salary FROM departments d LEFT JOIN dept_summary ds ON d.id = ds.dept_id; -- CTE for data transformation WITH normalized AS ( SELECT id, TRIM(LOWER(name)) AS clean_name, INITCAP(department) AS title_case_dept, COALESCE(email, 'N/A') AS email FROM employees ) SELECT * FROM normalized;

Recursive CTEs Advanced

Recursive CTEs reference themselves, enabling hierarchical and graph traversals.

-- Organization hierarchy WITH RECURSIVE org_chart AS ( -- Base case: top-level managers SELECT id, name, manager_id, 0 AS level, ARRAY[id] AS path FROM employees WHERE manager_id IS NULL UNION ALL -- Recursive case: employees under managers SELECT e.id, e.name, e.manager_id, oc.level + 1, oc.path || e.id FROM employees e JOIN org_chart oc ON e.manager_id = oc.id WHERE e.id NOT IN (SELECT UNNEST(oc.path)) -- Prevent cycles ) SELECT REPEAT(' ', level) || name AS org_tree FROM org_chart ORDER BY path; -- Number series WITH RECURSIVE numbers AS ( SELECT 1 AS n UNION ALL SELECT n + 1 FROM numbers WHERE n < 100 ) SELECT n FROM numbers; -- Date series (missing dates) WITH RECURSIVE date_range AS ( SELECT '2024-01-01'::DATE AS dt UNION ALL SELECT dt + 1 FROM date_range WHERE dt < '2024-12-31' ) SELECT d.dt FROM date_range d LEFT JOIN sales s ON s.sale_date = d.dt WHERE s.id IS NULL; -- Days with no sales

Important: Always include a cycle detection mechanism in recursive CTEs (like checking against a path array) to prevent infinite loops with circular references.

Advanced CTE Patterns Advanced

Powerful CTE patterns for common data analysis tasks in PostgreSQL.

-- Running totals and moving averages WITH daily_sales AS ( SELECT order_date::DATE AS dt, SUM(total) AS daily_total FROM orders GROUP BY order_date::DATE ) SELECT dt, daily_total, SUM(daily_total) OVER (ORDER BY dt) AS running_total, AVG(daily_total) OVER ( ORDER BY dt ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS moving_avg_7day FROM daily_sales ORDER BY dt; -- Cohort analysis WITH first_order AS ( SELECT customer_id, DATE_TRUNC('month', MIN(order_date)) AS cohort_month FROM orders GROUP BY customer_id ), cohort_data AS ( SELECT fo.cohort_month, DATE_TRUNC('month', o.order_date) AS order_month, COUNT(DISTINCT o.customer_id) AS customers FROM orders o JOIN first_order fo ON o.customer_id = fo.customer_id GROUP BY fo.cohort_month, DATE_TRUNC('month', o.order_date) ) SELECT cohort_month, order_month, EXTRACT(YEAR FROM order_month) - EXTRACT(YEAR FROM cohort_month) + (EXTRACT(MONTH FROM order_month) - EXTRACT(MONTH FROM cohort_month)) / 12.0 AS month_number, customers FROM cohort_data; -- Gap and island analysis WITH numbered_logins AS ( SELECT user_id, login_date, login_date - ROW_NUMBER() OVER ( PARTITION BY user_id ORDER BY login_date ) AS grp FROM logins ) SELECT user_id, MIN(login_date) AS streak_start, MAX(login_date) AS streak_end, COUNT(*) AS streak_length FROM numbered_logins GROUP BY user_id, grp ORDER BY streak_length DESC;

Correlated Subqueries Advanced

A correlated subquery references columns from the outer query and is re-executed for each row.

-- Basic correlated subquery SELECT e.name, e.salary, e.dept_id, (SELECT AVG(salary) FROM employees e2 WHERE e2.dept_id = e.dept_id) AS dept_avg FROM employees e; -- Find employees earning above department average SELECT e.name, e.salary FROM employees e WHERE e.salary > ( SELECT AVG(e2.salary) FROM employees e2 WHERE e2.dept_id = e.dept_id ); -- Rank within group using correlated subquery SELECT e.name, e.salary, e.dept_id, (SELECT COUNT(*) + 1 FROM employees e2 WHERE e2.dept_id = e.dept_id AND e2.salary > e.salary) AS salary_rank FROM employees e; -- Most recent order per customer (correlated) SELECT c.name, (SELECT MAX(order_date) FROM orders o WHERE o.customer_id = c.id) AS last_order FROM customers c; -- Compare to window function equivalent (usually better) SELECT DISTINCT ON (customer_id) customer_id, order_date FROM orders ORDER BY customer_id, order_date DESC;

Performance tip: Correlated subqueries can be slow on large tables. Consider using JOINs, CTEs, or window functions as alternatives when possible.

Subquery Optimization Advanced

Tips for writing efficient subqueries in PostgreSQL.

Do's

  • Use EXISTS instead of IN for large subqueries
  • Use CTEs for complex, multi-step queries
  • Use window functions instead of correlated subqueries
  • Add indexes on columns used in subquery joins
  • Use LATERAL for top-N per group queries

Don'ts

  • Avoid NOT IN with nullable columns
  • Don't nest too deeply (3+ levels)
  • Avoid correlated subqueries in SELECT when possible
  • Don't use DISTINCT when EXISTS would suffice
  • Avoid SELECT * in subqueries
-- Use EXPLAIN ANALYZE to check performance EXPLAIN ANALYZE SELECT * FROM employees WHERE dept_id IN ( SELECT id FROM departments WHERE location = 'NYC' ); -- Convert correlated subquery to JOIN -- SLOW: Correlated subquery SELECT name, salary FROM employees e WHERE salary > ( SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id ); -- FAST: JOIN with derived table SELECT e.name, e.salary FROM employees e JOIN ( SELECT dept_id, AVG(salary) AS avg_salary FROM employees GROUP BY dept_id ) d ON e.dept_id = d.dept_id WHERE e.salary > d.avg_salary; -- EXISTS instead of IN for large sets -- SLOW: WHERE id IN (SELECT customer_id FROM orders) -- FAST: WHERE EXISTS ( SELECT 1 FROM orders WHERE orders.customer_id = customers.id );

Pro tip: Use EXPLAIN (ANALYZE, BUFFERS) to see actual execution times and I/O statistics. Compare different approaches to find the most efficient pattern for your data.
