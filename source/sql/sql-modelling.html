Data Modelling

Normalization, star schema, and snowflake design patterns.

What is Data Modelling? Basic

Data modelling is the process of creating a visual representation of data structures, relationships, and constraints. It defines how data is stored, organized, and accessed.

Purpose of Data Modelling

  • Structure: Define tables, columns, and relationships
  • Integrity: Enforce data quality through constraints
  • Performance: Optimize queries and storage
  • Communication: Common language for stakeholders
  • Documentation: Blueprint for development

Modelling Levels

-- Conceptual Model (High-level) -- "Customer places Orders for Products" -- Logical Model (Detailed structure) -- Customer(id, name, email) -- Order(id, customer_id, order_date) -- Physical Model (Implementation) CREATE TABLE customers ( customer_id SERIAL PRIMARY KEY, email VARCHAR(255) UNIQUE NOT NULL );

Modelling Process Flow

Requirements → Conceptual Model → Logical Model → Physical Model → Implementation

Entity-Relationship Diagrams Basic

ER diagrams visualize entities, their attributes, and relationships. They are the foundation of relational database design.

-- ER Diagram Symbols -- [Entity] -------- [Entity] -- | = Mandatory (1) -- O = Optional (0) -- Crow's foot = Many -- Common Relationships: -- 1:1 (One-to-One) -- User --||--||-- Profile -- Each user has exactly one profile -- 1:N (One-to-Many) -- Customer --||--|< Order -- One customer has many orders -- M:N (Many-to-Many) -- Student >|o--o< Course -- Students enroll in many courses -- Courses have many students -- Requires junction table CREATE TABLE enrollments ( student_id INTEGER REFERENCES students(id), course_id INTEGER REFERENCES courses(id), enrolled_at TIMESTAMPTZ DEFAULT NOW(), PRIMARY KEY (student_id, course_id) );

Relationship Cardinality

1:1 │ 1:N │ M:N
User ↔ Profile → Customer → Orders → Students ↔ Courses
UNIQUE FK → FK on Many Side → Junction Table

Entities and Relationships Basic

Entities are real-world objects or concepts. Relationships define how entities interact with each other.

Entity Types

-- Strong Entity (exists independently) CREATE TABLE customers ( id SERIAL PRIMARY KEY, name VARCHAR(100) ); -- Weak Entity (depends on strong entity) CREATE TABLE orders ( customer_id INTEGER REFERENCES customers(id), order_number INTEGER, -- Weak entity needs parent key PRIMARY KEY (customer_id, order_number) ); -- Associative Entity (resolves M:N) CREATE TABLE order_items ( order_id INTEGER REFERENCES orders(id), product_id INTEGER REFERENCES products(id), quantity INTEGER, price DECIMAL(10,2), PRIMARY KEY (order_id, product_id) );

Relationship Cardinality

-- One-to-One (1:1) -- Use foreign key with UNIQUE constraint CREATE TABLE user_profiles ( user_id INTEGER PRIMARY KEY REFERENCES users(id), bio TEXT, avatar_url VARCHAR(500) ); -- One-to-Many (1:N) - Most common -- Foreign key on "many" side CREATE TABLE orders ( id SERIAL PRIMARY KEY, customer_id INTEGER REFERENCES customers(id), total DECIMAL(10,2) ); -- Many-to-Many (M:N) -- Requires junction table CREATE TABLE product_categories ( product_id INTEGER REFERENCES products(id), category_id INTEGER REFERENCES categories(id), PRIMARY KEY (product_id, category_id) );

Primary Keys Basic

A primary key uniquely identifies each row in a table. It cannot contain NULL values and must be unique.

-- Single column primary key CREATE TABLE users ( id SERIAL PRIMARY KEY, username VARCHAR(50) UNIQUE NOT NULL, email VARCHAR(255) UNIQUE NOT NULL ); -- Using UUID as primary key CREATE EXTENSION IF NOT EXISTS "uuid-ossp"; CREATE TABLE products ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), name VARCHAR(200) NOT NULL, sku VARCHAR(50) UNIQUE ); -- Composite primary key CREATE TABLE enrollments ( student_id INTEGER, course_id INTEGER, semester VARCHAR(20), grade CHAR(2), PRIMARY KEY (student_id, course_id, semester) ); -- Surrogate key with GENERATED ALWAYS CREATE TABLE invoices ( id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY, invoice_number VARCHAR(20) UNIQUE NOT NULL, amount DECIMAL(10,2) ); -- Natural key (business key) CREATE TABLE countries ( iso_code CHAR(2) PRIMARY KEY, -- US, UK, IN name VARCHAR(100) NOT NULL );

Best practice: Use surrogate keys (auto-increment or UUID) for most tables. Use natural keys only when they are truly stable and guaranteed unique.

Foreign Keys Basic

Foreign keys establish relationships between tables by referencing the primary key of another table.

-- Basic foreign key CREATE TABLE orders ( id SERIAL PRIMARY KEY, customer_id INTEGER REFERENCES customers(id), order_date DATE DEFAULT CURRENT_DATE ); -- Named foreign key constraint CREATE TABLE order_items ( id SERIAL PRIMARY KEY, order_id INTEGER, product_id INTEGER, quantity INTEGER NOT NULL, CONSTRAINT fk_order FOREIGN KEY (order_id) REFERENCES orders(id), CONSTRAINT fk_product FOREIGN KEY (product_id) REFERENCES products(id) ); -- ON DELETE actions CREATE TABLE reviews ( id SERIAL PRIMARY KEY, product_id INTEGER REFERENCES products(id) ON DELETE CASCADE, -- Delete review if product deleted user_id INTEGER REFERENCES users(id) ON DELETE SET NULL, -- Set NULL if user deleted rating INTEGER CHECK (rating BETWEEN 1 AND 5), comment TEXT ); -- ON DELETE options: -- CASCADE - Delete referencing rows -- SET NULL - Set FK column to NULL -- SET DEFAULT - Set FK column to default -- RESTRICT - Prevent deletion -- NO ACTION - Same as RESTRICT (deferred) -- Self-referencing foreign key CREATE TABLE employees ( id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL, manager_id INTEGER REFERENCES employees(id) ON DELETE SET NULL ); -- Check foreign key exists SELECT tc.constraint_name, tc.table_name, kcu.column_name, ccu.table_name AS foreign_table_name FROM information_schema.table_constraints AS tc JOIN information_schema.key_column_usage AS kcu ON tc.constraint_name = kcu.constraint_name JOIN information_schema.constraint_column_usage AS ccu ON ccu.constraint_name = tc.constraint_name WHERE tc.constraint_type = 'FOREIGN KEY';

Composite Keys Intermediate

Composite keys use multiple columns to uniquely identify a row. Common in junction tables and time-series data.

-- Composite primary key CREATE TABLE order_items ( order_id INTEGER NOT NULL, line_number INTEGER NOT NULL, product_id INTEGER NOT NULL, quantity INTEGER, price DECIMAL(10,2), PRIMARY KEY (order_id, line_number) ); -- Junction table with composite key CREATE TABLE user_roles ( user_id INTEGER NOT NULL, role_id INTEGER NOT NULL, assigned_at TIMESTAMPTZ DEFAULT NOW(), assigned_by INTEGER, PRIMARY KEY (user_id, role_id), FOREIGN KEY (user_id) REFERENCES users(id), FOREIGN KEY (role_id) REFERENCES roles(id) ); -- Time-series composite key CREATE TABLE stock_prices ( ticker VARCHAR(10) NOT NULL, trade_date DATE NOT NULL, open DECIMAL(10,2), high DECIMAL(10,2), low DECIMAL(10,2), close DECIMAL(10,2), volume BIGINT, PRIMARY KEY (ticker, trade_date) ); -- Querying with composite keys SELECT * FROM order_items WHERE order_id = 1001 AND line_number = 1; -- Partial key query (get all lines for order) SELECT * FROM order_items WHERE order_id = 1001 ORDER BY line_number;

Surrogate vs Natural Keys Intermediate

Choosing between surrogate (artificial) and natural (business) keys is a critical modelling decision.

Surrogate Keys

-- Auto-increment integer id SERIAL PRIMARY KEY -- Identity column (SQL standard) id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY -- UUID (globally unique) id UUID PRIMARY KEY DEFAULT uuid_generate_v4() -- Pros: -- Always stable/immutable -- Simple joins (single column) -- No business logic dependency -- Smaller index size -- Cons: -- Meaningless without join -- Extra column to maintain

Natural Keys

-- Email as key email VARCHAR(255) PRIMARY KEY -- Country ISO code iso_code CHAR(2) PRIMARY KEY -- SKU (Stock Keeping Unit) sku VARCHAR(50) PRIMARY KEY -- Composite natural key PRIMARY KEY (state_code, county_fips) -- Pros: -- Meaningful, self-documenting -- No extra column needed -- Can enforce uniqueness -- Cons: -- May change over time -- Larger index size -- Complex composite keys -- Cascading updates if changed

Rule of thumb: Use surrogate keys by default. Consider natural keys only for reference/lookup tables with truly immutable values (countries, currencies, states).

Entity Relationship Identification

Identify Entities → Define Attributes → Map Relationships → Set Cardinality → Add Constraints

Normalization Overview Basic

Normalization is the process of organizing data to reduce redundancy and improve data integrity. It progresses through normal forms (1NF, 2NF, 3NF, BCNF).

-- Problem: Unnormalized Data -- Single table with redundancy orders_unnormalized ( order_id, customer_name, customer_email, customer_phone, product_1_name, product_1_qty, product_2_name, product_2_qty, order_date ) -- Issues: -- 1. Update anomaly: Change email in multiple rows -- 2. Insert anomaly: Cannot add customer without order -- 3. Delete anomaly: Deleting order loses customer info -- 4. Wasted space: Customer data repeated -- Goal: Each fact stored in exactly one place

Normalization Progression

Unnormalized (0NF) → 1NF → 2NF → 3NF → BCNF
Atomic Values → Full Dependency → No Transitive → All Superkeys

Update Anomalies

Update Anomaly │ Insert Anomaly │ Delete Anomaly
Same data in multiple rows → Cannot add without related data → Lose unrelated data

First Normal Form (1NF) Basic

1NF requires that each column contains atomic (indivisible) values and each row is unique.

Violates 1NF

-- Multi-valued column (BAD) orders ( order_id, products -- "Widget,Gadget,Gizmo" ) -- Repeating groups (BAD) customers ( customer_id, phone_1, phone_2, phone_3 )

Satisfies 1NF

-- Separate table for products order_items ( order_id, product_id, quantity ) -- Separate table for phones customer_phones ( customer_id, phone_number, phone_type -- 'mobile', 'home', 'work' ) -- Atomic values users ( id SERIAL PRIMARY KEY, first_name VARCHAR(50), last_name VARCHAR(50), -- Not: full_name with multiple values email VARCHAR(255) )
-- Converting to 1NF -- Before (violates 1NF) CREATE TABLE orders_bad ( id INTEGER PRIMARY KEY, customer VARCHAR(100), products VARCHAR(500) -- "A,2,B,1,C,3" ); -- After (1NF compliant) CREATE TABLE orders ( id SERIAL PRIMARY KEY, customer VARCHAR(100) ); CREATE TABLE order_items ( order_id INTEGER REFERENCES orders(id), product_name VARCHAR(100), quantity INTEGER, PRIMARY KEY (order_id, product_name) );

Second Normal Form (2NF) Intermediate

2NF requires 1NF plus no partial dependencies - non-key attributes must depend on the entire primary key, not just part of it.

Violates 2NF

-- Composite PK with partial dependency order_items_bad ( order_id, \ product_id, |-- Composite PK product_name, |-- Depends only on product_id product_price, |-- Depends only on product_id quantity, |-- Depends on both (OK) discount |-- Depends on both (OK) ) -- product_name depends on product_id only! -- This is a partial dependency

Satisfies 2NF

-- Split into separate tables CREATE TABLE products ( id SERIAL PRIMARY KEY, name VARCHAR(200), price DECIMAL(10,2) ); CREATE TABLE order_items ( order_id INTEGER, product_id INTEGER REFERENCES products(id), quantity INTEGER, discount DECIMAL(5,2), PRIMARY KEY (order_id, product_id) ); -- Now product details stored once in products table

Third Normal Form (3NF) Intermediate

3NF requires 2NF plus no transitive dependencies - non-key attributes cannot depend on other non-key attributes.

Violates 3NF

-- Transitive dependency employees_bad ( id -- PK, name, department_id, department_name, -- Depends on department_id department_floor, -- Depends on department_id manager_id, manager_name -- Depends on manager_id ) -- Transitive chains: -- id -> department_id -> department_name -- id -> manager_id -> manager_name

Satisfies 3NF

CREATE TABLE departments ( id SERIAL PRIMARY KEY, name VARCHAR(100), floor INTEGER ); CREATE TABLE managers ( id SERIAL PRIMARY KEY, name VARCHAR(100) ); CREATE TABLE employees ( id SERIAL PRIMARY KEY, name VARCHAR(100), department_id INTEGER REFERENCES departments(id), manager_id INTEGER REFERENCES managers(id) ); -- No transitive dependencies!

Boyce-Codd Normal Form (BCNF) Advanced

BCNF is a stricter version of 3NF. For every functional dependency X -> Y, X must be a superkey.

-- Example violating 3NF/BCNF -- Student takes course, has instructor -- Instructor teaches only one course course_instructors_bad ( student_id, course_id, instructor_id ) -- Functional dependencies: -- (student_id, course_id) -> instructor_id [PK determines all] -- instructor_id -> course_id [Instructor teaches one course] -- Problem: instructor_id -> course_id -- instructor_id is NOT a superkey -- This violates BCNF -- BCNF Solution: CREATE TABLE course_assignments ( instructor_id SERIAL PRIMARY KEY, course_id INTEGER REFERENCES courses(id) ); CREATE TABLE student_enrollments ( student_id INTEGER, course_id INTEGER, PRIMARY KEY (student_id, course_id) ); -- Now all determinants are superkeys!

Denormalization Intermediate

Denormalization intentionally adds redundancy to improve read performance. Used strategically in high-read workloads.

When to Denormalize

  • Heavy read workload with many JOINs
  • Data warehouse / reporting tables
  • Pre-aggregated summaries needed
  • Historical data rarely updated
  • API response time critical

Denormalization Techniques

-- Add redundant columns CREATE TABLE order_summary ( order_id PRIMARY KEY, customer_id, customer_name, -- Redundant! customer_email, -- Redundant! order_total, item_count -- Pre-computed! ); -- Pre-computed aggregates CREATE MATERIALIZED VIEW daily_sales AS SELECT sale_date, SUM(amount) AS total_sales, COUNT(*) AS transaction_count FROM sales GROUP BY sale_date;
-- Denormalized product catalog (read-optimized) CREATE TABLE product_catalog ( product_id SERIAL PRIMARY KEY, product_name VARCHAR(200), category_name VARCHAR(100), -- Denormalized from categories category_path VARCHAR(500), -- Full category hierarchy brand_name VARCHAR(100), -- Denormalized from brands avg_rating DECIMAL(3,2), -- Pre-computed review_count INTEGER, -- Pre-computed price DECIMAL(10,2), in_stock BOOLEAN, search_vector TSVECTOR -- Full-text search index ); -- Maintain denormalized data with triggers CREATE OR REPLACE FUNCTION update_product_rating() RETURNS TRIGGER AS $$ BEGIN UPDATE product_catalog SET avg_rating = ( SELECT AVG(rating) FROM reviews WHERE product_id = NEW.product_id ), review_count = ( SELECT COUNT(*) FROM reviews WHERE product_id = NEW.product_id ) WHERE product_id = NEW.product_id; RETURN NEW; END; $$ LANGUAGE plpgsql;

Star Schema Intermediate

Star schema is a data warehouse pattern with a central fact table surrounded by dimension tables. Optimized for queries and reporting.

-- Star Schema Structure -- dim_date -- | -- dim_product--fact_sales--dim_customer -- | -- dim_store -- Dimension tables (descriptive attributes) CREATE TABLE dim_date ( date_key INTEGER PRIMARY KEY, -- 20240315 full_date DATE, year INTEGER, quarter INTEGER, month INTEGER, month_name VARCHAR(20), day_of_week VARCHAR(20), is_weekend BOOLEAN, fiscal_year INTEGER ); CREATE TABLE dim_product ( product_key SERIAL PRIMARY KEY, product_id INTEGER, -- Natural key product_name VARCHAR(200), category VARCHAR(100), subcategory VARCHAR(100), brand VARCHAR(100), unit_cost DECIMAL(10,2) ); CREATE TABLE dim_customer ( customer_key SERIAL PRIMARY KEY, customer_id INTEGER, name VARCHAR(100), segment VARCHAR(50), -- 'Retail', 'Corporate' city VARCHAR(100), state VARCHAR(50), country VARCHAR(50) ); -- Fact table (numeric measures) CREATE TABLE fact_sales ( sale_id BIGSERIAL PRIMARY KEY, date_key INTEGER REFERENCES dim_date(date_key), product_key INTEGER REFERENCES dim_product(product_key), customer_key INTEGER REFERENCES dim_customer(customer_key), quantity INTEGER, unit_price DECIMAL(10,2), discount DECIMAL(5,2), sales_amount DECIMAL(12,2), -- quantity * price * (1-discount) profit DECIMAL(12,2) ); -- Star schema query (simple, fast) SELECT d.year, d.quarter, p.category, SUM(f.sales_amount) AS total_sales, SUM(f.profit) AS total_profit, COUNT(*) AS transactions FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_product p ON f.product_key = p.product_key GROUP BY d.year, d.quarter, p.category ORDER BY d.year, d.quarter;

Snowflake Schema Intermediate

Snowflake schema extends star schema by normalizing dimension tables into sub-dimensions, creating a snowflake pattern.

-- Snowflake: Normalized dimensions CREATE TABLE dim_category ( category_id SERIAL PRIMARY KEY, category_name VARCHAR(100) ); CREATE TABLE dim_brand ( brand_id SERIAL PRIMARY KEY, brand_name VARCHAR(100), manufacturer VARCHAR(100) ); CREATE TABLE dim_product ( product_key SERIAL PRIMARY KEY, product_name VARCHAR(200), category_id INTEGER REFERENCES dim_category(category_id), brand_id INTEGER REFERENCES dim_brand(brand_id), unit_cost DECIMAL(10,2) ); CREATE TABLE dim_city ( city_id SERIAL PRIMARY KEY, city_name VARCHAR(100), state_province VARCHAR(100), country VARCHAR(100), region VARCHAR(50) ); CREATE TABLE dim_customer ( customer_key SERIAL PRIMARY KEY, customer_id INTEGER, name VARCHAR(100), city_id INTEGER REFERENCES dim_city(city_id), segment VARCHAR(50) ); -- Star vs Snowflake trade-offs -- Star: Simpler queries, faster (fewer JOINs) -- Snowflake: Less redundancy, easier maintenance

Star Schema

  • Denormalized dimensions
  • Fewer tables, simpler queries
  • Better for query performance
  • Redundant data in dimensions

Snowflake Schema

  • Normalized dimensions
  • More tables, complex queries
  • Better for storage efficiency
  • Easier to maintain dimension data

Fact and Dimension Tables Intermediate

Understanding the difference between fact and dimension tables is essential for data warehouse design.

Fact Tables

-- Contains measurable events -- Numeric, additive measures -- Types of facts: -- Transaction: Individual events -- Periodic Snapshot: State at point in time -- Accumulating Snapshot: Lifecycle events CREATE TABLE fact_orders ( order_id INTEGER, date_key INTEGER, customer_key INTEGER, product_key INTEGER, -- Measures (numeric) quantity INTEGER, unit_price DECIMAL(10,2), discount_amount DECIMAL(10,2), tax_amount DECIMAL(10,2), total_amount DECIMAL(12,2) ); -- Degenerate dimension (order number) -- Stored in fact table, no separate dim

Dimension Tables

-- Contains descriptive attributes -- Text, categorical data CREATE TABLE dim_customer ( customer_key SERIAL PRIMARY KEY, -- Descriptive attributes customer_id INTEGER, full_name VARCHAR(100), email VARCHAR(255), segment VARCHAR(50), tier VARCHAR(20), registration_date DATE, -- SCD columns effective_date DATE, expiry_date DATE, is_current BOOLEAN ); -- Conformed dimension -- Used across multiple fact tables -- (dim_date used by sales, inventory, etc.)

Slowly Changing Dimensions Advanced

SCD methods handle how dimension data changes over time. Different types preserve different amounts of history.

-- SCD Type 0 (Fixed): No changes allowed -- Dimension data is static -- SCD Type 1 (Overwrite): No history kept UPDATE dim_customer SET city = 'Seattle', address = '456 Oak Ave' WHERE customer_id = 123; -- SCD Type 2 (Add Row): Full history CREATE TABLE dim_customer_scd2 ( customer_key SERIAL PRIMARY KEY, customer_id INTEGER, -- Natural key name VARCHAR(100), city VARCHAR(100), segment VARCHAR(50), effective_date DATE NOT NULL, expiry_date DATE, is_current BOOLEAN DEFAULT TRUE ); -- Insert new version when customer moves -- Step 1: Expire current record UPDATE dim_customer_scd2 SET expiry_date = '2024-03-14', is_current = FALSE WHERE customer_id = 123 AND is_current = TRUE; -- Step 2: Insert new version INSERT INTO dim_customer_scd2 (customer_id, name, city, segment, effective_date, is_current) VALUES (123, 'John Doe', 'Seattle', 'Premium', '2024-03-15', TRUE); -- Query current customer SELECT * FROM dim_customer_scd2 WHERE customer_id = 123 AND is_current = TRUE; -- Query historical sales by customer location at time of sale SELECT f.order_id, f.sales_amount, c.city AS city_at_purchase FROM fact_sales f JOIN dim_customer_scd2 c ON f.customer_key = c.customer_key AND f.sale_date BETWEEN c.effective_date AND COALESCE(c.expiry_date, '9999-12-31'); -- SCD Type 3 (Add Column): Limited history -- Keep previous value in separate column customer_scd3 ( customer_id, current_city, previous_city, city_change_date ); -- SCD Type 4 (Mini-dim): History table -- Main table + history table customer_current (customer_id, city, segment) customer_history (customer_id, city, segment, effective_date)

SCD Type Selection

Type 0 │ Type 1 │ Type 2 │ Type 3 │ Type 4
No Change → Overwrite → Add Row → Add Column → Mini-Dim

SCD Type 2 Process

Detect Change → Expire Current Row → Insert New Row → Update Foreign Keys
effective_date = NOW() → expiry_date = change_date → is_current = TRUE → Keep full history

Constraints Basic

Constraints enforce data integrity rules at the database level. They ensure data quality and consistency.

CREATE TABLE products ( id SERIAL, -- PRIMARY KEY constraint sku VARCHAR(50) PRIMARY KEY, -- NOT NULL constraint name VARCHAR(200) NOT NULL, -- UNIQUE constraint barcode VARCHAR(100) UNIQUE, -- CHECK constraint price DECIMAL(10,2) CHECK (price >= 0), -- CHECK with multiple conditions status VARCHAR(20) CHECK (status IN ('active', 'inactive', 'discontinued')), -- DEFAULT constraint created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW(), is_active BOOLEAN DEFAULT TRUE, stock_count INTEGER DEFAULT 0, -- FOREIGN KEY constraint (inline) category_id INTEGER REFERENCES categories(id), -- UNIQUE composite constraint UNIQUE (name, category_id) ); -- Named constraints (better for maintenance) ALTER TABLE products ADD CONSTRAINT chk_discount CHECK (discount BETWEEN 0 AND 1); -- Exclusion constraint (PostgreSQL) CREATE TABLE reservations ( id SERIAL PRIMARY KEY, room_id INTEGER, during TSRANGE, EXCLUDE USING gist (room_id WITH =, during WITH &&) ); -- Deferrable constraints ALTER TABLE orders ADD CONSTRAINT fk_customer FOREIGN KEY (customer_id) REFERENCES customers(id) DEFERRABLE INITIALLY DEFERRED;

Indexes Intermediate

Indexes improve query performance by creating efficient lookup structures. Trade-off: faster reads, slower writes.

-- B-tree index (default, most common) CREATE INDEX idx_customers_email ON customers(email); -- Unique index CREATE UNIQUE INDEX idx_users_username ON users(username); -- Composite index CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date DESC); -- Partial index (filtered) CREATE INDEX idx_orders_pending ON orders(created_at) WHERE status = 'pending'; -- Expression index CREATE INDEX idx_customers_lower_email ON customers(LOWER(email)); -- Hash index (equality only) CREATE INDEX idx_sessions_token ON sessions USING hash(session_token); -- GIN index (full-text, arrays, JSONB) CREATE INDEX idx_products_search ON products USING gin(search_vector); CREATE INDEX idx_orders_tags ON orders USING gin(tags); -- GiST index (geometric, range types) CREATE INDEX idx_locations_point ON locations USING gist(location_point); CREATE INDEX idx_events_period ON events USING gist(event_period); -- BRIN index (large sorted tables) CREATE INDEX idx_events_timestamp ON events USING brin(created_at); -- Covering index (includes extra columns) CREATE INDEX idx_orders_cover ON orders(customer_id) INCLUDE (order_date, total); -- Analyze index usage SELECT schemaname, tablename, indexname, idx_scan FROM pg_stat_user_indexes ORDER BY idx_scan DESC; -- Find unused indexes SELECT indexrelid::regclass AS index FROM pg_stat_user_indexes WHERE idx_scan = 0;

Index Selection Guide

Range/Sort Queries → B-tree
Full-text / JSONB / Arrays → GIN
Geometric / Range Types → GiST
Large Sorted Tables → BRIN
Equality Only → Hash

Index Performance Trade-offs

CREATE INDEX → Faster SELECT │ Slower INSERT │ Slower UPDATE │ More Storage

Table Partitioning Advanced

Partitioning splits large tables into smaller pieces. Improves query performance and maintenance operations.

-- Range partitioning (by date) CREATE TABLE events ( id BIGSERIAL, event_type VARCHAR(50), payload JSONB, created_at TIMESTAMPTZ NOT NULL ) PARTITION BY RANGE (created_at); -- Create partitions CREATE TABLE events_2024_q1 PARTITION OF events FOR VALUES FROM ('2024-01-01') TO ('2024-04-01'); CREATE TABLE events_2024_q2 PARTITION OF events FOR VALUES FROM ('2024-04-01') TO ('2024-07-01'); -- List partitioning (by region) CREATE TABLE regional_sales ( id BIGSERIAL, region VARCHAR(20) NOT NULL, amount DECIMAL(12,2) ) PARTITION BY LIST (region); CREATE TABLE sales_us PARTITION OF regional_sales FOR VALUES IN ('US', 'CA', 'MX'); CREATE TABLE sales_eu PARTITION OF regional_sales FOR VALUES IN ('UK', 'DE', 'FR', 'ES'); -- Hash partitioning (even distribution) CREATE TABLE user_sessions ( session_id UUID, user_id INTEGER, data JSONB ) PARTITION BY HASH (session_id); CREATE TABLE user_sessions_0 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 0); CREATE TABLE user_sessions_1 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 1); CREATE TABLE user_sessions_2 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 2); CREATE TABLE user_sessions_3 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 3); -- Sub-partitioning (range + list) CREATE TABLE metrics ( id BIGSERIAL, metric_date DATE NOT NULL, region VARCHAR(20), value DOUBLE PRECISION ) PARTITION BY RANGE (metric_date); CREATE TABLE metrics_2024 PARTITION OF metrics FOR VALUES FROM ('2024-01-01') TO ('2025-01-01') PARTITION BY LIST (region); -- Detach partition (for archiving) ALTER TABLE events DETACH PARTITION events_2023_q4; -- Attach partition ALTER TABLE events ATTACH PARTITION events_2024_q3 FOR VALUES FROM ('2024-07-01') TO ('2024-10-01');

Partition Type Selection

Time-series Data? → RANGE Partition
Fixed Categories? → LIST Partition
Even Distribution? → HASH Partition

Partition Management Flow

Create Parent Table → Create Partitions → Query Planning → Partition Pruning
PARTITION BY → PARTITION OF → Attach/Detach → Archive Old

Lakehouse Pattern Advanced

Lakehouse combines data lake flexibility with data warehouse reliability. Uses open table formats like Delta Lake, Iceberg, or Hudi.

-- Lakehouse architecture layers -- Layer 1: Bronze (Raw) -- Ingested raw data, append-only CREATE TABLE bronze_events ( raw_data JSONB, source_file VARCHAR(500), ingested_at TIMESTAMPTZ DEFAULT NOW() ); -- Layer 2: Silver (Cleaned) -- Validated, deduplicated, typed CREATE TABLE silver_events ( event_id UUID PRIMARY KEY, event_type VARCHAR(50), user_id INTEGER, event_time TIMESTAMPTZ, properties JSONB, created_at TIMESTAMPTZ DEFAULT NOW() ); -- Layer 3: Gold (Business) -- Aggregated, denormalized for analytics CREATE TABLE gold_daily_metrics ( metric_date DATE, product_id INTEGER, daily_revenue DECIMAL(12,2), daily_orders INTEGER, avg_order_value DECIMAL(10,2), unique_customers INTEGER, PRIMARY KEY (metric_date, product_id) ); -- Medallion pattern benefits: -- Bronze: Audit trail, replay capability -- Silver: Single source of truth -- Gold: Performance-optimized queries

Medallion Architecture Flow

Source Data → Bronze (Raw) → Silver (Clean) → Gold (Business)
Ingest → Validate → Transform → Aggregate

Data Lakehouse Benefits

Data Lake Flexibility + Data Warehouse Reliability = Lakehouse
ACID Transactions │ Schema Enforcement │ Time Travel │ Open Formats

Data Vault Modelling Advanced

Data Vault is a modelling approach for data warehouses that emphasizes auditability, flexibility, and scalability.

-- Data Vault Components -- Hubs: Business keys CREATE TABLE hub_customer ( hub_customer_key UUID PRIMARY KEY DEFAULT uuid_generate_v4(), customer_bk VARCHAR(50) NOT NULL, -- Business key load_date TIMESTAMPTZ DEFAULT NOW(), record_source VARCHAR(100), UNIQUE (customer_bk) ); -- Links: Relationships between hubs CREATE TABLE link_customer_order ( link_key UUID PRIMARY KEY DEFAULT uuid_generate_v4(), hub_customer_key UUID REFERENCES hub_customer(hub_customer_key), hub_order_key UUID REFERENCES hub_order(hub_order_key), load_date TIMESTAMPTZ DEFAULT NOW(), record_source VARCHAR(100), UNIQUE (hub_customer_key, hub_order_key) ); -- Satellites: Descriptive attributes with history CREATE TABLE sat_customer_details ( hub_customer_key UUID REFERENCES hub_customer(hub_customer_key), load_date TIMESTAMPTZ DEFAULT NOW(), load_end_date TIMESTAMPTZ, hash_diff UUID, -- Hash of all attributes name VARCHAR(100), email VARCHAR(255), segment VARCHAR(50), record_source VARCHAR(100), PRIMARY KEY (hub_customer_key, load_date) ); -- Vault to Business (Information Marts) -- Views or tables for analytics CREATE VIEW customer_360 AS SELECT h.customer_bk, s.name, s.email, s.segment, s.load_date AS last_updated FROM hub_customer h JOIN LATERAL ( SELECT * FROM sat_customer_details WHERE hub_customer_key = h.hub_customer_key ORDER BY load_date DESC LIMIT 1 ) s ON TRUE; -- Data Vault benefits: -- Auditability: Full history preserved -- Flexibility: Easy to add new sources -- Scalability: Parallel loading possible

Data Vault Components

Hubs ←→ Links ←→ Satellites
Business Keys → Relationships → Attributes + History

Data Vault to Information Marts

Source Systems → Raw Vault → Business Vault → Info Marts
Ingest → Hubs+Links+Sats → Business Logic → Star Schema Views

Data Modelling Best Practices Basic

Essential guidelines for effective data modelling in PostgreSQL.

Design Principles

  • Start with entities: Identify real-world objects first
  • Define relationships: Establish how entities relate
  • Normalize to 3NF: Then denormalize strategically
  • Use surrogate keys: Unless natural keys are truly stable
  • Name consistently: snake_case, plural table names
  • Document decisions: ER diagrams, data dictionaries

Common Mistakes

  • Over-normalization: Too many tables, slow queries
  • Under-normalization: Data redundancy, update anomalies
  • Missing indexes: Slow foreign key lookups
  • No constraints: Invalid data enters database
  • Wrong data types: VARCHAR for everything
  • Ignoring time: No timestamp columns for auditing
-- Well-designed example table CREATE TABLE orders ( -- Identity id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, order_number VARCHAR(20) UNIQUE NOT NULL, -- Foreign keys customer_id INTEGER NOT NULL REFERENCES customers(id), -- Business data order_date DATE NOT NULL DEFAULT CURRENT_DATE, status VARCHAR(20) NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'confirmed', 'shipped', 'delivered', 'cancelled')), -- Financial subtotal DECIMAL(12,2) NOT NULL CHECK (subtotal >= 0), tax_amount DECIMAL(10,2) NOT NULL DEFAULT 0, discount_amount DECIMAL(10,2) NOT NULL DEFAULT 0, total_amount DECIMAL(12,2) GENERATED ALWAYS AS (subtotal + tax_amount - discount_amount) STORED, -- Audit columns created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), created_by INTEGER REFERENCES users(id), -- Constraints CONSTRAINT chk_discount_valid CHECK (discount_amount <= subtotal) ); -- Auto-update updated_at CREATE OR REPLACE FUNCTION update_timestamp() RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = NOW(); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER orders_update_timestamp BEFORE UPDATE ON orders FOR EACH ROW EXECUTE FUNCTION update_timestamp(); -- Index for common queries CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date DESC); CREATE INDEX idx_orders_status ON orders(status) WHERE status NOT IN ('delivered', 'cancelled');

Design Checklist

Identify Entities → Define Keys → Map Relationships → Normalize → Add Constraints → Create Indexes

OLTP vs OLAP Decision

Transactional Workload? → 3NF + OLTP │ Analytical Workload? → Star Schema + OLAP
Many small writes → ACID compliance │ Few large reads → Denormalized dims
