SQL Joins

Combine data from multiple tables using different join strategies in PostgreSQL.

Join Concepts Basic

Joins combine rows from two or more tables based on related columns. Understanding join types is essential for relational database queries.

Sample Tables

-- employees table CREATE TABLE employees ( id INT, name VARCHAR(100), dept_id INT, manager_id INT ); -- departments table CREATE TABLE departments ( id INT, dept_name VARCHAR(50), location VARCHAR(50) );

Join Types Overview

  • INNER JOIN - Matching rows in both tables
  • LEFT JOIN - All rows from left table
  • RIGHT JOIN - All rows from right table
  • FULL JOIN - All rows from both tables
  • CROSS JOIN - Cartesian product

INNER JOIN Basic

Returns only rows that have matching values in both tables. This is the most common join type.

-- Basic INNER JOIN SELECT e.name, e.salary, d.dept_name FROM employees e INNER JOIN departments d ON e.dept_id = d.id; -- Equivalent using WHERE (old style) SELECT e.name, d.dept_name FROM employees e, departments d WHERE e.dept_id = d.id; -- Join on multiple columns SELECT * FROM sales s INNER JOIN products p ON s.product_id = p.id AND s.region = p.region; -- Join with additional filters SELECT e.name, d.dept_name FROM employees e INNER JOIN departments d ON e.dept_id = d.id WHERE e.salary > 50000 AND d.location = 'NYC'; -- Using USING (when column names match) SELECT name, dept_name FROM employees INNER JOIN departments USING (dept_id); -- Join with table aliases SELECT e.name AS employee, d.dept_name AS department, p.project_name FROM employees e JOIN departments d ON e.dept_id = d.id JOIN projects p ON p.lead_id = e.id;

NATURAL JOIN Intermediate

Automatically joins tables based on columns with matching names. Use with caution.

-- NATURAL JOIN (joins on all matching column names) SELECT name, dept_name FROM employees NATURAL JOIN departments; -- Implicitly: ON employees.dept_id = departments.id -- NATURAL LEFT JOIN SELECT name, dept_name FROM employees NATURAL LEFT JOIN departments; -- WARNING: Be careful with NATURAL JOIN! -- If both tables have columns like 'created_at', 'updated_at', -- they will be used in the join condition unexpectedly. -- Safer alternative: Use USING SELECT name, dept_name FROM employees JOIN departments USING (dept_id); -- PostgreSQL also supports NATURAL with specific types SELECT * FROM table1 NATURAL INNER JOIN table2;

Warning: NATURAL JOIN can cause unexpected behavior if tables have columns with the same name that shouldn't be used for joining. Prefer explicit ON or USING clauses in production code.

LEFT JOIN Basic

Returns all rows from the left table and matching rows from the right table. Non-matching rows show NULL for right table columns.

-- Basic LEFT JOIN SELECT e.name, e.salary, COALESCE(d.dept_name, 'No Department') AS department FROM employees e LEFT JOIN departments d ON e.dept_id = d.id; -- Find employees without a department SELECT e.name FROM employees e LEFT JOIN departments d ON e.dept_id = d.id WHERE d.id IS NULL; -- LEFT JOIN with USING SELECT name, dept_name FROM employees LEFT JOIN departments USING (dept_id); -- Find products without sales SELECT p.product_name FROM products p LEFT JOIN sales s ON p.id = s.product_id WHERE s.id IS NULL; -- Count with LEFT JOIN (includes zeros) SELECT d.dept_name, COUNT(e.id) AS employee_count FROM departments d LEFT JOIN employees e ON d.id = e.dept_id GROUP BY d.dept_name;

Tip: Use LEFT JOIN when you need all rows from the left table, regardless of whether there's a match. It's commonly used for "find records without X" queries.

RIGHT JOIN Basic

Returns all rows from the right table and matching rows from the left table. Less commonly used than LEFT JOIN.

-- Basic RIGHT JOIN SELECT e.name, d.dept_name FROM employees e RIGHT JOIN departments d ON e.dept_id = d.id; -- Find departments without employees SELECT d.dept_name FROM employees e RIGHT JOIN departments d ON e.dept_id = d.id WHERE e.id IS NULL; -- RIGHT JOIN is just LEFT JOIN with tables swapped -- These are equivalent: SELECT e.name, d.dept_name FROM employees e RIGHT JOIN departments d ON e.dept_id = d.id; SELECT e.name, d.dept_name FROM departments d LEFT JOIN employees e ON e.dept_id = d.id; -- PostgreSQL: RIGHT JOIN with USING SELECT * FROM employees e RIGHT JOIN departments d USING (dept_id);

Best Practice: Most developers prefer LEFT JOIN over RIGHT JOIN for readability. Simply swap the table order to convert RIGHT JOIN to LEFT JOIN.

FULL OUTER JOIN Intermediate

Returns all rows from both tables, matching where possible and showing NULL where not.

-- Basic FULL OUTER JOIN SELECT e.name AS employee_name, d.dept_name FROM employees e FULL OUTER JOIN departments d ON e.dept_id = d.id; -- Results include: -- 1. Employees with departments (matched) -- 2. Employees without departments (NULL dept_name) -- 3. Departments without employees (NULL employee_name) -- Find all unmatched rows (orphaned records) SELECT COALESCE(e.name, '[No Employee]') AS employee, COALESCE(d.dept_name, '[No Department]') AS department FROM employees e FULL OUTER JOIN departments d ON e.dept_id = d.id WHERE e.id IS NULL OR d.id IS NULL; -- PostgreSQL: FULL OUTER JOIN with USING SELECT * FROM employees FULL OUTER JOIN departments USING (dept_id);

Note: PostgreSQL supports FULL OUTER JOIN natively. MySQL does not - you need to simulate it with LEFT JOIN UNION RIGHT JOIN.

CROSS JOIN Intermediate

Returns the Cartesian product - every row from the first table paired with every row from the second.

-- Basic CROSS JOIN SELECT e.name, p.project_name FROM employees e CROSS JOIN projects p; -- If 100 employees and 10 projects = 1000 rows -- Implicit CROSS JOIN syntax SELECT e.name, p.project_name FROM employees e, projects p; -- Generate combinations (size × color) SELECT s.size, c.color FROM (VALUES ('S'), ('M'), ('L')) AS s(size) CROSS JOIN (VALUES ('Red'), ('Blue'), ('Green')) AS c(color); -- Cross join for calendar/report generation SELECT d.date, e.name FROM generate_series( '2024-01-01'::DATE, '2024-01-31'::DATE, '1 day'::INTERVAL ) AS d(date) CROSS JOIN employees e; -- Cross join with WHERE (becomes inner join) SELECT e.name, d.dept_name FROM employees e CROSS JOIN departments d WHERE e.dept_id = d.id; -- Now it's like INNER JOIN

Self Join Intermediate

Join a table with itself to find relationships within the same data.

-- Find employees and their managers SELECT e.name AS employee, m.name AS manager FROM employees e LEFT JOIN employees m ON e.manager_id = m.id; -- Find employees earning more than their manager SELECT e.name AS employee, e.salary AS emp_salary, m.name AS manager, m.salary AS mgr_salary FROM employees e JOIN employees m ON e.manager_id = m.id WHERE e.salary > m.salary; -- Find employees in the same department (pairs) SELECT e1.name AS employee1, e2.name AS employee2, e1.dept_id FROM employees e1 JOIN employees e2 ON e1.dept_id = e2.dept_id AND e1.id < e2.id; -- Avoid duplicates and self-pairs -- Find consecutive orders (for gap analysis) SELECT o1.order_date, o1.customer_id, o1.order_date - o2.order_date AS days_since_last FROM orders o1 LEFT JOIN orders o2 ON o1.customer_id = o2.customer_id AND o2.order_date = ( SELECT MAX(order_date) FROM orders o3 WHERE o3.customer_id = o1.customer_id AND o3.order_date < o1.order_date ); -- Find mutual follows (social network) SELECT f1.follower_id, f1.following_id FROM follows f1 JOIN follows f2 ON f1.follower_id = f2.following_id AND f1.following_id = f2.follower_id;

LATERAL Join Advanced

PostgreSQL's LATERAL join allows a subquery to reference columns from preceding tables in the FROM clause.

-- Top 3 earners per department SELECT d.dept_name, top_employees.name, top_employees.salary FROM departments d LEFT JOIN LATERAL ( SELECT name, salary FROM employees e WHERE e.dept_id = d.id ORDER BY salary DESC LIMIT 3 ) top_employees ON TRUE; -- Latest order per customer with details SELECT c.name, latest.order_date, latest.total FROM customers c LEFT JOIN LATERAL ( SELECT order_date, total FROM orders o WHERE o.customer_id = c.id ORDER BY order_date DESC LIMIT 1 ) latest ON TRUE; -- Unnest arrays with LATERAL SELECT p.id, tag FROM posts p, LATERAL UNNEST(p.tags) AS tag; -- Generate series per employee SELECT e.name, month FROM employees e CROSS JOIN LATERAL generate_series(1, 12) AS month;

Pro tip: LATERAL joins are powerful for "top N per group" queries and are often more efficient than correlated subqueries in PostgreSQL.

Multiple Joins Intermediate

Chain multiple joins together to combine data from three or more tables.

-- Three-table join SELECT e.name AS employee, d.dept_name, p.project_name FROM employees e JOIN departments d ON e.dept_id = d.id JOIN project_assignments pa ON pa.employee_id = e.id JOIN projects p ON p.id = pa.project_id WHERE d.location = 'NYC'; -- Complex query with all join types SELECT c.customer_name, o.order_date, p.product_name, oi.quantity, oi.quantity * p.price AS line_total FROM customers c LEFT JOIN orders o ON c.id = o.customer_id INNER JOIN order_items oi ON o.id = oi.order_id INNER JOIN products p ON oi.product_id = p.id WHERE o.order_date >= '2024-01-01'; -- Using subquery in join SELECT e.name, dept_stats.avg_salary FROM employees e JOIN ( SELECT dept_id, AVG(salary) AS avg_salary FROM employees GROUP BY dept_id ) dept_stats ON e.dept_id = dept_stats.dept_id WHERE e.salary > dept_stats.avg_salary; -- Join with CTE WITH dept_summary AS ( SELECT dept_id, COUNT(*) AS count FROM employees GROUP BY dept_id ) SELECT e.name, ds.count FROM employees e JOIN dept_summary ds ON e.dept_id = ds.dept_id;

Complex Join Conditions Intermediate

Joins can use inequality operators and complex conditions beyond simple equality.

-- Range joins (find overlapping periods) SELECT b1.booking_id AS booking1, b2.booking_id AS booking2, b1.room_id FROM bookings b1 JOIN bookings b2 ON b1.room_id = b2.room_id AND b1.booking_id != b2.booking_id AND b1.start_date < b2.end_date AND b1.end_date > b2.start_date; -- Price range joins SELECT o.order_id, p.promo_code FROM orders o JOIN promotions p ON o.order_total >= p.min_order AND o.order_total <= p.max_order AND o.order_date BETWEEN p.start_date AND p.end_date; -- Non-equi join (closest value) SELECT e.name, e.salary, b.bracket FROM employees e JOIN salary_brackets b ON e.salary >= b.min_salary AND e.salary < b.max_salary; -- Self-referential range join (date gaps) SELECT d1.date AS gap_start, MIN(d2.date) - 1 AS gap_end FROM dates d1 LEFT JOIN dates d2 ON d2.date > d1.date WHERE NOT EXISTS ( SELECT 1 FROM dates d3 WHERE d3.date = d1.date + 1 ) GROUP BY d1.date;

Anti & Semi Joins Advanced

Semi joins return rows that match, anti joins return rows that don't match. PostgreSQL has efficient patterns for these.

Semi Join ( EXISTS )

-- Find employees who have placed orders -- Using EXISTS (semi join) SELECT e.name FROM employees e WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.employee_id = e.id ); -- Equivalent to IN (but often faster) SELECT name FROM employees WHERE id IN ( SELECT employee_id FROM orders );

Anti Join (NOT EXISTS)

-- Find products never ordered -- Using NOT EXISTS (anti join) SELECT p.product_name FROM products p WHERE NOT EXISTS ( SELECT 1 FROM order_items oi WHERE oi.product_id = p.id ); -- Using LEFT JOIN + NULL check SELECT p.product_name FROM products p LEFT JOIN order_items oi ON p.id = oi.product_id WHERE oi.product_id IS NULL; -- NOT IN can fail with NULLs! -- Use NOT EXISTS or LEFT JOIN instead

Warning: Avoid NOT IN with subqueries that may contain NULL values - it will return no results. Use NOT EXISTS or LEFT JOIN ... WHERE IS NULL instead.
