AWS Lambda Basic

Lambda runs code without provisioning servers. Pay only for compute time consumed.

# Lambda with Python import json import boto3 s3 = boto3.client('s3') def lambda_handler(event, context): # Get S3 event bucket = event['Records'][0]['s3']['bucket']['name'] key = event[0]['s3']['object']['key'] # Process file response = s3.get_object(Bucket=bucket, Key=key) data = response['Body'].read() # Transform and save processed = process_data(data) s3.put_object(Bucket='output-bucket', Key=key, Body=processed) return {'statusCode': 200, 'body': 'OK'} # Deploy with AWS CLI aws lambda create-function \ --function-name process-file \ --runtime python3.11 \ --handler lambda_function.lambda_handler \ --role arn:aws:iam::123456789:role/lambda-role \ --zip-file fileb://function.zip # Lambda limits (default) # Memory: 128 MB - 10 GB # Timeout: 15 minutes max # Package size: 50 MB (zipped), 250 MB (unzipped) # Concurrent executions: 1000 (default)

Azure Functions Basic

Azure Functions provides event-driven serverless compute with support for multiple languages.

# Azure Functions with Python import azure.functions as func import logging app = func.FunctionApp() @app.route(route="hello", methods=["GET"]) def hello(req: func.HttpRequest) -> func.HttpResponse: name = req.params.get('name') return func.HttpResponse( f"Hello, {name}!", status_code=200 ) # Create and deploy func init my-app --python func new --name http-trigger --template "HTTP trigger" func azure functionapp publish my-app # Function types # HTTP Trigger - REST APIs # Timer Trigger - Scheduled jobs # Blob Trigger - File uploads # Queue Trigger - Message processing # Event Hub Trigger - Streaming # Cosmos DB Trigger - Document changes

Google Cloud Functions Basic

Google Cloud Functions runs event-driven code with automatic scaling and pay-per-use pricing.

# Cloud Functions Gen2 def hello_http(request): request_json = request.get_json() if request.args and 'name' in request.args: name = request.args.get('name') elif request_json and 'name' in request_json: name = request_json['name'] else: name = 'World' return f'Hello {name}!' # Deploy gcloud functions deploy hello-http \ --gen2 \ --runtime=python311 \ --region=us-central1 \ --trigger-http \ --allow-unauthenticated # Event-driven triggers gcloud functions deploy process-upload \ --gen2 \ --runtime=python311 \ --trigger-event-filters="type=google.cloud.storage.object.v1.finalized" \ --trigger-event-filters="bucket=my-bucket" # Limits: # Memory: 128 MB - 32 GB # Timeout: 9 min (HTTP), 60 min (event) # Max instances: 1000 (default)

Workflow Orchestration Intermediate

Serverless workflow services coordinate multiple functions and services into business processes.

# AWS Step Functions (State Machine) { "Comment": "Order Processing Workflow", "StartAt": "ValidateOrder", "States": { "ValidateOrder": { "Type": "Task", "Resource": "arn:aws:lambda:us-east-1:123:function:validate", "Next": "ProcessPayment", "Catch": [{ "ErrorEquals": ["States.ALL"], "Next": "HandleError" }] }, "ProcessPayment": { "Type": "Task", "Resource": "arn:aws:lambda:us-east-1:123:function:payment", "Next": "FulfillOrder" }, "FulfillOrder": { "Type": "Task", "Resource": "arn:aws:lambda:us-east-1:123:function:fulfill", "End": true }, "HandleError": { "Type": "Task", "Resource": "arn:aws:lambda:us-east-1:123:function:error", "End": true } } } # Azure Durable Functions # Google Cloud Workflows

API Gateway Intermediate

API Gateway creates REST/HTTP APIs that route requests to Lambda functions or other backend services.

# AWS API Gateway + Lambda # Create REST API aws apigateway create-rest-api \ --name my-api \ --description "My API" # Create resource and method aws apigateway create-resource \ --rest-api-id abc123 \ --parent-id xyz789 \ --path-part hello aws apigateway put-method \ --rest-api-id abc123 \ --resource-id def456 \ --http-method GET \ --authorization-type NONE aws apigateway put-integration \ --rest-api-id abc123 \ --resource-id def456 \ --http-method GET \ --type AWS_PROXY \ --integration-uri arn:aws:lambda:us-east-1:123:function:hello # Deploy API aws apigateway create-deployment \ --rest-api-id abc123 \ --stage-name prod # Result: https://abc123.execute-api.us-east-1.amazonaws.com/prod/hello # Azure API Management # Google Cloud API Gateway / API Gateway

Event-Driven Architecture Intermediate

Event-driven architecture uses events to trigger and communicate between decoupled services.

# Event Sources # AWS Event Sources for Lambda - S3 (object created/deleted) - DynamoDB Streams (table changes) - SQS (message in queue) - SNS (notification) - EventBridge (scheduled/custom events) - API Gateway (HTTP request) - Kinesis (stream records) # EventBridge Rule (scheduled) { "ScheduleExpression": "rate(5 minutes)", "Target": { "Arn": "arn:aws:lambda:us-east-1:123:function:process" } } # EventBridge Rule (custom event) { "detail-type": ["OrderPlaced"], "source": ["com.myapp.orders"], "detail": { "status": ["confirmed"] } }

Event-Driven Flow

Event Source → Event Bus/Queue → Lambda/Function → Action

Fan-Out Pattern Intermediate

Fan-out pattern processes tasks in parallel by distributing work to multiple consumers.

# Fan-Out Pattern (SNS + SQS) # 1. SNS Topic broadcasts event aws sns create-topic --name order-events # 2. Multiple SQS queues subscribe aws sqs create-queue --name email-queue aws sqs create-queue --name inventory-queue aws sqs create-queue --name analytics-queue aws sns subscribe \ --topic-arn arn:aws:sns:us-east-1:123:order-events \ --protocol sqs \ --notification-endpoint arn:aws:sqs:us-east-1:123:email-queue # 3. Each Lambda processes from its queue # email-processor Lambda # inventory-processor Lambda # analytics-processor Lambda # Benefits: # - Parallel processing # - Independent scaling # - Failure isolation # - Retry handling per consumer

Fan-Out Architecture

Order Event → SNS Topic
↓
Email Queue → Email Lambda
Inventory Queue → Inventory Lambda
Analytics Queue → Analytics Lambda

Saga Pattern Advanced

Saga pattern manages distributed transactions by defining compensating actions for each step.

# Saga Pattern (Order Processing) # Steps with compensations: 1. Reserve Inventory → Compensate: Release Inventory 2. Process Payment → Compensate: Refund Payment 3. Create Shipment → Compensate: Cancel Shipment 4. Update Order Status → Compensate: Revert Status # Step Functions Saga { "StartAt": "ReserveInventory", "States": { "ReserveInventory": { "Type": "Task", "Resource": "...inventory:reserve", "Catch": [{ "Next": "Fail" }], "Next": "ProcessPayment" }, "ProcessPayment": { "Type": "Task", "Resource": "...payment:charge", "Catch": [{ "Next": "ReleaseInventory" # Compensate }], "Next": "CreateShipment" }, "ReleaseInventory": { "Type": "Task", "Resource": "...inventory:release", "Next": "Fail" } } } # Alternatives: # - Choreography: Services react to events # - Orchestration: Central coordinator (Step Functions)

Saga Compensation Flow

Reserve Inventory → Process Payment → Create Shipment
Release Inventory ← Refund Payment ← Cancel Shipment
