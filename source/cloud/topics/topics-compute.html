Virtual Machines Basic

Virtual machines (VMs) are emulated computers that run on physical hardware via hypervisors. They provide full control over the OS and software stack.

# AWS EC2 # Launch instance aws ec2 run-instances \ --image-id ami-0abc123 \ --instance-type t3.micro \ --key-name my-key \ --security-group-ids sg-123456 \ --subnet-id subnet-123456 # List instances aws ec2 describe-instances \ --filters "Name=instance-state-name,Values=running" # Stop / Start / Terminate aws ec2 stop-instances --instance-ids i-123456 aws ec2 start-instances --instance-ids i-123456 aws ec2 terminate-instances --instance-ids i-123456 # Azure VM az vm create \ --resource-group myRG \ --name myVM \ --image Ubuntu2204 \ --admin-username azureuser \ --generate-ssh-keys # GCP Compute Engine gcloud compute instances create my-vm \ --zone us-central1-a \ --machine-type e2-medium \ --image-family ubuntu-2204-lts \ --image-project ubuntu-os-cloud

VM Instance Types Basic

VM types are optimized for different workloads: general purpose, compute-optimized, memory-optimized, and GPU.

# AWS Instance Families # General Purpose (balanced) t3, t3a, t4g # Burstable (web servers) m5, m6i, m7i # Fixed (app servers) # Compute Optimized (CPU-intensive) c5, c6i, c7i # Batch processing, HPC # Memory Optimized (RAM-intensive) r5, r6i, r7i # In-memory databases x1, x2ie # SAP HANA, in-memory analytics # Storage Optimized (I/O-intensive) i3, i4i # NoSQL databases, data warehousing d3, d3en # Hadoop, data processing # GPU (ML/Video) g4dn, g5 # ML inference, graphics p4d, p5 # ML training inf1, inf2 # ML inference (AWS Inferentia) # Naming convention: family + generation + size # m5.2xlarge = m family, 5th gen, 2xlarge size # 2xlarge = 8 vCPUs, 32 GB RAM

Instance Type Selection

Web Server? → General (t3/m5)
ML Training? → GPU (p4d/g5)
In-Memory DB? → Memory (r5/x1)

Container Services Intermediate

Containers package applications with their dependencies, ensuring consistent environments across development, testing, and production.

# Docker basics # Build image docker build -t my-app:1.0 . # Run container docker run -d -p 8080:80 --name my-app my-app:1.0 # Docker Compose # docker-compose.yml version: '3.8' services: web: image: nginx:latest ports: - "80:80" api: build: ./api environment: - DATABASE_URL=postgres://db:5432/mydb db: image: postgres:15 volumes: - pgdata:/var/lib/postgresql/data volumes: pgdata: # Run multi-container app docker-compose up -d

Kubernetes (K8s) Advanced

Kubernetes is the industry-standard container orchestration platform for automating deployment, scaling, and management.

# Kubernetes key concepts # Pod - Smallest deployable unit # Deployment - Manages ReplicaSets # Service - Network endpoint # ConfigMap / Secret - Configuration # Ingress - HTTP routing # deployment.yaml apiVersion: apps/v1 kind: Deployment metadata: name: my-app spec: replicas: 3 selector: matchLabels: app: my-app template: metadata: labels: app: my-app spec: containers: - name: my-app image: my-app:1.0 ports: - containerPort: 80 # Apply configuration kubectl apply -f deployment.yaml # View resources kubectl get pods kubectl get deployments kubectl get services # Scale deployment kubectl scale deployment my-app --replicas=5 # View logs kubectl logs -f deployment/my-app

AWS ECS / Fargate Intermediate

ECS (Elastic Container Service) is Amazon's managed container orchestration. Fargate adds serverless compute for containers.

# ECS Task Definition { "family": "my-app", "networkMode": "awsvpc", "requiresCompatibilities": ["FARGATE"], "cpu": "256", "memory": "512", "containerDefinitions": [{ "name": "app", "image": "my-app:latest", "portMappings": [{ "containerPort": 8080, "protocol": "tcp" }] }] } # Create service aws ecs create-service \ --cluster my-cluster \ --service-name my-service \ --task-definition my-app \ --desired-count 3 \ --launch-type FARGATE # Fargate benefits: # - No server management # - Pay per task (CPU/memory) # - Auto-scaling built-in

Azure AKS Intermediate

Azure Kubernetes Service (AKS) simplifies Kubernetes deployment with managed control plane.

# Create AKS cluster az aks create \ --resource-group myRG \ --name myAKSCluster \ --node-count 3 \ --node-vm-size Standard_B2s \ --enable-managed-identity \ --generate-ssh-keys # Get credentials az aks get-credentials \ --resource-group myRG \ --name myAKSCluster # Deploy app kubectl apply -f deployment.yaml # Scale az aks scale \ --resource-group myRG \ --name myAKSCluster \ --node-count 5 # AKS features: # - Free control plane # - Azure AD integration # - Azure Monitor integration # - Virtual Node scaling

Google GKE Intermediate

Google Kubernetes Engine (GKE) provides production-ready Kubernetes with deep Google Cloud integration.

# Create GKE cluster gcloud container clusters create my-cluster \ --zone us-central1-a \ --num-nodes 3 \ --machine-type e2-medium \ --enable-autoscaling \ --min-nodes 1 \ --max-nodes 10 # Get credentials gcloud container clusters get-credentials my-cluster \ --zone us-central1-a # Deploy to GKE kubectl apply -f deployment.yaml # GKE features: # - Autopilot mode (fully managed) # - Binary Authorization # - Workload Identity # - Node auto-repair/replacement

Machine Images Basic

Machine images are pre-configured templates containing OS, software, and application code.

# AWS AMI (Amazon Machine Image) # Find Ubuntu AMI aws ec2 describe-images \ --owners 099720109477 \ --filters "Name=name,Values=ubuntu/images/hvm-ssd/ubuntu-jammy-22.04*" \ --query 'Images[*].[ImageId,Name]' \ --output table # Create custom AMI from instance aws ec2 create-image \ --instance-id i-123456 \ --name "my-app-v1.0" \ --description "Application v1.0" # Azure Image az image create \ --resource-group myRG \ --name myImage \ --source myVM # GCP Custom Image gcloud compute images create my-image \ --source-disk my-disk \ --source-disk-zone us-central1-a # Packer - Create images across clouds # HashiCorp tool for machine images

Infrastructure as Code Intermediate

IaC defines infrastructure using code, enabling version control, reproducibility, and automation.

# Terraform (Multi-cloud) # main.tf provider "aws" { region = "us-east-1" } resource "aws_instance" "web" { ami = "ami-0abc123" instance_type = "t3.micro" tags = { Name = "my-instance" } } # Deploy terraform init terraform plan terraform apply # AWS CloudFormation (AWS-only) # Azure ARM / Bicep (Azure-only) # GCP Deployment Manager (GCP-only) # Pulumi (Multi-cloud, real languages) import * as aws from "@pulumi/aws" const instance = new aws.ec2.Instance("web", { ami: "ami-0abc123", instanceType: "t3.micro", });

IaC Workflow

Write Code → Plan (terraform plan) → Apply (terraform apply) → Infrastructure
