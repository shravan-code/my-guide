Cloud Basics

A focused guide to Cloud Basics.

IaaS (Infrastructure as a Service) Basic

IaaS provides virtualized computing resources over the internet. You manage the OS, middleware, and applications while the provider manages the hardware.

What You Manage

  • Operating system
  • Runtime environment
  • Applications
  • Data
  • Middleware

Provider Manages

  • Physical servers
  • Networking hardware
  • Storage hardware
  • Virtualization
  • Power and cooling
# AWS EC2 (IaaS example) aws ec2 run-instances \ --image-id ami-0abc123 \ --instance-type t3.medium \ --key-name my-key # Azure VM (IaaS example) az vm create \ --resource-group myRG \ --name myVM \ --image Ubuntu2204 \ --size Standard_B2s # GCP Compute Engine (IaaS example) gcloud compute instances create my-vm \ --zone us-central1-a \ --machine-type e2-medium \ --image-family ubuntu-2204-lts

IaaS Responsibility Model

Applications ↓ Data ↓ Runtime ↓ Middleware ↓ OS
You manage ↑ │ Provider manages ↓
Virtualization ↓ Servers ↓ Storage ↓ Networking

PaaS (Platform as a Service) Basic

PaaS provides a managed platform for developing, running, and managing applications without managing infrastructure.

What You Manage

  • Applications
  • Data

Provider Manages

  • Everything in IaaS
  • OS patching
  • Runtime updates
  • Middleware
# AWS Elastic Beanstalk (PaaS) eb init my-app --platform node.js eb create my-environment eb deploy # Azure App Service (PaaS) az webapp up \ --resource-group myRG \ --name my-app \ --runtime NODE:18-lts # Google App Engine (PaaS) gcloud app deploy # Heroku (PaaS) git push heroku main

SaaS (Software as a Service) Basic

SaaS delivers fully managed software applications over the internet. Users access via browser or API without managing anything.

What You Manage

  • Your data
  • User access
  • Configuration

Provider Manages

  • Everything else
  • Application code
  • Infrastructure
  • Updates and patches
# Common SaaS Examples # Collaboration - Microsoft 365 (Word, Excel, Teams) - Google Workspace (Docs, Sheets, Gmail) - Slack, Zoom, Dropbox # CRM / Business - Salesforce - HubSpot - ServiceNow # Data & Analytics - Snowflake - Databricks - MongoDB Atlas # Developer Tools - GitHub - GitLab - Datadog

Serverless Computing Intermediate

Serverless abstracts server management completely. You write functions, and the platform handles scaling, patching, and availability.

# AWS Lambda (Serverless Function) import json def lambda_handler(event, context): name = event.get('name', 'World') return { 'statusCode': 200, 'body': json.dumps({'message': f'Hello {name}!'}) } # Deploy aws lambda create-function \ --function-name my-function \ --runtime python3.11 \ --handler lambda_function.lambda_handler \ --role arn:aws:iam::123456789:role/lambda-role # Azure Functions func init my-app --python func new --name http-trigger --template "HTTP trigger" func azure functionapp publish my-app

Serverless Benefits

No Server Management │ Auto-scaling │ Pay per Execution
Event-driven │ High Availability │ Managed Runtime

Public Cloud Basic

Public cloud resources are owned and operated by a third-party provider, shared across multiple organizations.

Advantages

  • No capital expenditure
  • Pay-as-you-go pricing
  • Massive scalability
  • Global infrastructure

Considerations

  • Less control over hardware
  • Data residency concerns
  • Shared infrastructure
  • Internet dependency

Major Public Cloud Providers

AWS │ Azure │ GCP

Private Cloud Basic

Private cloud infrastructure is dedicated to a single organization, either on-premises or hosted by a third party.

Advantages

  • Full control over infrastructure
  • Enhanced security
  • Regulatory compliance
  • Custom configurations

Considerations

  • Higher upfront costs
  • Requires IT expertise
  • Limited scalability
  • Maintenance overhead
# Private cloud platforms # VMware vSphere - Full virtualization stack - vCenter management - NSX networking # OpenStack - Open source cloud platform - Nova (compute), Swift (storage), Neutron (network) - Kubernetes integration # Azure Stack / Azure Arc - Extend Azure to on-premises - Consistent hybrid management

Hybrid Cloud Intermediate

Hybrid cloud combines public and private clouds with orchestration between them, enabling data and application portability.

Hybrid Cloud Architecture

On-Premises ←→ Orchestration Layer ←→ Public Cloud
Legacy Apps │ VPN / Direct Connect │ Cloud-Native Apps

Use Cases

  • Gradual cloud migration
  • Regulatory compliance (data residency)
  • Burst capacity (cloud bursting)
  • Edge computing integration

Tools

  • Azure Arc
  • AWS Outposts
  • Google Anthos
  • VMware Cloud

Multi-Cloud Intermediate

Multi-cloud uses services from multiple cloud providers to avoid vendor lock-in and optimize for best-of-breed services.

Benefits

  • Best-of-breed services
  • Vendor independence
  • Geographic coverage
  • Cost optimization

Challenges

  • Complexity
  • Data transfer costs
  • Inconsistent APIs
  • Skills gap
# Multi-cloud abstraction with Terraform # AWS Provider provider "aws" { region = "us-east-1" } # Azure Provider provider "azurerm" { features {} } # GCP Provider provider "google" { project = "my-project" region = "us-central1" } # Kubernetes abstraction with Pulumi or Crossplane # Unified API across clouds

Scalability Basic

Scalability is the ability to handle growing workload by adding resources.

Vertical Scaling (Scale Up)

Small VM → Large VM
  • Add more CPU/RAM to existing machine
  • Simpler, no code changes
  • Limited by hardware ceiling
  • Often requires downtime

Horizontal Scaling (Scale Out)

1 VM → N VMs
  • Add more machines
  • Near-unlimited capacity
  • Requires load balancing
  • Better fault tolerance
# AWS Auto Scaling Group (Horizontal) aws autoscaling create-auto-scaling-group \ --auto-scaling-group-name my-asg \ --min-size 2 \ --max-size 10 \ --desired-capacity 3 \ --vpc-zone-identifier "subnet-123,subnet-456" # Azure VM Scale Set (Horizontal) az vmss create \ --name my-vmss \ --capacity 3 \ --min-capacity 1 \ --max-capacity 10

Elasticity Basic

Elasticity is the ability to automatically scale resources up and down based on demand.

Elasticity Flow

Load Increase → CloudWatch Alarm → Auto Scale Out
Load Decrease → Cooldown Period → Scale In
# Auto-scaling policy (AWS) # Scale out when CPU > 70% aws autoscaling put-scaling-policy \ --policy-name scale-out \ --auto-scaling-group-name my-asg \ --scaling-adjustment 1 \ --adjustment-type ChangeInCapacity # CloudWatch alarm to trigger aws cloudwatch put-metric-alarm \ --alarm-name cpu-high \ --metric-name CPUUtilization \ --threshold 70 \ --comparison-operator GreaterThanThreshold \ --period 300 \ --evaluation-periods 2 \ --statistic Average # Serverless is inherently elastic # Lambda: 0 to 1000 concurrent executions # DynamoDB: Auto-scaling read/write capacity

High Availability (HA) Intermediate

High availability ensures applications remain operational despite failures, targeting 99.9%+ uptime.

# HA Patterns # 1. Multi-AZ Deployment # Deploy across availability zones aws rds create-db-instance \ --multi-az \ --db-instance-identifier my-db # 2. Load Balancer + Auto Scaling # Distribute traffic across instances aws elbv2 create-load-balancer \ --name my-alb \ --subnets subnet-123 subnet-456 # 3. Database Replication # Read replicas for failover aws rds create-db-instance-read-replica \ --db-instance-identifier my-replica \ --source-db-instance-identifier my-db # 4. CDN for static assets # CloudFront, Azure CDN, Cloud CDN # Uptime Targets 99% = 8.76 hours downtime/year 99.9% = 8.76 hours downtime/year 99.99% = 52.56 minutes downtime/year 99.999% = 5.26 minutes downtime/year

HA Architecture

Route 53 / DNS → Load Balancer → Auto Scaling Group
AZ-1a │ AZ-1b │ AZ-1c

Pricing Models Basic

Cloud providers offer multiple pricing models to optimize costs for different usage patterns.

On-Demand

  • Pay per second/hour
  • No upfront commitment
  • Best for: unpredictable workloads
  • Most expensive per hour

Reserved / Committed

  • 1-3 year commitment
  • 30-70% discount
  • Best for: steady-state workloads
  • Pay upfront or monthly

Spot / Preemptible

  • Unused capacity
  • Up to 90% discount
  • Can be terminated with 2 min notice
  • Best for: fault-tolerant batch jobs

Savings Plans

  • Flexible commitment (not instance-specific)
  • 1-3 year term
  • Covers compute, Fargate, Lambda
  • Best for: consistent spend

Pricing Decision

Steady workload? → Reserved/Savings
Batch processing? → Spot Instances
Variable/unpredictable? → On-Demand

Total Cost of Ownership (TCO) Intermediate

TCO includes all costs of cloud computing beyond just the compute price.

# Cloud cost components # Direct costs - Compute (VMs, containers, functions) - Storage (capacity, transactions, transfer) - Networking (data transfer, load balancers) - Managed services (RDS, ElastiCache, etc.) # Indirect costs - Data egress (outbound transfer) - Support plans (Basic free, Business $) - Licensing (Windows, Oracle, etc.) - Training and hiring # Hidden costs - Over-provisioned resources - Unused resources (orphaned EBS, idle LBs) - Cross-region data transfer - API request charges # Optimization strategies 1. Right-sizing (match instance to workload) 2. Reserved capacity (1-3 year commitments) 3. Auto-scaling (scale down when idle) 4. Storage tiering (move cold data to cheaper tiers) 5. Cleanup (delete unused resources)
