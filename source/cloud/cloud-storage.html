Cloud Storage

A focused guide to Cloud Storage.

Object Storage Basic

Object storage stores data as objects (files + metadata) in a flat namespace. Ideal for unstructured data, images, backups, and data lakes.

Characteristics

  • Flat namespace (no directory hierarchy)
  • HTTP/REST API access
  • Unlimited scalability
  • Metadata per object
  • Eventually consistent

Use Cases

  • Static website hosting
  • Data lake storage
  • Backup and archives
  • Media files (images, videos)
  • Application logs

Object Storage Structure

Bucket/Container → Object (Key + Data + Metadata)
s3://my-bucket/ → data/2024/report.pdf

Block Storage Basic

Block storage divides data into fixed-size blocks and attaches to VMs as volumes. Used for databases and operating systems.

Characteristics

  • Fixed-size blocks (4KB-64KB)
  • Attached to single VM
  • Low latency, high IOPS
  • Formatted with filesystem
  • Snapshots supported

Use Cases

  • Operating system disks
  • Database storage
  • Transaction-heavy workloads
  • Boot volumes
# AWS EBS Volume Types # gp3 (General Purpose SSD) - Default - 3,000 IOPS baseline - 125 MiB/s throughput - $0.08/GB/month # io2 (Provisioned IOPS SSD) - Up to 64,000 IOPS - 1,000 MiB/s throughput - Best for databases # st1 (Throughput Optimized HDD) - 500 IOPS baseline - Big data, data warehouses # sc1 (Cold HDD) - 250 IOPS baseline - Infrequent access

File Storage Basic

File storage provides NFS/SMB file systems accessible by multiple machines simultaneously.

Characteristics

  • Hierarchical directory structure
  • POSIX compatible
  • Shared access (NFS/SMB)
  • Metadata on files

Use Cases

  • Shared application data
  • Content management
  • Home directories
  • Legacy applications
# AWS EFS - Elastic File System # Create EFS aws efs create-file-system \ --performance-mode generalPurpose \ --throughput-mode bursting \ --tags Key=Name,Value=my-efs # Mount on EC2 sudo mount -t efs fs-12345678:/ /mnt/efs # Azure Files - SMB/NFS az storage share create \ --name my-share \ --account-name mystorageaccount # Mount on Linux sudo mount -t cifs //account.file.core.windows.net/share /mnt/azure

AWS S3 (Simple Storage Service) Intermediate

S3 is Amazon's object storage service, offering industry-leading scalability, durability, and security.

# S3 Key Concepts # Bucket - Container for objects aws s3 mb s3://my-bucket-name # Objects - Key + Data + Metadata # Key = path-like identifier # data/2024/report.pdf # Upload / Download aws s3 cp local-file.txt s3://my-bucket/data/ aws s3 cp s3://my-bucket/data/file.txt ./ # Sync directory aws s3 sync ./local-folder s3://my-bucket/folder/ # List objects aws s3 ls s3://my-bucket/data/ aws s3 ls s3://my-bucket/ --recursive # Presigned URLs (temporary access) aws s3 presign s3://my-bucket/file.txt --expires-in 3600 # S3 Select (query objects) aws s3api select-object-content \ --bucket my-bucket \ --key data.csv \ --expression "SELECT s.id FROM s3object s WHERE s.amount > 100" \ --expression-type SQL \ --input-serialization CSV \ --output-serialization JSON

Azure Blob Storage Intermediate

Azure Blob Storage is Microsoft's object storage solution for unstructured data.

# Azure Blob Key Concepts # Storage Account - Top-level container az storage account create \ --name mystorageaccount \ --resource-group myRG \ --location eastus # Container - Like S3 bucket az storage container create \ --name my-container \ --account-name mystorageaccount # Blob Types # Block Blobs - Upload files (default) # Append Blobs - Log files # Page Blobs - Random read/write (VM disks) # Upload / Download az storage blob upload \ --container-name my-container \ --file ./local-file.txt \ --name data/local-file.txt az storage blob download \ --container-name my-container \ --name data/local-file.txt \ --file ./downloaded-file.txt # List blobs az storage blob list \ --container-name my-container \ --output table # Shared Access Signature (SAS) az storage blob generate-sas \ --container-name my-container \ --name data/file.txt \ --permissions r \ --expiry 2024-12-31

Google Cloud Storage Intermediate

Google Cloud Storage (GCS) is Google's unified object storage service.

# GCS Key Concepts # Create bucket gsutil mb -l us-central1 gs://my-bucket/ # Upload / Download gsutil cp local-file.txt gs://my-bucket/data/ gsutil cp gs://my-bucket/data/file.txt ./ # Sync directory gsutil -m rsync -r ./local-folder gs://my-bucket/folder/ # List objects gsutil ls gs://my-bucket/ gsutil ls -r gs://my-bucket/** # Signed URLs gsutil signurl -d 1h service-account.json gs://my-bucket/file.txt # Storage Classes # Standard - Frequently accessed # Nearline - >30 days # Coldline - >90 days # Archive - >365 days # BigQuery integration bq load --source_format=CSV my_dataset.my_table gs://my-bucket/data.csv

Storage Tiers Basic

Storage tiers optimize cost based on access frequency. Hot for frequent access, cold/archive for infrequent.

# Storage Tier Comparison # AWS S3 Tiers ┌──────────────┬────────────────┬─────────────────────┐ │ Tier │ Min Duration │ Cost (per GB/month) │ ├──────────────┼────────────────┼─────────────────────┤ │ Standard │ None │ $0.023 │ │ Standard-IA │ 30 days │ $0.0125 │ │ One Zone-IA │ 30 days │ $0.01 │ │ Glacier │ 90 days │ $0.004 │ │ Glacier Deep │ 180 days │ $0.00099 │ └──────────────┴────────────────┴─────────────────────┘ # Azure Blob Tiers ┌──────────────┬────────────────┬─────────────────────┐ │ Hot │ None │ Higher storage cost │ │ Cool │ 30 days │ Lower storage cost │ │ Cold │ 90 days │ Even lower cost │ │ Archive │ 180 days │ Lowest cost │ └──────────────┴────────────────┴─────────────────────┘ # GCS Storage Classes ┌──────────────┬────────────────┬─────────────────────┐ │ Standard │ None │ General purpose │ │ Nearline │ 30 days │ Infrequent access │ │ Coldline │ 90 days │ Rare access │ │ Archive │ 365 days │ Long-term archive │ └──────────────┴────────────────┴─────────────────────┘

Tier Selection

Daily Access? → Hot / Standard
Monthly Access? → Cool / Nearline
Yearly Access? → Archive

Lifecycle Policies Intermediate

Lifecycle policies automatically transition objects between storage tiers or delete them based on age.

# AWS S3 Lifecycle Policy (JSON) { "Rules": [ { "ID": "MoveToIA", "Status": "Enabled", "Prefix": "data/", "Transitions": [ {"Days": 30, "StorageClass": "STANDARD_IA"}, {"Days": 90, "StorageClass": "GLACIER"} ], "Expiration": {"Days": 365} } ] } # GCS Lifecycle Rule gsutil lifecycle set lifecycle.json gs://my-bucket # lifecycle.json { "lifecycle": { "rule": [ { "action": {"type": "SetStorageClass", "storageClass": "NEARLINE"}, "condition": {"age": 30, "matchesStorageClass": ["STANDARD"]} }, { "action": {"type": "Delete"}, "condition": {"age": 365} } ] } } # Azure Blob Lifecycle Management # Configure via Azure Storage Management Policy # Rules: Base blob, snapshot, version actions

Object Versioning Intermediate

Versioning keeps multiple variants of objects, enabling recovery from accidental deletions or overwrites.

# AWS S3 Versioning # Enable versioning aws s3api put-bucket-versioning \ --bucket my-bucket \ --versioning-configuration Status=Enabled # List versions aws s3api list-object-versions --bucket my-bucket # Delete specific version aws s3api delete-object \ --bucket my-bucket \ --key file.txt \ --versionId abc123 # Restore deleted version aws s3api copy-object \ --bucket my-bucket \ --copy-source my-bucket/file.txt?versionId=abc123 \ --key file.txt # GCS Object Versioning gsutil versioning set on gs://my-bucket/ # Azure Blob Versioning az storage blob service-properties update \ --account-name myaccount \ --enable-versioning true # Soft Delete (recover deleted objects) # S3: Delete markers + versioning # Azure: Soft delete with retention period # GCS: Object lifecycle + retention

Storage Encryption Intermediate

Cloud storage encrypts data at rest and in transit using various methods.

# Encryption Options # 1. Server-Side Encryption (SSE) # Provider manages keys (default) # AWS: SSE-S3, SSE-KMS, SSE-C # Azure: Storage Service Encryption # GCS: Google-managed keys # AWS S3 with KMS aws s3api put-bucket-encryption \ --bucket my-bucket \ --server-side-encryption-configuration '{ "Rules": [{ "ApplyServerSideEncryptionByDefault": { "SSEAlgorithm": "aws:kms", "KMSMasterKeyID": "alias/my-key" } }] }' # 2. Client-Side Encryption # Encrypt before upload # You manage all keys # 3.
