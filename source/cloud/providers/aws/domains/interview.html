50+ AWS Data Engineering Q&A

Real interview questions with detailed answers covering architecture, optimization, performance tuning, and best practices.

Architecture & Design Patterns

Q1: Explain the 9-layer data engineering architecture model.▼

Answer: The 9-layer model provides a structured approach to designing data pipelines:

1. Data Source - Origin of data (DBs, APIs, files, IoT)

2. Ingestion - Moving data in (DMS, Kinesis, Glue, DataSync)

3. Processing - Transforming data (Glue, EMR, Lambda)

4. Storage - Persisting data (S3, DynamoDB, Redshift)

5. Orchestration - Managing workflow (MWAA, Step Functions)

6. Query/Analytics - Consuming data (Athena, QuickSight)

7. Monitoring - Observability (CloudWatch, CloudTrail)

8. Security - Protection (IAM, KMS, VPC)

9. Catalog/Metadata - Discovery (Glue Catalog, Lake Formation)

Q2: What are the differences between data lake and data warehouse?▼

Answer:

Data Lake:

  • Stores raw data in native format (Parquet, ORC, JSON, CSV)
  • Schema-on-read approach
  • Uses S3 as primary storage
  • Lower storage costs (Glacier Intelligent-Tiering)
  • Good for ML, exploration, diverse data types
  • Services: S3 + Glue Catalog + Lake Formation

Data Warehouse:

  • Stores processed, structured data
  • Schema-on-write approach
  • Columnar storage (Redshift, Redshift Spectrum)
  • Optimized for analytical queries
  • Higher compute costs
  • Services: Redshift, RDS, Aurora
Q3: Explain the medallion data lake architecture pattern.▼

Answer: The medallion pattern organizes data into three zones:

Bronze (Raw): Exact copy of source data, immutable, for audit trail

Silver (Cleaned): Cleansed, validated, standardized data with basic transformations

Gold (Curated): Business-ready data, aggregated for analytics and BI

S3 Raw → Glue ETL → S3 Curated → Redshift/Athena │ │ └── DMS CDC ───────────────┘
Q4: When would you choose Lambda over Glue for ETL?▼

Answer: Choose Lambda when:

  • Processing time < 15 minutes (Lambda limit)
  • Small to medium data volumes
  • Simple transformations (not requiring Spark)
  • Event-driven (file arrival, API trigger)
  • Cost-effective for infrequent processing

Choose Glue when:

  • Large-scale ETL requiring Spark
  • Complex transformations (joins, aggregations)
  • Job bookmarking for incremental loads
  • Long-running jobs (>15 min)
  • Data catalog integration needed
Q5: What is the difference between Pushdown Predicates and Pushdown Fragments?▼

Answer:

Pushdown Predicates: Filter conditions pushed to the data source during reading. Reduces data read from source.

df.filter(col("date") > "2024-01-01").read() // Predicate pushed to S3 -- Only reads Parquet files matching the date filter

Pushdown Fragments: Partition pruning - skips entire files/directories based on partition metadata. Reduces I/O.

s3://bucket/data/year=2024/month=01/ // Partition prune -- Only reads the matching partition directory
Q6: Explain the CAP theorem and where each service fits.▼

Answer: CAP theorem states you can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance.

S3: CP - Strong consistency, highly available, sacrifices latency during partitions

DynamoDB: CP or CA (configurable) - Single digit ms latency

RDS/Aurora: CP - Primary replica with sync

Kinesis: CP - Ordered shards, durable writes

Redshift: Strong consistency with eventual leader node

Q7: How do you design for disaster recovery in data pipelines?▼

Answer:

  • Cross-region replication: S3 CRR, DynamoDB global tables, Redshift cross-region snapshots
  • Multi-AZ within region: RDS Multi-AZ, S3 default across AZs
  • Pipeline idempotency: Design jobs to be re-runnable without duplicates
  • State management: Store job state in DynamoDB/S3 for restart capability
  • Dead letter queues: SQS DLQ for failed messages
  • Testing: Regular DR drills, chaos engineering
Q8: What are the different types of slowly changing dimensions (SCD)?▼

Answer:

Type 0: Never change - keep original value

Type 1: Overwrite - no history kept

Type 2: Add new row - full history with effective dates

customer_id | name | effective_date | expiry_date | current -----------|-----------|---------------|-------------|-------- C001 | John Doe | 2024-01-01 | 2024-03-15 | false C001 | J. Smith | 2024-03-16 | NULL | true

Type 3: Add new column - store previous value

Type 4: Mini-dimension - separate history table

Storage & Data Lake

Q9: How do you optimize S3 performance for large-scale data lakes?▼

Answer:

  • Partitioning: Partition by date/category to reduce scans (s3://bucket/year=2024/month=01/)
  • File format: Use Parquet/ORC for columnar compression
  • File size: Target 128-256MB files for optimal Parquet
  • Bucket prefixing: Distribute reads across prefixes (data/0001/, data/0002/)
  • Lifecycle policies: Auto-tier to Glacier for old data
  • S3 Intelligent Tiering: Auto-move based on access patterns
Q10: S3 Standard vs Intelligent-Tiering vs Glacier - when to use each?▼

Answer:

S3 Standard: Frequently accessed data (<90 days), hot data, active processing

S3 Intelligent-Tiering: Unknown or unpredictable access patterns, automatic optimization

S3 Glacier: Archival, compliance, data < 1x/year access

S3 Glacier Deep Archive: Regulatory compliance, 7-10+ year retention

Cost comparison (approx $/GB/month): S3 Standard: $0.023 Intelligent-Tiering: $0.0125 + monitoring Glacier: $0.004 Glacier Deep Archive: $0.00099
Q11: What is S3 Event Notification and how do you use it?▼

Answer: S3 Event Notification triggers actions on bucket events (PUT, COPY, DELETE).

// Lambda trigger on S3 PUT exports.handler = async (event) => { const bucket = event.Records[0].s3.bucket.name; const key = decodeURIComponent(event.Records[0].s3.object.key); // Process the new file }; // Event types: s3:ObjectCreated:*, s3:ObjectRemoved:*, etc. // Destinations: SNS, SQS, Lambda, EventBridge

Use cases: trigger ETL on file arrival, trigger Lambda for processing, feed SQS for async processing.

Q12: How does S3 Select work and when should you use it?▼

Answer: S3 Select allows querying CSV/JSON/Parquet directly in S3 without fetching entire objects.

SELECT * FROM s3object s WHERE s.year > 2020 AND s.category = 'electronics' // 400x faster, 80% less data transferred // Use when: selective columns/rows, compressed data, Lambda processing
Q13: Explain Lake Formation vs Glue Catalog.▼

Answer:

Glue Data Catalog: Central metadata repository, Hive metastore compatible, used by Athena, Redshift Spectrum, EMR

Lake Formation: Built on Glue Catalog, adds:

  • Fine-grained table/column-level permissions
  • Row-level security
  • Data masking
  • Centralized access control
  • Blueprint-based lake formation
Q14: What are the benefits of using Parquet over CSV/JSON?▼

Answer:

  • Columnar format: Read only required columns (10x faster)
  • Compression: Snappy/Gzip (60-80% smaller)
  • Predicate pushdown: Skip irrelevant row groups
  • Schema evolution: Handle changing schemas gracefully
  • Stats in footer: Min/max for faster filtering
Q15: How do you handle schema evolution in data pipelines?▼

Answer:

  • Parquet with schema evolution: Add optional columns without breaking reads
  • Glue schema registry: Enforce compatible schemas
  • Explicit schema definition: Define schema explicitly in code
  • Backwards compatibility: New columns must be optional
  • Version control: Track schema changes in Glue Catalog
# Glue DynamicFrame handles schema evolution dyf = glueContext.create_dynamic_frame.from_catalog( table_name = "my_table", transformation_ctx = "datasource0" ) # Unknown columns handled as 'position' or 'withOrigin'

Streaming & Real-Time Processing

Q16: How do you calculate the number of Kinesis shards needed?▼

Answer: Use all three formulas and take the MAX:

Shards = MAX( ceil(TotalMB/s ÷ 1 MB/s), // Ingress limit ceil(TotalRecords/s ÷ 1000), // Records limit (1KB records) ceil(TotalMB/s × Consumers ÷ 2 MB/s) // Egress limit ) // Example: 50K records/sec, 500 bytes each, 2 consumers Ingress: 25 MB/s ÷ 1 MB/s = 25 shards Records: 50,000 ÷ 1000 = 50 shards Egress: 25 MB/s × 2 ÷ 2 MB/s = 25 shards Answer: MAX(25, 50, 25) = 50 shards
Q17: Kinesis Data Streams vs Firehose - when to use each?▼

Answer:

Kinesis Data Streams:

  • Custom consumers (Lambda, KCL apps)
  • Replay capability (up to 365 days)
  • Real-time processing
  • You manage shards

Firehose:

  • Zero-maintenance delivery to S3/Redshift/ES
  • No replay - fire and forget
  • Auto-scaling
  • Lambda transformation support
Q18: What is Kinesis Data Analytics and when would you use it?▼

Answer: Managed Apache Flink/SQL service for real-time analytics on streaming data.

-- Real-time aggregation SELECT TUMBLE_START(event_time, INTERVAL '1' MINUTE) as window, product_id, COUNT(*) as order_count, SUM(amount) as total_amount FROM OrdersStream GROUP BY TUMBLE(event_time, INTERVAL '1' MINUTE), product_id

Use when: Real-time dashboards, fraud detection, anomaly detection, windowed aggregations.

Q19: How do you handle late-arriving data in streaming pipelines?▼

Answer:

  • Watermarks: Define allowed lateness (WATERMARK BY event_time AS event_time - INTERVAL '5' MINUTE)
  • Late data handling: Side outputs for late events, rerouting to DLQ
  • Session windows: Capture late joins in session windows
  • Accumulation mode: DISCARDING vs ACCUMULATING
// Kinesis Firehose: Late data to S3 prefix // Glue Kinesis: Late data to separate output .withWatermark("event_time", "5 minutes") .sideOutputLateData(lateOutputTag)
Q20: Explain exactly-once processing semantics in Kinesis.▼

Answer: True exactly-once is hard. Options:

  • Kinesis + DynamoDB: Use record sequence numbers + idempotency keys in DynamoDB
  • Firehose: At-least-once, handle duplicates in destination
  • Application-level: Write to destination only after commit, use transaction IDs
// Idempotency pattern val recordId = record.getSequenceNumber() + "-" + checkpointId if (!dynamodb.exists(recordId)) { processRecord(record) dynamodb.put(recordId, "processed") }
Q21: What is MSK and how does it compare to Kinesis?▼

Answer:

MSK (Managed Streaming for Kafka):

  • Full Kafka compatibility
  • You manage topics/partitions
  • Custom consumer groups
  • Offset management

Kinesis:

  • Managed shards
  • Simpler API
  • Enhanced fan-out for multiple consumers
  • Serverless with On-Demand mode
Q22: How do you monitor streaming pipeline health?▼

Answer:

  • CloudWatch metrics: IncomingRecords, GetRecords.IteratorAgeMilliseconds, ReadProvisionedThroughputExceeded
  • Custom metrics: Records processed, latency, error rates
  • Iterator age: Key indicator of consumer lag
  • Alarm thresholds: Alert when metrics exceed limits
# CloudWatch Alarm for iterator age IteratorAgeMilliseconds > 60000 for 1 datapoint over 5 minutes → Auto-scale shards or increase consumers

Processing & ETL

Q23: Glue vs EMR - when to use each?▼

Answer:

Use Glue when:

  • Serverless preference, no infrastructure management
  • ETL jobs, data integration
  • Occasional/scheduled processing
  • Need job bookmarking for incremental loads
  • Data Catalog integration required
  • Cost predictability (pay per DPU-second)

Use EMR when:

  • ML training with SparkML
  • Persistent clusters for continuous processing
  • HDFS required for intermediate storage
  • Complex streaming with Apache Flink
  • Need GPU instances for ML
  • Cost optimization with Spot instances (60-70% savings)
Q24: How do you optimize Glue job performance?▼

Answer:

  • Data format: Use Parquet/ORC with appropriate compression
  • Partitioning: Partition output to reduce reads
  • DPUs: Scale workers (2-10 DPU per executor)
  • Job bookmarking: Enable for incremental processing
  • Dynamic frames: Use for schema evolution handling
  • Pushdowns: Filter early in the pipeline
# Glue Job Configuration glueJob.setJobMaterializationCloudWatchLogsEnabled(true) glueJob.setNumberOfWorkers(10) # 10 workers = 5 executors glueJob.setWorkerType("G.1X") # 4 vCPU, 16GB RAM glueJob.setExecutionProperty(f"${job_run_id}")
Q25: What are Glue job bookmarks and how do they work?▼

Answer: Bookmarks track processed files to enable incremental loads without reprocessing.

# Enable bookmarking datasource0 = glueContext.create_dynamic_frame.from_options( connection_type="s3", connection_options={"paths": ["s3://bucket/input/"]}, format="parquet", transformation_ctx="datasource0", enable_monitoring=True # or use job bookmark )

Stores state in DynamoDB. On next run, bookmarks filter out already-processed files based on last modified time.

Q26: Explain the Lambda cold start problem and solutions.▼

Answer:

Cold start times:

  • Python: 100-500ms
  • Node.js: 50-300ms
  • Java: 3-10 seconds

Solutions:

  • Provisioned Concurrency: Always-warm instances ($$$)
  • SnapStart: Lambda snapshots for Java (up to 90% faster)
  • Smaller packages: Minimize deployment size
  • Connection pooling: Reuse DB connections
  • Init code outside handler: Run once per instance
Q27: How do you handle errors and retries in Lambda?▼

Answer:

# Lambda retry behavior # Synchronous: 2 retries on error # Async: 2 retries (0, 1 min delay) + DLQ # DLQ Configuration lambda_client.put_function_event_invoke_config( FunctionName='my-function', DestinationConfig={ 'OnFailure': { 'Destination': 'arn:aws:sqs:region:account:dlq' } }, MaxRetries=2 ) # Retry with exponential backoff in code def lambda_handler(event, context): for attempt in range(3): try: process(event) break except Exception as e: if attempt == 2: raise time.sleep(2 ** attempt)
Q28: What is Step Functions and when should you use it?▼

Answer: Orchestration service for coordinating distributed applications and microservices.

Use when:

  • Multi-step workflows requiring state management
  • Long-running processes (>15 min Lambda limit)
  • Parallel execution of tasks
  • Human approval workflows
  • Distributed transactions across services
  • Retry/compensation logic for failures

Q29: How does DMS CDC work and what are its limitations?▼

Answer: DMS CDC captures database changes using log-based extraction.

How it works:

  • Enables log mining on source database
  • Reads transaction logs (CDC)
  • Applies changes to target
  • Supports full load + CDC

Limitations:

  • Not all DBs support log-based CDC
  • Requires proper configuration (binary logging, etc.)
  • Some DDL changes may not replicate
  • LOB handling can be tricky
Q30: What are the different types of joins in Spark? Which is most performant?▼

Answer:

Broadcast Hash Join (BHJ): Small table to all nodes. Fastest - no shuffle needed.

Shuffle Hash Join: Both tables partitioned. Shuffle required.

Sort Merge Join (SMJ): Large tables sorted and merged. Good for ordered data.

# Hint for broadcast join from pyspark.sql.functions import broadcast result = large_df.join(broadcast(small_df), "key") # autoBroadcastJoinThreshold (default 10MB) spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "100MB")

Data Warehouse & Analytics

Q31: Explain Redshift distribution keys and when to use each.▼

Answer:

KEY: Co-locate join keys on same node. Use for fact-dimension joins.

ALL: Copy small table to all nodes. Use for tables < 5MB.

EVEN: Round-robin distribution. Default, use when no clear pattern.

CREATE TABLE sales ( sale_id INT, product_id INT, amount DECIMAL(10,2), sale_date DATE ) DISTKEY(product_id); -- For joining with product_dim CREATE TABLE product_dim ( product_id INT, product_name VARCHAR(200), category VARCHAR(50) ) DISTSTYLE ALL; -- Small dimension table
Q32: What are Redshift sort keys and how do they improve performance?▼

Answer: Sort keys determine data ordering within slices, enabling zone maps for faster scans.

COMPOUND sort key: First column used for pruning. Best for range filters on first column.

INTERLEAVED sort key: Equal weight to all columns. Good for diverse filter patterns.

CREATE TABLE events ( event_date DATE, event_type VARCHAR(50), user_id INT, payload JSON ) COMPOUND SORTKEY(event_date, event_type); -- Best for: WHERE event_date > '2024-01-01' -- Best for: WHERE event_date = '2024-01-01' AND event_type = 'click'
Q33: What is Redshift WLM and how do you configure it?▼

Answer: Workload Management (WLM) routes queries to queues based on user groups.

-- WLM Configuration { "maxConcurrency": 5, -- Max queries per queue "memoryPercent": 50, -- Memory allocated to queue "timeout": 30000, -- Query timeout (ms) "queues": [ {"name": "etl_queue", "userGroups": ["etl_group"], "priority": "highest"}, {"name": "reporting", "userGroups": ["analyst"], "memoryPercent": 40}, {"name": "default", "memoryPercent": 20} ] }
Q34: How does Redshift Spectrum work and when should you use it?▼

Answer: Spectrum queries S3 data directly using Redshift's compute without loading data.

Use when:

  • Infrequently accessed data
  • Data too large for Redshift storage
  • Need both hot (Redshift) and cold (S3) queries
  • Data lake architecture
-- Create external table CREATE EXTERNAL TABLE spectrum.sales ( sale_id INT, product_id INT, amount DECIMAL(10,2), sale_date DATE ) STORED AS PARQUET LOCATION 's3://bucket/data/sales/'; -- Query SELECT * FROM spectrum.sales WHERE sale_date > '2024-01-01';
Q35: What is the difference between Athena and Redshift?▼

Answer:

Athena:

  • Serverless, pay per query
  • Presto-based, good for ad-hoc
  • No maintenance, instant scaling
  • Best for S3 data (no loading)

Redshift:

  • Persistent clusters
  • Better for repetitive analytics
  • Materialized views, window functions
  • Lower cost per query for large scans

Q37: What is materialized view in Redshift and when to use?▼

Answer: Pre-computed result set stored for fast retrieval.

CREATE MATERIALIZED VIEW mv_daily_sales AS SELECT sale_date, category, COUNT(*) as total_orders, SUM(amount) as total_revenue FROM sales GROUP BY sale_date, category; -- Auto-refresh ALTER MATERIALIZED VIEW mv_daily_sales AUTO REFRESH YES; -- Query like regular table SELECT * FROM mv_daily_sales WHERE sale_date > '2024-01-01';

Use for: Complex aggregations, frequent queries, dashboard data.

Q38: How do you handle slowly changing dimensions in Redshift?▼

Answer: Implement SCD Type 2 with effective dates.

CREATE TABLE customer_dim ( customer_key INT IDENTITY(1,1) PRIMARY KEY, customer_id VARCHAR(50) NOT NULL, customer_name VARCHAR(200), effective_date DATE NOT NULL, expiry_date DATE, is_current BOOLEAN DEFAULT true ); -- Merge logic (upsert) MERGE INTO customer_dim target USING source_table source ON target.customer_id = source.customer_id AND target.is_current = true WHEN MATCHED AND target.customer_name != source.customer_name THEN UPDATE SET expiry_date = CURRENT_DATE - 1, is_current = false WHEN NOT MATCHED THEN INSERT (customer_id, customer_name, effective_date, is_current) VALUES (source.customer_id, source.customer_name, CURRENT_DATE, true);

Security & Governance

Q39: How do you secure data at rest and in transit in AWS?▼

Answer:

At Rest:

  • S3: SSE-S3 (AWS managed), SSE-KMS (customer managed), SSE-C (customer key)
  • Redshift: Database encryption, KMS integration
  • DynamoDB: At-rest encryption with KMS
  • RDS: Transparent data encryption (TDE)

In Transit:

  • TLS/SSL for all connections
  • VPC endpoints for private connectivity
  • Security groups limiting access
Q40: What is IAM role assumption and when to use it?▼

  • Create LF-taggable columns for filtering

Q42: What are VPC endpoints and why are they important?▼

Gateway endpoints: S3, DynamoDB (free, routing table based)

Performance & Optimization

Q45: How do you optimize S3 list operations for large buckets?▼

Answer:

  • Use prefixes: List within specific prefix (s3://bucket/2024/01/)
  • Parallel listing: List multiple prefixes in parallel
  • Inventory: Use S3 Inventory for listing objects
  • Avoid deep hierarchies: Flat prefix structure
Q46: How do you handle small file problem in S3?▼

Answer: Small files cause inefficient processing. Solutions:

  • Compaction: Merge small files before processing
  • Glue dynamic frames: GroupByKey for consolidation
  • Target file size: Aim for 128-256MB Parquet files
# Glue compaction dyf = dyf.repartition(100) # Consolidate to 100 files datasink = glueContext.write_dynamic_frame.from_options( frame=dyf, connection_type="s3", format="glueparquet", connection_options={"path": "s3://bucket/output/"} )
Q47: How do you reduce Glue job costs?▼

Answer:

  • Worker type: G.2X for memory-intensive, G.1X for standard
  • Job efficiency: Minimize data scans, filter early
  • Bookmarks: Skip already-processed data
  • Max capacity: Set reasonable max DPU limits
  • Trigger vs job: Glue triggers share resources
Q48: What is columnar storage and why is it important for analytics?▼

Answer: Data stored column-by-column instead of row-by-row.

Benefits:

  • Column projection: Read only needed columns (10x faster)
  • Better compression: Similar values compress better
  • Vectorized queries: Process batches of values

Parquet and ORC are popular columnar formats in AWS.

Q49: How do you handle cross-account data access?▼

Answer:

# Account B reads from Account A's S3 # Account A bucket policy: { "Effect": "Allow", "Principal": {"AWS": "arn:aws:iam::111222333:role/reader-role"}, "Action": ["s3:GetObject", "s3:ListBucket"], "Resource": ["arn:aws:s3:::bucket-a/*", "arn:aws:s3:::bucket-a"] } # Account B role trusts Account A { "Effect": "Allow", "Principal": {"Service": "s3.amazonaws.com"}, "Action": "sts:AssumeRole", "Condition": {"ArnLike": {"aws:SourceArn": "arn:aws:s3:::bucket-a"}} }
Q50: What is the difference between catalog-based and schema-on-read?▼

Answer:

Schema-on-write (Catalog):

  • Define schema before loading data
  • Errors caught early
  • Examples: Redshift, RDS, traditional DW

Schema-on-read:

  • Schema applied at query time
  • Store data in any format
  • More flexible, but errors at query time
  • Examples: S3 + Athena, data lakes
