Data Source (Input Layer)

Where your data originates

Purpose

The entry point of your data pipeline. Data sources provide the raw data that will flow through ingestion, processing, and storage layers.

Types of Data Sources

Databases (OLTP)
Relational: RDS, Aurora, PostgreSQL, MySQL
NoSQL: DynamoDB, MongoDB
Data Warehouses: Redshift
Files & APIs
S3 (CSV, JSON, Parquet, Avro)
REST/GraphQL APIs
FTP/SFTP Drops
Event Streams
Logs & Events
Application Logs
CloudWatch Logs
Custom Events
IoT & Streaming
Sensors and Devices
Clickstreams
Time-series Data

Key Requirements

Data Availability

Ensure source systems are accessible and data is available when needed.

Format Clarity

Understand schema, data types, and encoding before designing pipelines.

Volume Estimation

Know your data volume and growth rate for capacity planning.

Before designing any pipeline, document your data sources thoroughly: schema diagrams, sample data, update frequencies, and any known data quality issues.

2

Ingestion Layer

Moving data from source to pipeline

Purpose

Responsible for moving data from source systems into your data infrastructure. Handles throughput requirements and schema consistency.

Ingestion Modes

Batch Ingestion
Scheduled Bulk Transfers
Good for Daily/Hourly Loads
Cost-effective for Large Volumes
Higher Latency Acceptable
Streaming Ingestion
Continuous Real-time Flow
Sub-second Latency
Handles Millions of Events/sec
Higher Cost per Event

AWS Services

AWS DMS
Kinesis Streams
Kinesis Firehose
Amazon MSK
DataSync

Key Requirements

Throughput Handling

Must handle peak load without data loss or excessive latency.

Schema Consistency

Maintain data types and formats during transfer.

Error Handling

Retry failed records, dead-letter queue for failures.

Streaming ingestion (Kinesis) costs more per event but prevents data loss. Batch (DMS) is cheaper but introduces latency. Choose based on your SLAs.

Processing Layer (Core Engine)

Transforming data - ETL / ELT

Purpose

The computational core of your data platform. Transforms, cleans, aggregates, and enriches data from raw to analysis-ready state.

Processing Patterns

ETL (Extract-Transform-Load)
Transform Before Loading
Clean Data at Source
Reduces Target Storage
Better for Complex Transforms
ELT (Extract-Load-Transform)
Load Raw, Transform Later
Full Data Preservation
Flexible Transformations
Better for Ad-hoc Analysis

AWS Services

AWS Glue
EMR
Lambda
AWS Batch

Internal Components

DAG Execution

Directed Acyclic Graph for task orchestration

Distributed Compute

Spark, Hadoop, Flink under the hood

Memory Management

Handling large datasets in memory

Shuffle Handling

Data movement between nodes during joins

Use Glue for serverless ETL, EMR for complex/big data processing (ML, large Spark jobs), Lambda for lightweight event-driven transforms.

Storage Layer

Storing raw and processed data

Purpose

Provides durable, scalable storage for data at rest. Stores both raw data (landing zone) and processed data (curated zone).

Storage Tiers

Raw / Landing Zone
Immutable Source of Truth
Never Overwritten
Enables Reprocessing
Schema-on-read
Curated / Analytics Zone
Processed, Validated Data
Optimized for Queries
Business-ready Format
Schema-enforced

AWS Services

S3 Data Lake
Redshift
Aurora
Timestream

Key Requirements

Partitioning Strategy

Partition by date/event_type for query pruning.

File Format

Use Parquet/ORC for analytics, JSON for flexibility.

Compression

Snappy, Gzip, Zstd for storage optimization.

Example Architecture

Raw/
Curated/
Trusted/
Archive/

Always use Parquet for analytical workloads - it provides columnar compression and predicate pushdown, reducing query costs by 10-100x.

5

Orchestration Layer

Controlling pipeline flow

Purpose

Coordinates pipeline execution, manages dependencies, handles retries, and provides scheduling. Critical for production reliability.

Orchestration Responsibilities

Dependency Management

Ensure tasks run in correct order

Scheduling

Cron-based or event-driven triggers

Retries

Automatic retry with backoff on failures

State Management

Track pipeline execution state

Failure Recovery

Idempotent reruns, checkpoints

Notifications

Alert on failures or SLAs

AWS Services

Step Functions
Amazon MWAA
EventBridge

When to Use Each

Step Functions
Workflows with Complex Branching/Conditions
Microservice Orchestration
Long-running Workflows (up to 1 year)
MWAA (Airflow)
Data Engineering Teams Familiar with Airflow
Complex DAGs with Many Dependencies
Large Ecosystem of Operators
EventBridge
Simple Scheduled Tasks
Event-driven Pipelines
Cost-effective for Basic Orchestration

Orchestration is often overlooked but is critical for production. Without proper orchestration, you have a collection of scripts, not a reliable pipeline.

Monitoring & Logging

Tracking pipeline health

Purpose

Provides observability into pipeline execution. Critical for production reliability, debugging, and SLA compliance.

What to Monitor

Failures

Job failures, exceptions, dead-letter queue buildup

Latency

Pipeline duration, data freshness, end-to-end delays

Throughput

Records processed/sec, MB/sec, queue depth

Cost

Cost per run, cost per GB, budget tracking

AWS Services

CloudWatch
CloudTrail
X-Ray

Key Metrics

CloudWatch Metrics
Glue: job_duration, records_written, data_units
Kinesis: success, iterator_age
Lambda: invocations, duration, errors
S3: number_of_objects, bucket_size_bytes
CloudWatch Alarms
Job Failure (Glue/EMR)
Iterator Age Growing (Kinesis)
Error Rate Exceeding Threshold
Data Freshness SLA Breach

Set up data quality metrics: record count, null percentage, duplicate rate. Data quality degradation often precedes pipeline failures.

AWS Services

IAM
KMS

Never use IAM users for application access. Always use IAM roles with temporary credentials. Store all secrets in Secrets Manager, never in code or config files.

Metadata & Catalog

Schema tracking and data discovery

Purpose

Provides schema registry, data lineage, and discovery capabilities. Prevents your data lake from becoming a data swamp.

Why Catalog Matters

Schema Tracking

Know data types, partitions, and schema versions

Data Discovery

Find datasets by keyword, tags, or owner

Lineage

Track data flow from source to consumption

Without a Catalog

Data Swamp

"What does this S3 prefix contain?"

Schema Drift

"Why did this query break?"

Lost Datasets

"Where did we put the Q3 report?"

Duplicate Work

"Did we already build this metric?"

AWS Services

Glue Data Catalog
Lake Formation

Catalog Best Practices

Use Crawlers

Auto-discover schema from S3, JDBC

Add Descriptions

Document tables and columns

Classify Data

Tag PII, sensitive fields

Track Partitions

Keep partition metadata current

Version Schemas

Track schema changes over time

Start with Glue Crawlers to auto-discover schemas, then add business descriptions and classifications. A catalog with good descriptions is worth more than a perfect schema.
