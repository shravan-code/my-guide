Data Types

Structured, semi-structured, and unstructured data — understanding data categories and relationships.

Structured Data Basic

Structured data is organized in a fixed schema with predefined rows and columns. It fits neatly into relational databases and spreadsheets.

Characteristics

  • Schema: Predefined and rigid
  • Storage: Relational databases (RDBMS)
  • Query: SQL
  • Examples: Customer records, transactions, inventory

Example Structure

-- Relational table structure customers ( id INTEGER PRIMARY KEY, name VARCHAR(100), email VARCHAR(255), created_at TIMESTAMP ) -- Excel/CSV structure id,name,email,created_at 1,Alice,alice@co.com,2024-01-15 2,Bob,bob@co.com,2024-02-20

Structured Data Storage

Data Source → Schema Definition → RDBMS → SQL Queries

Semi-Structured Data Basic

Semi-structured data has some organizational properties (tags, markers) but doesn't conform to a rigid schema.

Characteristics

  • Schema: Flexible, self-describing
  • Storage: Document stores, JSON columns
  • Query: JSONPath, XPath, NoSQL queries
  • Examples: JSON, XML, YAML, emails

JSON Example

{ "user_id": 123, "name": "Alice", "address": { "city": "Seattle", "state": "WA" }, "tags": ["premium", "verified"] }
-- PostgreSQL JSONB storage CREATE TABLE user_profiles ( id SERIAL PRIMARY KEY, profile JSONB NOT NULL ); -- Query semi-structured data SELECT profile ->> 'name' AS name, profile #>> '{address,city}' AS city FROM user_profiles WHERE profile @> '{"tags":["premium"]}'::JSONB;

Unstructured Data Basic

Unstructured data has no predefined schema or organization. It requires special processing to extract value.

Characteristics

  • Schema: None
  • Storage: Data lakes, object storage
  • Processing: ML, NLP, computer vision
  • Examples: Text documents, images, video, audio

Examples by Type

  • Text: Emails, social media posts, articles
  • Images: Photos, satellite imagery, X-rays
  • Audio: Podcasts, voice recordings, music
  • Video: Surveillance, meetings, streaming

Unstructured Data Processing

Raw Data → Feature Extraction → Structured Output → Analysis

Cardinality Intermediate

Cardinality refers to the uniqueness of values in a column. It affects indexing strategy and query performance.

Cardinality Levels

  • High: Nearly unique values (email, UUID)
  • Medium: Some repeats (city, product category)
  • Low: Few distinct values (status, boolean)
  • Unique: Every value is different (primary key)

Analysis Query

-- Check column cardinality SELECT COUNT(DISTINCT status) AS cardinality, COUNT(*) AS total_rows, ROUND( COUNT(DISTINCT status)::DECIMAL / COUNT(*) * 100, 2 ) AS uniqueness_pct FROM orders;

Cardinality & Index Strategy

High Cardinality → B-tree Index
Low Cardinality → Bitmap Index / GIN
Text Search → GIN (Full-text)

Dimensionality Intermediate

Dimensionality refers to the number of features or attributes in a dataset. High dimensionality can lead to performance and analysis challenges.

Dimensionality Concepts

  • Low-dimensional: Few columns (2-10)
  • High-dimensional: Many columns (100+)
  • Curse of Dimensionality: Distance metrics lose meaning
  • Dimensionality Reduction: PCA, t-SNE, UMAP

Example Impact

-- Low-dimensional (easy to query) users(id, name, age) -- High-dimensional (100+ features) -- Common in ML datasets: -- - User behavior vectors -- - Image embeddings -- - NLP token features -- Reduce with PCA in Python # from sklearn.decomposition import PCA # pca = PCA(n_components=10) # reduced = pca.fit_transform(data)

Sparsity Intermediate

Sparsity describes datasets where most values are zero or null. Common in recommendation systems and NLP.

Sparse vs Dense

-- Dense matrix (most values present) [[1, 2, 3], [4, 5, 6], [7, 8, 9]] -- Sparse matrix (mostly zeros) [[1, 0, 0], [0, 0, 6], [0, 8, 0]]

Storage Options

  • Compressed Sparse Row (CSR): Efficient row operations
  • Compressed Sparse Column (CSC): Efficient column operations
  • Dictionary: Store only non-zero values
  • PostgreSQL: JSONB for sparse columns
-- Sparse data in PostgreSQL CREATE TABLE user_preferences ( user_id INTEGER PRIMARY KEY, preferences JSONB DEFAULT '{}'::JSONB ); -- Only store non-null preferences INSERT INTO user_preferences VALUES (1, '{"theme": "dark", "lang": "en"}'), (2, '{"notifications": true}'); -- Sparse = efficient storage -- Dense = would store null for all other columns

Data at Rest vs Motion Basic

Data exists in different states throughout its lifecycle. Each state requires different handling and security measures.

Data at Rest

  • Stored in databases, files, data lakes
  • Encrypted storage (AES-256)
  • Access controls and auditing
  • Backup and recovery procedures

Data in Motion

  • Transmitted over networks
  • Encrypted transport (TLS/SSL)
  • API rate limiting
  • Message queuing (Kafka, RabbitMQ)

Data States in Pipeline

Source (Rest) → Extract (Motion) → Staging (Rest) → Transform (Motion) → Warehouse (Rest)

Data Volumes Basic

Understanding data scale helps choose appropriate storage, processing, and architecture patterns.

-- Data volume scale -- Kilobyte (KB) ~ 1,000 bytes = 1 page of text -- Megabyte (MB) ~ 1,000,000 bytes = 1 book -- Gigabyte (GB) ~ 1,000,000,000 = 1 movie (HD) -- Terabyte (TB) ~ 1,000,000,000,000 = 500 hours of HD video -- Petabyte (PB) ~ 1,000 TB = 10 years of HD video -- Exabyte (EB) ~ 1,000 PB = All words ever spoken -- PostgreSQL table size check SELECT relname AS table_name, pg_size_pretty(pg_total_relation_size(relid)) AS total_size, pg_size_pretty(pg_relation_size(relid)) AS table_size, pg_size_pretty(pg_total_relation_size(relid) - pg_relation_size(relid)) AS index_size FROM pg_catalog.pg_statio_user_tables ORDER BY pg_total_relation_size(relid) DESC;

Volume → Tool Selection

< 1 GB → SQLite / Postgres
1 GB - 1 TB → PostgreSQL / MySQL
1 TB - 1 PB → Spark / Data Warehouse
> 1 PB → Distributed Systems

One-to-One Relationships Basic

Each record in Table A relates to exactly one record in Table B, and vice versa.

-- One-to-One: User profile CREATE TABLE users ( id SERIAL PRIMARY KEY, username VARCHAR(50) UNIQUE NOT NULL, email VARCHAR(255) UNIQUE NOT NULL ); CREATE TABLE user_profiles ( user_id INTEGER PRIMARY KEY REFERENCES users(id), bio TEXT, avatar_url VARCHAR(500), birth_date DATE ); -- Query SELECT u.username, p.bio FROM users u JOIN user_profiles p ON u.id = p.user_id;

One-to-One Structure

users [1] ↔ [1] user_profiles

One-to-Many Relationships Basic

The most common relationship type. One record in Table A relates to many records in Table B.

-- One-to-Many: Customer to Orders CREATE TABLE customers ( id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL ); CREATE TABLE orders ( id SERIAL PRIMARY KEY, customer_id INTEGER REFERENCES customers(id), order_date DATE DEFAULT CURRENT_DATE, total DECIMAL(10,2) ); -- Query: Orders per customer SELECT c.name, COUNT(o.id) AS order_count, SUM(o.total) AS total_spent FROM customers c LEFT JOIN orders o ON c.id = o.customer_id GROUP BY c.id, c.name;

One-to-Many Structure

customers [1] → [*] orders

Many-to-Many Relationships Intermediate

Records in both tables can relate to multiple records in the other. Requires a junction table.

-- Many-to-Many: Students and Courses CREATE TABLE students ( id SERIAL PRIMARY KEY, name VARCHAR(100) ); CREATE TABLE courses ( id SERIAL PRIMARY KEY, title VARCHAR(200), credits INTEGER ); -- Junction table CREATE TABLE enrollments ( student_id INTEGER REFERENCES students(id), course_id INTEGER REFERENCES courses(id), enrolled_at TIMESTAMPTZ DEFAULT NOW(), grade CHAR(2), PRIMARY KEY (student_id, course_id) ); -- Query: Courses per student SELECT s.name, STRING_AGG(c.title, ', ') AS courses FROM students s JOIN enrollments e ON s.id = e.student_id JOIN courses c ON e.course_id = c.id GROUP BY s.id, s.name;

Many-to-Many Structure

students [*] ←→ enrollments ←→ courses [*]
