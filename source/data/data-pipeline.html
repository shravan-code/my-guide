Data Pipelines

A focused guide to Data Pipelines.

ETL (Extract, Transform, Load) Basic

ETL extracts data from sources, transforms it in a staging area, then loads into the target system.

ETL Flow

Extract → Transform → Load
Source Systems → Staging Server → Data Warehouse

Characteristics

  • Transform before loading
  • Requires staging area
  • Better for complex transformations
  • Data warehouse optimized

ELT (Extract, Load, Transform) Basic

ELT loads raw data directly into the target system, then transforms it using the target's compute power.

ELT Flow

Extract → Load → Transform
Source Systems → Data Lake → SQL Transform

Characteristics

  • Transform after loading
  • No staging server needed
  • Leverages target compute (dbt, Spark)
  • Modern cloud-native approach
-- dbt-style ELT transformations -- Stage: Raw data (loaded as-is) CREATE TABLE raw_events AS SELECT * FROM external_source; -- Transform: Clean and model CREATE TABLE stg_events AS SELECT event_id, event_type, payload ->> 'user_id' AS user_id, TO_TIMESTAMP(event_time) AS event_timestamp FROM raw_events WHERE event_type IS NOT NULL; -- Serve: Business model CREATE TABLE dim_events AS SELECT * FROM stg_events;

Batch vs Streaming Intermediate

Batch processes data in chunks; streaming processes data continuously as it arrives.

Batch Processing

  • Processes large volumes periodically
  • Higher latency (minutes to hours)
  • Complete data for accuracy
  • Tools: Spark, Hadoop, dbt

Streaming Processing

  • Processes data in real-time
  • Low latency (milliseconds)
  • Approximate results possible
  • Tools: Kafka, Flink, Spark Streaming

Batch vs Streaming Decision

Latency < 1 min? → Streaming
Complex joins? → Batch
Need exactly-once? → Batch or Micro-batch

Data Sources Basic

Data pipelines ingest from diverse sources, each with unique characteristics and challenges.

-- Common data sources -- 1. Databases (RDBMS) -- PostgreSQL, MySQL, SQL Server -- Full/incremental extraction via CDC -- 2. APIs (REST, GraphQL) -- JSON responses, pagination, rate limits -- 3. Files (CSV, JSON, Parquet) -- SFTP, S3, local filesystem -- 4. Event streams -- Kafka, Kinesis, Pub/Sub -- 5. SaaS applications -- Salesforce, HubSpot, Stripe (via APIs) -- 6. IoT sensors -- MQTT, time-series telemetry

Source → Ingestion Pattern

Database → CDC / Full Load
API → REST Polling / Webhook
Files → Scheduled File Watch
Events → Stream Consumer

Staging Area Basic

The staging area is an intermediate storage zone where raw data lands before transformation.

Purpose

  • Decouple extraction from transformation
  • Preserve raw data for reprocessing
  • Handle schema evolution
  • Enable data quality checks

Staging Design

-- Staging schema (minimal transformation) CREATE SCHEMA staging; CREATE TABLE staging.raw_orders ( raw_data JSONB, source_file VARCHAR(500), loaded_at TIMESTAMPTZ DEFAULT NOW() ); -- Or structured staging CREATE TABLE staging.orders ( id VARCHAR(50), data JSONB, _loaded_at TIMESTAMPTZ DEFAULT NOW() );

Transformation Intermediate

Transformations clean, enrich, and reshape raw data into analysis-ready formats.

-- Common transformations -- 1. Type casting SELECT id::INTEGER, amount::DECIMAL(10,2), created_at::TIMESTAMPTZ FROM staging.orders; -- 2. Cleaning SELECT TRIM(name) AS name, LOWER(email) AS email, COALESCE(amount, 0) AS amount FROM staging.orders; -- 3. Enrichment (join with dimension) SELECT o.*, c.name AS customer_name, c.segment FROM staging.orders o LEFT JOIN dim_customer c ON o.customer_id = c.id; -- 4. Aggregation SELECT customer_id, SUM(amount) AS total_spent, COUNT(*) AS order_count FROM staging.orders GROUP BY customer_id;

Data Serving Intermediate

Data serving delivers processed data to consumers: dashboards, APIs, ML models, or applications.

Data Serving Patterns

Data Warehouse → BI Dashboards
API Layer → Applications
Feature Store → ML Models
Cache (Redis) → Low-latency Apps
-- Materialized view for serving CREATE MATERIALIZED VIEW mv_daily_sales AS SELECT sale_date, product_id, SUM(amount) AS total_sales, COUNT(*) AS transaction_count FROM sales GROUP BY sale_date, product_id; -- Refresh periodically REFRESH MATERIALIZED VIEW mv_daily_sales; -- Create index for fast lookup CREATE INDEX idx_mv_date_product ON mv_daily_sales(sale_date, product_id);

Orchestration Intermediate

Orchestration manages dependencies, scheduling, and monitoring of pipeline tasks.

Key Concepts

  • DAG: Directed Acyclic Graph of tasks
  • Dependencies: Task ordering (upstream/downstream)
  • Retries: Automatic failure recovery
  • Alerting: Notify on failures

Popular Tools

  • Apache Airflow: Most popular, Python-based
  • Dagster: Data-aware orchestration
  • Prefect: Dynamic workflow orchestration
  • dbt: SQL transformation orchestration

Pipeline Orchestration Flow

Extract A → Transform → Load
↘
Extract B ↗ Join → Serve

Scheduling Basic

Define when and how often pipelines run based on business requirements.

-- Common schedules -- Real-time / Near real-time -- Trigger: Event or message arrival -- Latency: seconds to minutes -- Hourly -- Cron: 0 * * * * -- Use: Frequent reporting dashboards -- Daily (common) -- Cron: 0 2 * * * -- Run at 2 AM after source systems settle -- Weekly -- Cron: 0 3 * * 0 -- Run Sunday 3 AM for weekly aggregates -- Monthly -- Cron: 0 4 1 * * -- Run 1st of month at 4 AM -- Airflow DAG example # with DAG('daily_sales', schedule='0 2 * * *') as dag: # extract >> transform >> load

Monitoring Intermediate

Monitor pipeline health, data quality, and SLA compliance.

-- Pipeline metrics table CREATE TABLE pipeline_metrics ( id SERIAL PRIMARY KEY, pipeline_name VARCHAR(100), run_date DATE, status VARCHAR(20), -- success, failed, running records_processed INTEGER, duration_seconds INTEGER, started_at TIMESTAMPTZ, completed_at TIMESTAMPTZ, error_message TEXT ); -- Check pipeline health SELECT pipeline_name, MAX(started_at) AS last_run, CASE WHEN MAX(started_at) < NOW() - INTERVAL '24 hours' THEN 'OVERDUE' ELSE 'OK' END AS sla_status, ROUND(AVG(duration_seconds), 0) AS avg_duration_sec FROM pipeline_metrics WHERE run_date > CURRENT_DATE - 7 GROUP BY pipeline_name; -- Data quality alert SELECT * FROM pipeline_metrics WHERE status = 'failed' AND started_at > NOW() - INTERVAL '1 hour';

Apache Airflow Intermediate

Airflow is the most popular workflow orchestration platform for scheduling and monitoring pipelines.

# airflow_dag.py from airflow import DAG from airflow.operators.python import PythonOperator from datetime import datetime def extract_data(): # Extract from source pass def transform_data(): # Transform data pass def load_data(): # Load to warehouse pass with DAG( 'daily_etl', start_date=datetime(2024, 1, 1), schedule='0 2 * * *', catchup=False ) as dag: extract = PythonOperator( task_id='extract', python_callable=extract_data ) transform = PythonOperator( task_id='transform', python_callable=transform_data ) load = PythonOperator( task_id='load', python_callable=load_data ) extract >> transform >> load

dbt (Data Build Tool) Intermediate

dbt enables analysts to transform data using SQL with software engineering practices like version control and testing.

-- models/staging/stg_orders.sql SELECT order_id, customer_id, order_date::date AS order_date, amount::decimal(10,2) AS amount, status FROM {{ source('raw', 'orders') }} WHERE order_id IS NOT NULL; -- models/marts/fct_orders.sql SELECT o.order_id, o.order_date, c.customer_name, c.segment, o.amount FROM {{ ref('stg_orders') }} o LEFT JOIN {{ ref('dim_customers') }} c ON o.customer_id = c.customer_id; -- dbt commands # dbt run -- Execute transformations # dbt test -- Run data quality tests # dbt run --select +fct_orders -- Run dependencies # dbt docs generate -- Generate documentation
