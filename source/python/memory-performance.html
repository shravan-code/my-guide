What is Concurrency?

Concurrency is the ability of a program to execute multiple tasks simultaneously or in an interleaved manner. It improves throughput, responsiveness, and resource utilization. Python supports several concurrency models, each with distinct characteristics and trade-offs.

Types of Concurrency

  • Parallelism - True simultaneous execution on multiple CPU cores
  • Concurrency - Task interleaving, not necessarily simultaneous
  • Asynchrony - Non-blocking operations with callbacks/coroutines

Workload Types

  • CPU-bound - Heavy computation (data processing, ML)
  • IO-bound - Waiting on network, disk, or user input
  • Mixed - Combination of both workloads
Basic

The Global Interpreter Lock (GIL)

The GIL is a mutex that protects access to Python objects, preventing multiple threads from executing Python bytecode simultaneously. This simplifies memory management but limits true parallelism in threaded programs.

Why Does the GIL Exist?

CPython uses reference counting for memory management. Without the GIL, two threads could increment/decrement a reference count simultaneously, leading to memory leaks or premature deallocation. The GIL ensures thread-safety without expensive locking at every operation.

Key Insight: The GIL only affects CPU-bound Python code running in threads. I/O operations release the GIL, allowing other threads to run during waits.

How the GIL Works

# Simplified GIL mechanism import threading def cpu_bound_task(): # GIL is held during this execution result = 0 for i in range(10_000_000): result += i return result # Creating multiple threads - they still run sequentially due to GIL threads = [threading.Thread(target=cpu_bound_task) for _ in range(4)] # Parallel execution is limited by the GIL

Warning: Threading does NOT speed up CPU-bound Python code due to the GIL. Use multiprocessing for CPU-bound tasks.

Intermediate

Threading in Python

Threading allows multiple tasks to run within a single process, sharing memory space. While limited by the GIL for CPU-bound work, threading excels at I/O-bound tasks where threads wait for external resources.

When to Use Threading

Good For

  • Network requests (HTTP, APIs)
  • File I/O operations
  • Database queries
  • User interface responsiveness
  • Waiting on external services

Not Good For

  • CPU-intensive computations
  • Image processing
  • Data analysis on large datasets
  • Mathematical simulations
  • Machine learning training

Threading Example

import threading import time def fetch_data(url): """Simulates an I/O-bound network request""" print(f"Fetching: {url}") time.sleep(1) # GIL released during sleep return f"Data from {url}" urls = ["api.example.com", "api.github.com", "api.twitter.com"] # Sequential execution - takes 3 seconds # results = [fetch_data(url) for url in urls] # Concurrent execution with threading - takes ~1 second threads = [] for url in urls: t = threading.Thread(target=fetch_data, args=(url,)) threads.append(t) t.start() for t in threads: t.join() print("All requests complete!")

Multiprocessing in Python

Multiprocessing bypasses the GIL by creating separate Python processes, each with its own interpreter and memory space. This enables true parallelism for CPU-bound tasks across multiple CPU cores.

Process vs Thread Comparison

Processes

  • Separate memory space
  • True parallelism
  • Higher overhead
  • No GIL limitation
  • IPC required for communication

Threads

  • Shared memory space
  • Limited by GIL
  • Lower overhead
  • Cannot bypass GIL
  • Direct memory sharing

Multiprocessing Example

from multiprocessing import Pool, Process, Queue import os def cpu_intensive_task(n): """CPU-bound computation - runs in parallel with other processes""" print(f"Process {os.getpid()} computing {n}") return sum(i*i for i in range(n)) # Using Pool for simple parallel execution if __name__ == '__main__': numbers = [1000000, 2000000, 3000000, 4000000] # Sequential (slow for CPU-bound) # results = [cpu_intensive_task(n) for n in numbers] # Parallel - uses all available CPU cores with Pool(processes=4) as pool: results = pool.map(cpu_intensive_task, numbers) print(f"Results: {results}")

Inter-Process Communication

from multiprocessing import Process, Queue, Pipe, Manager # Queue - thread and process safe def producer(queue): for i in range(5): queue.put(i) queue.put(None) # Sentinel value def consumer(queue): while True: item = queue.get() if item is None: break print(f"Got: {item}") queue = Queue() p1 = Process(target=producer, args=(queue,)) p2 = Process(target=consumer, args=(queue,)) p1.start() p2.start() p1.join() p2.join()

Asyncio - Asynchronous I/O

Asyncio provides a single-threaded event loop for handling thousands of concurrent I/O operations without the overhead of threads. It uses coroutines and the event loop to switch between tasks during I/O waits.

When to Use Asyncio

Ideal For

  • Web servers (Flask, FastAPI with async)
  • WebSocket connections
  • Scraping many URLs concurrently
  • Database connection pooling
  • Chat servers, real-time apps

Avoid For

  • CPU-bound computation
  • Simple scripts with few I/O operations
  • Code requiring existing libraries (unless async-compatible)

Asyncio Fundamentals

import asyncio # Define an async coroutine async def fetch_data(url, delay): print(f"Fetching {url}...") await asyncio.sleep(delay) # Non-blocking wait return f"Data from {url}" # Run multiple coroutines concurrently async def main(): # Create tasks for concurrent execution tasks = [ fetch_data("api.example.com", 1), fetch_data("api.github.com", 2), fetch_data("api.twitter.com", 0.5), ] # Gather results from all tasks results = await asyncio.gather(*tasks) return results # Execute the async main function results = asyncio.run(main()) print(results) # Takes ~2 seconds total (max of delays)

Python Memory Layout

When Python executes a program, memory is divided into distinct areas, each serving a specific purpose. Understanding this layout helps diagnose memory issues and optimize performance.

Memory Regions

Code Segment

Stores compiled bytecode and constants. Read-only memory that holds the program's executable instructions.

Static Data

Global variables and static constants defined at module level. Persists for the program's lifetime.

Heap

Dynamic memory for objects and data structures. Managed by Python's memory allocator.

Stack

Function calls, local variables, and return addresses. Grows/shrinks automatically.

Intermediate

Heap vs Stack in Python

Understanding the distinction between heap and stack memory helps optimize memory usage and diagnose issues like memory leaks and stack overflow errors.

Stack Memory

  • Fixed size, managed automatically
  • Stores function calls and local variables
  • LIFO (Last In, First Out) access pattern
  • Fast allocation and deallocation
  • Limited size (~8MB default in Python)
  • Causes stack overflow on deep recursion

Heap Memory

  • Dynamic size, manually managed
  • Stores objects and data structures
  • Random access pattern
  • Garbage collected by Python
  • Can grow to system limits
  • Memory leaks possible here

Visualizing Stack and Heap

class DataContainer: # Instance variables live on the heap def __init__(self, value): self.value = value # Stored in heap (object) def process_data(): # Local variables on stack local_var = 42 # Object created on heap, reference on stack container = DataContainer(local_var) # container ref on stack, object on heap # List on heap, reference on stack data = [1, 2, 3] # data ref on stack, list on heap return container # Stack grows with each function call # Heap stores all dynamically created objects result = process_data() # When process_data returns, its stack frame is popped # Objects with no references are garbage collected

Tip: Use sys.getrecursionlimit() to check and sys.setrecursionlimit() to modify stack depth. But prefer iterative solutions when possible.

Garbage Collection in Python

Python uses a combination of reference counting and a cyclic garbage collector to manage memory automatically. This hybrid approach handles both immediate cleanup and detection of circular references.

Reference Counting

Every object maintains a reference count. When count reaches zero, memory is freed immediately. This is fast but cannot detect circular references.

import sys a = [1, 2, 3] # refcount = 1 b = a # refcount = 2 (same list object) c = [a] # refcount = 3 (list contains a) print(sys.getrefcount(a)) # Usually 4 (temporary ref in getrefcount call) del a # refcount = 2 del b # refcount = 1 del c # refcount = 0 -> object freed immediately

Cyclic Garbage Collector

The GC periodically detects and cleans up circular references that reference counting alone cannot handle.

GC Thresholds and Tuning

import gc # Get GC statistics print(gc.get_stats()) # Get collection thresholds print(gc.get_threshold()) # (700, 10, 10) # Manual collection (rarely needed) gc.collect() # Force full collection # Disable GC for performance-critical sections gc.disable() # ... critical code ... gc.enable() # Find unreachable objects unreachable = gc.garbage

Memory Leaks in Python

Memory leaks occur when objects are no longer needed but cannot be garbage collected due to lingering references. Common causes include global variables, callbacks, and caching.

Common Causes

Global References

Objects stored in module-level globals persist for the program's lifetime, even if no longer needed.

Circular References

Objects referencing each other form cycles that require GC to clean up, delaying memory release.

Closures

Functions capturing variables from enclosing scope can prevent garbage collection of those variables.

Caches

Unbounded caches (like dictionaries) grow indefinitely, holding references to old objects.

# BAD: Global list accumulates data all_data = [] # Global - never freed def process(item): all_data.append(item) # Grows forever! return process_item(item) # GOOD: Use bounded cache or clear periodically from functools import lru_cache @lru_cache(maxsize=1000) # Bounded cache def cached_process(item): return process_item(item) # GOOD: Clear cache when needed @lru_cache def bounded_process(item): return process_item(item) # bounded_process.cache_clear() # Clear when done

Warning: Always set maxsize for @lru_cache. Without it, the cache grows unbounded.

Advanced

CPU Caches and Memory Hierarchy

CPU caches are small, fast memory layers between the CPU and main memory. Understanding cache behavior helps write more cache-friendly code for better performance.

Cache Hierarchy

L1 Cache

  • Fastest, smallest (~32KB per core)
  • Access: ~1-2 nanoseconds
  • Split: instruction/data

L2 Cache

  • Larger, slower (~256KB per core)
  • Access: ~3-5 nanoseconds
  • Unified or split design

L3 Cache

  • Largest, slowest (~MB shared)
  • Access: ~10-20 nanoseconds
  • Shared across all cores

Main Memory

  • GB-sized, slow (~100+ nanoseconds)
  • RAM access latency
  • Cache line: 64 bytes

Profiling Tools

Profiling helps identify performance bottlenecks and memory issues. Python provides several built-in and third-party tools for different types of analysis.

CPU Profiling

# Built-in cProfile - identifies function-level bottlenecks import cProfile import pstats import io def slow_function(): total = 0 for i in range(10000): total += i * i return total profiler = cProfile.Profile() profiler.enable() result = slow_function() profiler.disable() # Print stats sorted by cumulative time s = io.StringIO() ps = pstats.Stats(profiler, stream=s).sort_stats('cumulative') ps.print_stats(10) # Top 10 functions print(s.getvalue())

Memory Profiling

# tracemalloc - built-in memory tracer import tracemalloc tracemalloc.start() # Your code here data = [i ** 2 for i in range(1000000)] current, peak = tracemalloc.get_traced_memory() print(f"Current: {current / 1024/1024:.2f} MB") print(f"Peak: {peak / 1024/1024:.2f} MB") tracemalloc.stop() # Compare memory snapshots tracemalloc.start() snapshot1 = tracemalloc.take_snapshot() # ... allocate memory ... large_list = [i for i in range(10000000)] snapshot2 = tracemalloc.take_snapshot() top_stats = snapshot2.compare_to(snapshot1, 'lineno') for stat in top_stats[:5]: print(stat)

Performance Optimization Patterns

Apply these patterns to optimize Python code performance. Always profile before and after to verify improvements.

Common Optimizations

Use Built-ins

Built-in functions are implemented in C and are much faster than Python loops.

# Slow total = 0 for x in data: total += x # Fast total = sum(data)

Local Variables

Local variable access is faster than global or attribute access.

# Slower (global lookup each iteration) for i in range(1000): math.sin(i) # Faster (local reference) sin = math.sin for i in range(1000): sin(i)

Generator Expressions

Lazy evaluation saves memory and can be faster for large datasets.

# List - creates full list in memory squares = [x*x for x in range(1000000)] # Generator - lazy evaluation squares = (x*x for x in range(1000000))

Use __slots__

Reduces memory footprint and can speed up attribute access.

class Point: __slots__ = ['x', 'y'] def __init__(self, x, y): self.x = x self.y = y

Pro Tip: "Premature optimization is the root of all evil." Profile first, then optimize the bottlenecks.
