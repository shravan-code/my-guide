Memory & Performance

Memory management, optimization, and profiling techniques.

What is Concurrency? Concurrency

Concurrency is the ability of a program to execute multiple tasks simultaneously or in an interleaved manner. It improves throughput, responsiveness, and resource utilization. Python supports several concurrency models, each with distinct characteristics and trade-offs.

Types of Concurrency

  • Parallelism - True simultaneous execution on multiple CPU cores
  • Concurrency - Task interleaving, not necessarily simultaneous
  • Asynchrony - Non-blocking operations with callbacks/coroutines

Workload Types

  • CPU-bound - Heavy computation (data processing, ML)
  • IO-bound - Waiting on network, disk, or user input
  • Mixed - Combination of both workloads

The Global Interpreter Lock (GIL) Concurrency

The GIL is a mutex that protects access to Python objects, preventing multiple threads from executing Python bytecode simultaneously. This simplifies memory management but limits true parallelism in threaded programs.

Why Does the GIL Exist?

CPython uses reference counting for memory management. Without the GIL, two threads could increment/decrement a reference count simultaneously, leading to memory leaks or premature deallocation.

# Reference counting without GIL (dangerous) # Thread 1: refcount = 5 → read → increment to 6 # Thread 2: refcount = 5 → read → increment to 6 # Result: Should be 7, but is 6 (lost update!) # The GIL prevents this race condition
  • Protects reference count operations
  • Prevents memory corruption
  • Simplifies C extension development

What the GIL Does NOT Protect

The GIL only protects Python bytecode operations. It does not protect against:

  • Non-Python operations (C extensions with their own locks)
  • Shared mutable data structures (you still need Locks)
  • I/O operations (handled by OS)
# GIL doesn't protect shared data! counter = 0 def unsafe_increment(): global counter for _ in range(100000): # Even with GIL, this is NOT thread-safe! # The check-then-increment is not atomic counter += 1

GIL Behavior: When It's Held vs Released

Understanding when the GIL is released is crucial for writing efficient concurrent code.

GIL is Released

  • I/O operations (file read/write, network)
  • time.sleep() calls
  • C extensions that release GIL
  • Some built-in operations

GIL is Held

  • Python bytecode execution
  • Attribute access/assignment
  • Function calls
  • Arithmetic operations
# Demonstration: GIL release during I/O import threading import time def io_task(name, duration): print(f"{name}: Starting") time.sleep(duration) # GIL released here! print(f"{name}: Finished") # These run concurrently despite GIL threads = [ threading.Thread(target=io_task, args=("Task1", 2)), threading.Thread(target=io_task, args=("Task2", 2)), ] for t in threads: t.start() for t in threads: t.join() # Total time: ~2 seconds (not 4!)

Key Insight: The GIL only affects CPU-bound Python code running in threads. I/O operations release the GIL, allowing other threads to run during waits.

How the GIL Works

# Simplified GIL mechanism import threading def cpu_bound_task(): # GIL is held during this execution result = 0 for i in range(10_000_000): result += i return result # Creating multiple threads - they still run sequentially due to GIL threads = [threading.Thread(target=cpu_bound_task) for _ in range(4)] # Parallel execution is limited by the GIL

Warning: Threading does NOT speed up CPU-bound Python code due to the GIL. Use multiprocessing for CPU-bound tasks.

Threading in Python Concurrency

Threading allows multiple tasks to run within a single process, sharing memory space. While limited by the GIL for CPU-bound work, threading excels at I/O-bound tasks where threads wait for external resources.

When to Use Threading

Good For

  • Network requests (HTTP, APIs)
  • File I/O operations
  • Database queries
  • User interface responsiveness
  • Waiting on external services

Not Good For

  • CPU-intensive computations
  • Image processing
  • Data analysis on large datasets
  • Mathematical simulations
  • Machine learning training
Example 1

Basic Thread Creation

The simplest way to create and start threads using threading.Thread.

import threading import time def worker(name, delay): print(f"{name}: Starting") time.sleep(delay) print(f"{name}: Done after {delay}s") # Create threads t1 = threading.Thread(target=worker, args=("Worker-1", 2)) t2 = threading.Thread(target=worker, args=("Worker-2", 1)) # Start threads t1.start() t2.start() # Wait for both to complete t1.join() t2.join() print("All workers finished!") # Total time: ~2 seconds (not 3!)
Example 2

Thread with Return Values

Threads don't return values directly. Use concurrent.futures or shared data structures.

from concurrent.futures import ThreadPoolExecutor def fetch_price(symbol): # Simulated API call time.sleep(0.5) prices = {"AAPL": 150.00, "GOOGL": 2800.00, "MSFT": 300.00} return prices.get(symbol, 0) # ThreadPoolExecutor handles thread lifecycle symbols = ["AAPL", "GOOGL", "MSFT"] with ThreadPoolExecutor(max_workers=3) as executor: futures = [executor.submit(fetch_price, s) for s in symbols] results = [f.result() for f in futures] print(f"Prices: {dict(zip(symbols, results))}")
Example 3

Real-World: Concurrent Web Scraping

A practical example of using threads to fetch multiple URLs concurrently.

import threading from urllib.request import urlopen from queue import Queue def download_worker(url_queue, results): while True: url = url_queue.get() if url is None: break try: with urlopen(url, timeout=5) as response: results[url] = len(response.read()) except Exception as e: results[url] = str(e) url_queue.task_done() # Setup urls = ["https://example.com", "https://httpbin.org/get"] url_queue = Queue() results = {} # Start worker threads threads = [] for _ in range(3): t = threading.Thread(target=download_worker, args=(url_queue, results)) t.start() threads.append(t) # Add URLs to queue for url in urls: url_queue.put(url) # Wait for completion url_queue.join() for _ in threads: url_queue.put(None) for t in threads: t.join()

Multiprocessing in Python Concurrency

Multiprocessing bypasses the GIL by creating separate Python processes, each with its own interpreter and memory space. This enables true parallelism for CPU-bound tasks across multiple CPU cores.

Process vs Thread Comparison

Processes

  • Separate memory space
  • True parallelism
  • Higher overhead
  • No GIL limitation
  • IPC required for communication

Threads

  • Shared memory space
  • Limited by GIL
  • Lower overhead
  • Cannot bypass GIL
  • Direct memory sharing

Multiprocessing Example

from multiprocessing import Pool, Process, Queue import os def cpu_intensive_task(n): """CPU-bound computation - runs in parallel with other processes""" print(f"Process {os.getpid()} computing {n}") return sum(i*i for i in range(n)) # Using Pool for simple parallel execution if __name__ == '__main__': numbers = [1000000, 2000000, 3000000, 4000000] # Sequential (slow for CPU-bound) # results = [cpu_intensive_task(n) for n in numbers] # Parallel - uses all available CPU cores with Pool(processes=4) as pool: results = pool.map(cpu_intensive_task, numbers) print(f"Results: {results}")

Inter-Process Communication

from multiprocessing import Process, Queue, Pipe, Manager # Queue - thread and process safe def producer(queue): for i in range(5): queue.put(i) queue.put(None) # Sentinel value def consumer(queue): while True: item = queue.get() if item is None: break print(f"Got: {item}") queue = Queue() p1 = Process(target=producer, args=(queue,)) p2 = Process(target=consumer, args=(queue,)) p1.start() p2.start() p1.join() p2.join()

Asyncio - Asynchronous I/O Concurrency

Asyncio provides a single-threaded event loop for handling thousands of concurrent I/O operations without the overhead of threads. It uses coroutines and the event loop to switch between tasks during I/O waits.

When to Use Asyncio

  • Web servers - FastAPI, aiohttp with async endpoints
  • WebSocket connections - Real-time bidirectional communication
  • Web scraping - Fetching thousands of URLs concurrently
  • Database pooling - Async database drivers (asyncpg, aiomysql)
  • Chat servers - IRC, Discord bots, real-time apps

Avoid Asyncio For

  • CPU-bound tasks - Use multiprocessing instead
  • Simple scripts - Overhead not worth it for few I/O ops
  • Synchronous libraries - Blocking calls stall the event loop
# ⚠️ BAD: Blocking call in async function async def bad_example(): time.sleep(5) # Blocks entire event loop! # GOOD: Use async version async def good_example(): await asyncio.sleep(5) # Non-blocking
Example 1

Basic Coroutine and Event Loop

The foundation of asyncio: defining async functions and running them in the event loop.

import asyncio # Define an async coroutine async def fetch_data(url, delay): print(f"Fetching {url}...") await asyncio.sleep(delay) # Non-blocking wait return f"Data from {url}" # Run multiple coroutines concurrently async def main(): tasks = [ fetch_data("api.example.com", 1), fetch_data("api.github.com", 2), fetch_data("api.twitter.com", 0.5), ] results = await asyncio.gather(*tasks) return results # Execute - takes ~2 seconds (max of delays) results = asyncio.run(main())
Example 2

Gather vs Task

asyncio.gather() runs coroutines concurrently and returns results in order. asyncio.create_task() schedules work and allows more control.

import asyncio async def compute(n): await asyncio.sleep(0.1) return n * n async def main(): # Method 1: gather() - wait for all, return in order results = await asyncio.gather( compute(2), compute(4), compute(6) ) print(results) # [4, 16, 36] # Method 2: create_task() - schedule and control task1 = asyncio.create_task(compute(10)) task2 = asyncio.create_task(compute(20)) await task1 # Wait for specific task await task2 # Then wait for another print(task1.result(), task2.result())
Example 3

Concurrency Control

Limit concurrent operations using Semaphore to avoid overwhelming resources.

import asyncio import aiohttp async def fetch_with_limit(url, session, semaphore): async with semaphore: async with session.get(url) as response: return await response.text() async def main(): urls = [f"https://api.example.com/{i}" for i in range(100)] semaphore = asyncio.Semaphore(10) # Max 10 concurrent async with aiohttp.ClientSession() as session: tasks = [fetch_with_limit(url, session, semaphore) for url in urls] results = await asyncio.gather(*tasks) return results
Example 4

Real-World: Async Web Scraper

Complete example of an async web scraper with rate limiting and error handling.

import asyncio import aiohttp async def scrape_url(session, url, semaphore): async with semaphore: try: async with session.get(url, timeout=10) as resp: if resp.status == 200: html = await resp.text() return {"url": url, "length": len(html)} return {"url": url, "error": resp.status} except Exception as e: return {"url": url, "error": str(e)} async def main(): urls = [f"https://example.com/page/{i}" for i in range(50)] semaphore = asyncio.Semaphore(5) # 5 concurrent requests async with aiohttp.ClientSession() as session: tasks = [scrape_url(session, url, semaphore) for url in urls] results = await asyncio.gather(*tasks) # Process results successful = [r for r in results if "error" not in r] print(f"Scraped {len(successful)}/{len(urls)} pages")

Python Memory Layout Memory

When Python executes a program, memory is divided into distinct areas, each serving a specific purpose. Understanding this layout helps diagnose memory issues and optimize performance.

Memory Regions

Code Segment

Stores compiled bytecode and constants. Read-only memory that holds the program's executable instructions.

Static Data

Global variables and static constants defined at module level. Persists for the program's lifetime.

Heap

Dynamic memory for objects and data structures. Managed by Python's memory allocator.

Stack

Function calls, local variables, and return addresses. Grows/shrinks automatically.

Deep Dive

Code Segment Details

The code segment stores the compiled bytecode of your Python source code. This area is read-only after initialization.

import dis def add(a, b): return a + b # View the bytecode dis.dis(add) # Constants are stored in code object print(add.__code__.co_consts) # (None,) print(add.__code__.co_varnames) # ('a', 'b')
  • Contains bytecode instructions
  • Stores function constants
  • Immutable during execution
  • Shared across function calls
Deep Dive

Heap Memory Management

Python's heap is managed by a layered memory allocator system designed for efficiency.

import sys # Check object sizes on heap my_list = [1, 2, 3, 4, 5] print(sys.getsizeof(my_list)) # 92 bytes my_dict = {"a": 1, "b": 2} print(sys.getsizeof(my_dict)) # 232 bytes # Heap allocator layers: # 1. PyObject malloc (small objects) # 2. Python's pymalloc (8-512 bytes) # 3. OS malloc (larger objects)

Use the sys module to inspect memory usage of objects.

import sys import ctypes # Get address of objects (demonstrates heap allocation) a = [1, 2, 3] b = [1, 2, 3] # Different objects at different addresses print(id(a)) # e.g., 140234866534784 print(id(b)) # e.g., 140234866535040 # Reference same object c = a print(id(c) == id(a)) # True - same heap address # Stack vs Heap example def demo_stack_heap(): x = 42 # x reference on stack, int on heap lst = [1, 2, 3] # lst reference on stack, list on heap return x, lst # Returns are copied (shallow)

Heap vs Stack in Python Memory

Understanding the distinction between heap and stack memory helps optimize memory usage and diagnose issues like memory leaks and stack overflow errors.

Stack Memory

  • Fixed size, managed automatically
  • Stores function calls and local variables
  • LIFO (Last In, First Out) access pattern
  • Fast allocation and deallocation
  • Limited size (~8MB default in Python)
  • Causes stack overflow on deep recursion

Heap Memory

  • Dynamic size, manually managed
  • Stores objects and data structures
  • Random access pattern
  • Garbage collected by Python
  • Can grow to system limits
  • Memory leaks possible here

Visualizing Stack and Heap

class DataContainer: # Instance variables live on the heap def __init__(self, value): self.value = value # Stored in heap (object) def process_data(): # Local variables on stack local_var = 42 # Object created on heap, reference on stack container = DataContainer(local_var) # container ref on stack, object on heap # List on heap, reference on stack data = [1, 2, 3] # data ref on stack, list on heap return container # Stack grows with each function call # Heap stores all dynamically created objects result = process_data() # When process_data returns, its stack frame is popped # Objects with no references are garbage collected

Tip: Use sys.getrecursionlimit() to check and sys.setrecursionlimit() to modify stack depth. But prefer iterative solutions when possible.

Garbage Collection in Python Memory

Python uses a combination of reference counting and a cyclic garbage collector to manage memory automatically. This hybrid approach handles both immediate cleanup and detection of circular references.

Core Concept

Reference Counting

Every object maintains a reference count. When count reaches zero, memory is freed immediately. This is fast but cannot detect circular references.

import sys a = [1, 2, 3] # refcount = 1 b = a # refcount = 2 (same list object) c = [a] # refcount = 3 (list contains a) print(sys.getrefcount(a)) # Usually 4 (temporary ref in getrefcount call) del a # refcount = 2 del b # refcount = 1 del c # refcount = 0 -> object freed immediately
  • Immediate deallocation when refcount = 0
  • No GC pause needed
  • O(1) time complexity
Problem

Circular References

When objects reference each other, reference counting alone cannot free them.

# Creating a circular reference class Node: def __init__(self, name): self.name = name self.other = None a = Node("A") b = Node("B") a.other = b # A references B b.other = a # B references A - cycle! # Even after del a, del b, objects persist # because refcount never reaches 0 del a del b # Objects A and B still exist in memory!
Solution

How the Cyclic GC Works

Python's cyclic garbage collector uses a generational approach to detect and clean up circular references.

Three Generations

  • Generation 0: Newly created objects
  • Generation 1: Survived one collection
  • Generation 2: Long-lived objects

Collection Trigger

  • Threshold exceeded (700, 10, 10)
  • Manual call to gc.collect()
  • Allocation failure
import gc # View generation thresholds print(gc.get_threshold()) # (700, 10, 10) # View collection counts print(gc.get_count()) # Objects in each generation # Force collection collected = gc.collect() print(f"Collected {collected} objects") # Disable GC for performance-critical sections gc.disable() # ... critical code with no circular refs ... gc.enable()
Example

Using Weak References to Avoid Cycles

weakref allows you to reference objects without increasing their reference count.

import weakref class Parent: def __init__(self, name): self.name = name self.children = [] class Child: def __init__(self, name, parent): self.name = name self.parent = weakref.ref(parent) # Weak reference parent = Parent("John") child = Child("Jane", parent) # Access weak reference print(child.parent()) # Returns Parent object or None # No circular reference - GC can clean up easily del parent # Child's parent reference returns None

GC Thresholds and Tuning

import gc # Get GC statistics print(gc.get_stats()) # Get collection thresholds print(gc.get_threshold()) # (700, 10, 10) # Manual collection (rarely needed) gc.collect() # Force full collection # Disable GC for performance-critical sections gc.disable() # ... critical code ... gc.enable() # Find unreachable objects unreachable = gc.garbage

Memory Leaks in Python Memory

Memory leaks occur when objects are no longer needed but cannot be garbage collected due to lingering references. Common causes include global variables, callbacks, and caching.

Common Causes

Global References

Objects stored in module-level globals persist for the program's lifetime, even if no longer needed.

Circular References

Objects referencing each other form cycles that require GC to clean up, delaying memory release.

Closures

Functions capturing variables from enclosing scope can prevent garbage collection of those variables.

Caches

Unbounded caches (like dictionaries) grow indefinitely, holding references to old objects.

# BAD: Global list accumulates data all_data = [] # Global - never freed def process(item): all_data.append(item) # Grows forever! return process_item(item) # GOOD: Use bounded cache or clear periodically from functools import lru_cache @lru_cache(maxsize=1000) # Bounded cache def cached_process(item): return process_item(item) # GOOD: Clear cache when needed @lru_cache def bounded_process(item): return process_item(item) # bounded_process.cache_clear() # Clear when done

Warning: Always set maxsize for @lru_cache. Without it, the cache grows unbounded.

CPU Caches and Memory Hierarchy Performance

CPU caches are small, fast memory layers between the CPU and main memory. Understanding cache behavior helps write more cache-friendly code for better performance.

Cache Hierarchy

L1 Cache

  • Fastest, smallest (~32KB per core)
  • Access: ~1-2 nanoseconds
  • Split: instruction/data

L2 Cache

  • Larger, slower (~256KB per core)
  • Access: ~3-5 nanoseconds
  • Unified or split design

L3 Cache

  • Largest, slowest (~MB shared)
  • Access: ~10-20 nanoseconds
  • Shared across all cores

Main Memory

  • GB-sized, slow (~100+ nanoseconds)
  • RAM access latency
  • Cache line: 64 bytes

Profiling Tools Performance

Profiling helps identify performance bottlenecks and memory issues. Python provides several built-in and third-party tools for different types of analysis.

Built-in

cProfile measures how much time each function takes. Perfect for finding CPU bottlenecks.

See exactly how much time each line takes. Install with pip install line_profiler.

Built-in

tracemalloc - Memory Tracing

Track memory allocations line by line to find memory leaks.

import tracemalloc tracemalloc.start() # Code to profile data = [dict(x=i, y=i**2) for i in range(100000)] # Take snapshot and analyze snapshot = tracemalloc.take_snapshot() top_stats = snapshot.statistics('lineno') for stat in top_stats[:5]: print(stat) # Output: script.py:10: size=1000 KiB, count=100000 # Compare two snapshots snapshot1 = tracemalloc.take_snapshot() # ... memory allocations ... snapshot2 = tracemalloc.take_snapshot() top_stats = snapshot2.compare_to(snapshot1, 'lineno')
Built-in

Measure execution time of small code snippets accurately.

import timeit # Compare list vs generator list_time = timeit.timeit( '[x**2 for x in range(1000)]', number=10000 ) gen_time = timeit.timeit( '(x**2 for x in range(1000))', number=10000 ) print(f"List: {list_time:.3f}s") print(f"Generator: {gen_time:.3f}s") # Compare string concatenation methods print(timeit.timeit("''.join(['a','b','c'])")) print(timeit.timeit("'a' + 'b' + 'c'"))

Performance Optimization Patterns Performance

Apply these patterns to optimize Python code performance. Always profile before and after to verify improvements.

Common Optimizations

Use Built-ins

Built-in functions are implemented in C and are much faster than Python loops.

# Slow total = 0 for x in data: total += x # Fast total = sum(data)

Local Variables

Local variable access is faster than global or attribute access.

# Slower (global lookup each iteration) for i in range(1000): math.sin(i) # Faster (local reference) sin = math.sin for i in range(1000): sin(i)

Generator Expressions

Lazy evaluation saves memory and can be faster for large datasets.

# List - creates full list in memory squares = [x*x for x in range(1000000)] # Generator - lazy evaluation squares = (x*x for x in range(1000000))

Use __slots__

Reduces memory footprint and can speed up attribute access.

class Point: __slots__ = ['x', 'y'] def __init__(self, x, y): self.x = x self.y = y

Pro Tip: "Premature optimization is the root of all evil." Profile first, then optimize the bottlenecks.

Key Takeaways Summary

Essential points to remember about Python memory and performance optimization.

Concurrency Models

Use Multiprocessing for CPU-bound tasks, Threading for simple I/O, and Asyncio for high-concurrency I/O.

GIL Awareness

The GIL only affects CPU-bound threads. I/O operations release the GIL, making threading effective for network/file operations.

Memory Management

Python uses reference counting + cyclic GC. Watch for memory leaks from globals, caches, and circular references.

Always Profile

Use cProfile, tracemalloc, or py-spy to identify bottlenecks. Don't optimize without data - premature optimization is the root of all evil.

Data Locality

Cache-friendly code is faster. Iterate arrays in memory order. Use generators for large datasets to save memory.

Use Built-ins

Python built-ins (sum, map, filter) are implemented in C. Local variables are faster than globals. Consider __slots__ for memory-heavy classes.
