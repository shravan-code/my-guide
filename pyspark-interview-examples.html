🔰 Basic Level Questions

1 What is PySpark and how does it differ from Apache Spark? Basic
💡 Answer

Apache Spark is a distributed computing system written in Scala. PySpark is the Python API for Spark that allows you to write Spark applications using Python.

Key Differences:

  • Language: Scala vs Python wrapper
  • Performance: Scala is slightly faster (no JVM-Python serialization)
  • PySpark uses Py4J to communicate with the JVM
  • Ease of Use: Python is easier with rich data science libraries
2 What is SparkContext and SparkSession? Basic
💡 Answer

SparkContext is the entry point to Spark functionality. It represents the connection to a Spark cluster and is used to create RDDs.

SparkSession (introduced in Spark 2.0) is the unified entry point that includes SparkContext, SQLContext, and HiveContext. It provides a single entry point for all Spark functionality.

# SparkContext (older way)
from pyspark import SparkContext
sc = SparkContext(master="local[*]", appName="MyApp")

# SparkSession (modern way)
from pyspark.sql import SparkSession
spark = SparkSession.builder \
    .appName("MyApp") \
    .master("local[*]") \
    .getOrCreate()

# Access SparkContext from SparkSession
sc = spark.sparkContext
3 What is Lazy Evaluation in Spark? Basic
💡 Answer

Lazy evaluation means that Spark does not execute transformations immediately. Instead, it builds a Directed Acyclic Graph (DAG) of operations. Execution only happens when an action is called.

# Transformations are LAZY (no execution)
filtered = df.filter(col("age") > 25)
selected = filtered.select("name", "email")

# Action triggers execution
result = selected.collect()  # NOW Spark executes
✨ Benefits

1. Spark can optimize the entire plan before execution

2. Reduces unnecessary data movement

3. Enables query optimization through Catalyst

5 What are Transformations and Actions in Spark? Basic
💡 Answer

Transformations create new RDDs/DataFrames from existing ones. They are LAZY.

Actions trigger execution and return values to the driver.

# Transformations (Lazy)
rdd.map(lambda x: x * 2)
df.filter(col("age") > 25)
df.select("name", "age")
df.groupBy("dept").sum("salary")

# Actions (Eager)
rdd.collect()
rdd.count()
rdd.take(5)
df.show()
df.count()
6 What is a Partition in Spark? Basic
💡 Answer

A partition is a chunk of data that can be processed in parallel on a single machine in the cluster. Each partition is processed independently by a single task.

# Check number of partitions
print(df.rdd.getNumPartitions())

# Repartition (full shuffle)
df_repartitioned = df.repartition(100)

# Coalesce (minimal shuffle)
df_coalesced = df.coalesce(50)

# Repartition by column
df_by_key = df.repartition(50, "department")
7 What is the difference between coalesce() and repartition()? Basic
💡 Answer

repartition(n): Triggers a full shuffle and creates n partitions. Use when increasing partitions or need even distribution.

coalesce(n): Minimizes shuffle by合并 partitions. Use when reducing partitions.

# repartition - full shuffle
df.repartition(100)  # Creates 100 new partitions with shuffle

# coalesce - minimal shuffle
df.coalesce(50)  # Reduces from 100 to 50 with minimal movement

# Best practice: Use coalesce when reducing partitions
8 What is caching and when should you use it? Basic
💡 Answer

Caching stores computed data in memory to avoid repeated computation. Use when:

  • A DataFrame is accessed multiple times
  • The data is not too large to fit in memory
  • The computation is expensive
# Cache DataFrame
df_filtered = df.filter(col("status") == "active")
df_filtered.cache()  # or .persist(StorageLevel.MEMORY_ONLY)

# Use multiple times
result1 = df_filtered.groupBy("dept").count()
result2 = df_filtered.filter(col("salary") > 50000)

# Unpersist when done
df_filtered.unpersist()

📊 Intermediate Level Questions

9 What is Data Skew and how do you handle it? Intermediate
💡 Answer

Data skew occurs when data is unevenly distributed across partitions. One partition may have much more data than others, causing some tasks to take much longer.

# Solution 1: Enable AQE (Automatic)
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# Solution 2: Salting
NUM_SALT = 50

# Add salt to dimension table
df2_salted = df2.withColumn("salt_key", 
    concat(col("key"), lit("_"), (rand() * NUM_SALT).cast("int"))
)

# Explode master table
df1_exploded = df1.withColumn("salt_key",
    concat(col("key"), lit("_"), explode(range(NUM_SALT)))
)

# Join
result = df1_exploded.join(df2_salted, "salt_key").drop("salt_key")
10 What are Broadcast Variables and when are they used? Intermediate
💡 Answer

Broadcast variables allow caching read-only data on all nodes to avoid sending it with each task. Used for:

  • Joining large DataFrame with small lookup table
  • Static reference data
  • Tables under ~20MB
# Create broadcast variable
lookup_dict = {"US": "USA", "UK": "United Kingdom", "DE": "Germany"}
bc_lookup = sc.broadcast(lookup_dict)

# Use in transformation
def lookup_country(code):
    return bc_lookup.value.get(code, "Unknown")

# For DataFrames - auto broadcast
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "100MB")
result = large_df.join(small_df, "key")  # Auto broadcast small_df
11 What is the difference between wide and narrow transformations? Intermediate
💡 Answer

Narrow Transformations: Each input partition contributes to only one output partition (e.g., map, filter).

Wide Transformations: Each input partition contributes to multiple output partitions (requires shuffle, e.g., groupBy, join, sortBy).

# Narrow: No shuffle needed
df.filter(col("age") > 25)  # Single partition
df.select("name", "age")    # Single partition
df.map(lambda x: x * 2)     # Single partition

# Wide: Shuffle required
df.groupBy("department").sum("salary")  # Shuffle
df.orderBy("name")                      # Shuffle
df.join(other_df, "id")                # Shuffle
12 What is AQE (Adaptive Query Execution)? Intermediate
💡 Answer

AQE enables Spark to dynamically optimize queries at runtime based on actual data statistics (Spark 3.0+).

# Enable AQE
spark = SparkSession.builder \
    .config("spark.sql.adaptive.enabled", "true") \
    .config("spark.sql.adaptive.skewJoin.enabled", "true") \
    .config("spark.sql.adaptive.coalescePartitions.enabled", "true") \
    .getOrCreate()

# AQE Features:
# 1. Dynamic join strategy selection
# 2. Dynamic partition coalescing
# 3. Dynamic skew handling
# 4. Runtime filtering
13 Explain the Spark architecture components. Intermediate
💡 Answer
  • Driver: Runs main(), creates SparkContext, schedules tasks
  • SparkContext: Connection to cluster, coordinates work
  • DAGScheduler: Creates DAG of stages
  • TaskScheduler: Launches tasks on executors
  • Cluster Manager: Allocates resources (YARN, K8s, etc.)
  • Worker Node: Runs executor processes
  • Executor: Runs tasks, stores cached data
14 What are the different join types in Spark? Intermediate
💡 Answer

Spark supports multiple join types: inner, outer, left, right, leftsemi, leftanti, and cross.

# All join types
result_inner = df1.join(df2, "key", "inner")           # Matching only
result_left = df1.join(df2, "key", "left")            # All from left
result_right = df1.join(df2, "key", "right")          # All from right
result_outer = df1.join(df2, "key", "outer")          # All from both
result_leftsemi = df1.join(df2, "key", "leftsemi")    # Left that match
result_leftanti = df1.join(df2, "key", "leftanti")    # Left that DON'T match
result_cross = df1.crossJoin(df2)                      # Cartesian product
15 How does Spark handle fault tolerance? Intermediate
💡 Answer

Spark achieves fault tolerance through:

  • RDD Lineage: DAG tracks transformations; lost partitions recomputed
  • Checkpointing: Save intermediate data to persistent storage
  • Task Retry: Failed tasks automatically retried
  • Executor Recovery: Cluster manager reschedules on other nodes
# Checkpointing for long lineages
spark.setCheckpointDir("hdfs:///checkpoint")
rdd.checkpoint()  # Save to reliable storage

# Or use DataFrame checkpointing
df.writeStream \
    .checkpoint("hdfs:///checkpoint") \
    .outputMode("complete") \
    .start()

16 What is the Catalyst Optimizer and how does it work? Advanced
💡 Answer

The Catalyst Optimizer is Spark's query optimizer that transforms logical plans into optimized physical plans.

Phases:

  1. Analysis: Resolve column names and types
  2. Logical Optimization: Apply optimization rules
  3. Physical Planning: Generate multiple plans
  4. Cost Model: Select best physical plan

18 Explain the difference between groupBy and window functions. Advanced
💡 Answer

groupBy: Aggregates data into summary rows (one per group).

Window Functions: Compute values across rows related to current row without collapsing rows.

from pyspark.sql.window import Window
from pyspark.sql.functions import row_number, rank, dense_rank, sum, avg

# groupBy - collapses to one row per group
df.groupBy("department").agg(
    sum("salary").alias("total_salary"),
    avg("salary").alias("avg_salary")
).show()

# Window - keeps all rows, adds calculated column
windowSpec = Window.partitionBy("department").orderBy(col("salary").desc())

df.withColumn("rank", row_number().over(windowSpec)) \
  .withColumn("running_total", sum("salary").over(
      Window.partitionBy("department").orderBy("hire_date")
  )).show()
19 What is the difference between map and flatMap? Advanced
💡 Answer

map: Transforms each element, returns one output per input.

flatMap: Transforms and flattens, returns zero or more outputs per input.

rdd = sc.parallelize(["hello world", "foo bar"])

# map - returns one output per input
rdd.map(lambda x: x.split(" ")).collect()
# Result: [['hello', 'world'], ['foo', 'bar']]

# flatMap - flattens the results
rdd.flatMap(lambda x: x.split(" ")).collect()
# Result: ['hello', 'world', 'foo', 'bar']

# Remove exact duplicates
df.dropDuplicates().show()

# Remove duplicates based on specific columns

21 How would you process a 1TB dataset with limited memory? Intermediate
💡 Answer
  • Increase partition count (more smaller partitions)
  • Use efficient file format (Parquet with compression)
  • Filter early to reduce data volume
  • Use selective column projection
  • Enable AQE for dynamic optimization
  • Use disk spilling if needed (MEMORY_AND_DISK)
  • Consider sampling for exploration
# Process large dataset efficiently
df = spark.read.parquet("s3://data/large_dataset/")

# Increase partitions
df = df.repartition(500)

# Filter early
df_filtered = df.filter(col("date") >= "2024-01-01")

# Select only needed columns
df_final = df_filtered.select("id", "name", "value", "date")

# Write in batches
df_final.write.mode("append").partitionBy("year", "month") \
    .parquet("s3://output/")

💡 Answer
  1. Spark UI: Check job/stage/task duration, identify bottlenecks
  2. Check for Skew: Look for tasks with much longer duration
  3. Data Size: Check partition sizes
  4. Explain Plan: Analyze query plan
  5. Memory: Check for GC issues
# Check explain plan
df.explain(True)

# Check partition statistics
df.groupBy(spark_partition_id()).count().orderBy(col("count").desc()).show()

# Check for skewed keys
df.groupBy("key").count().orderBy(col("count").desc()).show(10)

# Monitor with accumulator
error_count = spark.sparkContext.accumulator(0)
def count_errors(x):
    global error_count
    if has_error(x):
        error_count.add(1)
    return x
rdd.map(count_errors)

Common Performance Issues & Solutions

Before/After code examples with datasets

🔴 Issue 1: Selecting All Columns

Selecting all columns adds unnecessary data movement.

# ❌ BEFORE - Select all columns first
result = df.select("*") \
    .filter(col("status") == "active") \
    .select("id", "name", "salary") \
    .groupBy("department") \
    .agg(sum("salary"))
# ✅ AFTER - Filter and select only needed columns
result = df.filter(col("status") == "active") \
    .select("department", "salary") \
    .groupBy("department") \
    .agg(sum("salary").alias("total_salary"))

🔴 Issue 2: Not Handling Skewed Joins

Skewed data causes some tasks to take much longer.

Sample Data with Skew:
+-------+----+
|dept   |count|
+-------+----+
|Sales  |90000|  ← Skewed (90%)
|Engineering|5000|
|HR     |5000|
+-------+----+
# ❌ BEFORE - Causes skew issue
result = large_df.join(small_df, "key")
# One partition takes 90% of time
# ✅ AFTER - Enable AQE or use salting
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# Or manual salting
NUM_SALT = 50
small_salted = small_df.withColumn("salt", 
    concat(col("key"), lit("_"), (rand()*NUM_SALT).cast("int")))
large_exploded = large_df.withColumn("salt",
    concat(col("key"), lit("_"), 
           explode(array([lit(i) for i in range(NUM_SALT)]))))
result = large_exploded.join(small_salted, "salt")

🔴 Issue 3: Using Python UDFs

Python UDFs are slow due to serialization overhead.

# ❌ BEFORE - Slow Python UDF
@udf(returnType=StringType())
def categorize(salary):
    if salary > 7000:
        return "High"
    elif salary > 5000:
        return "Medium"
    return "Low"

result = df.withColumn("category", categorize(col("salary")))
# ✅ AFTER - Fast built-in functions
result = df.withColumn("category",
    when(col("salary") > 7000, "High")
    .when(col("salary") > 5000, "Medium")
    .otherwise("Low")
)

# Or use pandas_udf for complex logic
@pandas_udf(StringType())
def categorize_pandas(salary_series):
    return salary_series.apply(lambda x: 
        "High" if x > 7000 else "Medium" if x > 5000 else "Low"
    )
result = df.withColumn("category", categorize_pandas(col("salary")))

🔴 Issue 4: Not Repartitioning After Expensive Operations

After shuffle operations, data may be poorly distributed.

# ❌ BEFORE - Continue with unbalanced partitions
result = df.filter(...) \
    .groupBy("department").sum("salary") \
    .filter(...)  # Still has groupBy partitions
# ✅ AFTER - Repartition after expensive operations
result = df.filter(...) \
    .groupBy("department").sum("salary") \
    .filter(...) \
    .repartition(100)  # Re-balance for next operation

🔴 Issue 5: Caching Everything

Caching too much data causes memory pressure.

# ❌ BEFORE - Cache all intermediate DataFrames
df1 = df.filter(col("a") > 10).cache()
df2 = df.filter(col("b") > 20).cache()
df3 = df.filter(col("c") > 30).cache()
# All cached in memory - OOM!
# ✅ AFTER - Cache only what's needed multiple times
# Only cache if accessed 2+ times
# Use disk spill if needed
from pyspark.storagelevel import StorageLevel

result = df.filter(col("a") > 10) \
    .groupBy("dept").sum("value")  # Not cached if only used once

# Cache only when necessary
frequently_used = df.filter(col("status") == "active").cache()
# Use it multiple times
result1 = frequently_used.groupBy("dept").count()
result2 = frequently_used.filter(col("salary") > 5000)

🔴 Issue 6: Using collect() on Large Data

collect() brings all data to driver memory.

# ❌ BEFORE - collect() on huge data
all_data = df.collect()  # OOM if millions of rows!

# Or even worse
for row in df.collect():  # Process in driver
    process(row)
# ✅ AFTER - Process in parallel on executors

# Option 1: Use actions that don't collect
df.write.mode("overwrite").parquet("output/")

# Option 2: Take sample only
sample = df.sample(0.01).collect()

# Option 3: Process in parallel with map
def process_row(row):
    # Expensive processing
    return transformed_row

results = df.rdd.map(process_row).collect()

# Option 4: Use toPandas for small samples
pandas_sample = df.sample(0.001).toPandas()

🔴 Issue 7: Not Using Broadcast for Small Tables

Shuffling large table for simple join is expensive.

Sample:
large_df: 1 billion rows
small_df: 1,000 rows (lookup table)
# ❌ BEFORE - Shuffle large table
result = large_df.join(small_df, "key")
# Shuffles 1 billion rows across network
# ✅ AFTER - Broadcast small table
from pyspark.sql.functions import broadcast

result = large_df.join(
    broadcast(small_df),  # Sent to all executors once
    "key"
)
# No shuffle needed!

🔴 Issue 8: Not Enabling AQE

Missing automatic runtime optimizations.

# ❌ BEFORE - No AQE
spark = SparkSession.builder \
    .appName("MyApp") \
    .getOrCreate()
# Missing dynamic optimizations
# ✅ AFTER - Enable AQE
spark = SparkSession.builder \
    .appName("MyApp") \
    .config("spark.sql.adaptive.enabled", "true") \
    .config("spark.sql.adaptive.skewJoin.enabled", "true") \
    .config("spark.sql.adaptive.coalescePartitions.enabled", "true") \
    .config("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5") \
    .getOrCreate()
# AQE handles skew, coalesces partitions automatically

🔴 Issue 9: Not Using Appropriate File Format

Using CSV instead of Parquet for analytical workloads.

# ❌ BEFORE - Using CSV (slow, no compression, no column pruning)
df = spark.read.csv("data/large_file.csv", header=True, inferSchema=True)
# Reads entire file, no column pruning, poor compression
# ✅ AFTER - Using Parquet (fast, compressed, columnar)
# Write as Parquet
df.write.mode("overwrite") \
    .option("compression", "snappy") \
    .parquet("data/large_file.parquet")

# Read Parquet (only needed columns if using select)
df = spark.read.parquet("data/large_file.parquet") \
    .select("id", "name", "salary")  # Other columns not read!

🔴 Issue 10: Not Handling Schema Properly

Using inferSchema on large files is slow.

# ❌ BEFORE - Infer schema (scans data twice)
df = spark.read.csv("data/file.csv", header=True, inferSchema=True)
# Slow - reads file once to infer, again to load
# ✅ AFTER - Define schema explicitly
from pyspark.sql.types import StructType, StructField, StringType, IntegerType

schema = StructType([
    StructField("id", IntegerType(), False),
    StructField("name", StringType(), True),
    StructField("salary", IntegerType(), True),
    StructField("department", StringType(), True)
])

df = spark.read.schema(schema).csv("data/file.csv", header=True)
# No schema inference - faster!
